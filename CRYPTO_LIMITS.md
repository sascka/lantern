# Ограничение криптографических попыток Lantern v0.1

Статус: спецификация для `Lantern v0.1 experimental preview`.

Этот документ определяет постоянный лимит попыток Olm decrypt. Лимит действует
поверх транспортных квот и не сбрасывается при новом соединении или
перезапуске процесса.

Он не делает 64-битный MAC Olm v1 эквивалентным 128-битному и не гарантирует
доступность во враждебной сети. Его задача скромнее: ограничить скорость
поддельных попыток и стоимость обработки для честного узла.

## Что ограничивается

Лимит применяется только когда выполнены все условия:

- Envelope и открытая оболочка из `PAYLOAD.md` уже строго разобраны;
- `protected_payload` не превышает 8192 байта;
- `OlmMessage::from_parts()` успешно разобрал полные bytes;
- `recipient_hint` соответствует одному локальному контакту или текущему
  добавлению контакта;
- состояние допускает указанный `OlmMessage` type;
- следующий шаг собирается вызвать Account или Session decrypt.

Неизвестный hint отклоняется без Olm и не создаёт новый bucket. Повреждённая
оболочка тоже не расходует криптографический token. Её стоимость ограничивается
размерами, транспортной сессией и будущим общим лимитом соседа.

## Три уровня ограничения

Для одного профиля существуют:

| Bucket | Ёмкость | Восстановление | Назначение |
| --- | --- | --- | --- |
| профиль | 32 token | 1 за 10 секунд | все неудачные decrypt профиля |
| активный контакт | 8 token | 1 за 60 секунд | отдельная проверенная сессия |
| добавление контакта | 4 token | 1 за 300 секунд | первый pre-key decrypt |

Для активной сессии попытка требует одновременно один token профиля и один
token соответствующего контакта. Для первого pre-key message нужны token
профиля и token добавления контакта.

Если хотя бы один bucket пуст, decrypt не вызывается. Envelope остаётся в
ограниченной очереди до следующего разрешённого момента или до истечения TTL.

На профиль приходится не больше 128 contact bucket, один profile bucket и один
pending-contact bucket. Значит, число объектов limiter не превышает 130.

## Почему нужны оба уровня

Только contact bucket позволил бы атаковать много контактов параллельно. Только
profile bucket позволил бы одному контакту потратить весь общий запас.

Связка даёт:

- до восьми последовательных ошибок одного контакта без ожидания;
- после исчерпания не больше одной ошибки этого контакта в минуту;
- до 32 ошибок разных контактов коротким общим burst;
- после общего исчерпания не больше шести ошибок профиля в минуту;
- более строгую защиту one-time key во время первого контакта.

Числа являются консервативными начальными параметрами experimental MVP, а не
результатом независимого аудита. Их изменение требует нового ADR, тестов
границ и повторного расчёта числа попыток.

## Ключ bucket

Contact bucket связывается с локальным внутренним ID записи контакта. Этот ID:

- создаётся локально;
- не передаётся по сети;
- не равен identity key, session ID или `recipient_hint`;
- не принимается из Envelope;
- хранится только в будущем секретном хранилище.

Смена hint не создаёт новый bucket и не восстанавливает token. Повторное
подключение по сети также ничего не меняет.

Pending-contact bucket является одним постоянным bucket профиля. Отмена QR,
новый `invitation_id` или новый one-time key не сбрасывают его. Поэтому
многократное создание приглашений не обходит общий предел.

Удаление контакта удаляет его contact bucket. Повторное очное добавление создаёт
новую локальную запись с полным contact bucket, но profile bucket сохраняется.
Сеть сама не может создать контакт и использовать это для сброса лимита.

## Token bucket без настенных часов

Bucket хранит целое число доступных token от нуля до своей ёмкости. Дробных
token и floating point нет.

Во время работы используется только монотонная длительность процесса:

1. Для bucket хранится runtime anchor.
2. На проверке считается число полностью прошедших refill period.
3. За каждый полный период добавляется один token до capacity.
4. Anchor сдвигается на все учтённые полные периоды, даже когда bucket уже
   полный. Token сверх capacity отбрасываются, а неполный остаток времени
   сохраняется.

Настенные часы, timestamp Envelope и время соседа не участвуют. Перевод часов
вперёд или назад не восстанавливает token.

Между перезапусками сохраняется только текущее целое число token. Offline-время
не засчитывается: после запуска новый runtime anchor начинается с текущего
монотонного момента. Это более строго для доступности, но не позволяет
перезапуском или изменением часов мгновенно заполнить bucket.

Если платформа не учитывает сон устройства в монотонной длительности, token во
время сна не восстанавливаются. Если учитывает, восстановление всё равно не
превышает заданную скорость по прошедшей длительности. Поведение платформы
фиксируется отдельным тестом.

## Резервирование до decrypt

Нельзя сначала выполнить неудачный decrypt, а затем попытаться записать
уменьшенный bucket. Сбой или перезапуск между этими действиями позволил бы
повторить попытку бесплатно.

Одна попытка выполняется так:

1. Единственный crypto worker профиля выбирает следующий Envelope.
2. Limiter применяет накопившийся refill к нужным bucket.
3. Проверяется наличие token в обоих bucket.
4. По одному token вычитается и атомарно сохраняется в секретном хранилище.
5. Только после успешного commit вызывается decrypt на кандидатном Account или
   Session из `PAYLOAD.md`.
6. При любой криптографической или постоянной внутренней ошибке резервирование
   остаётся потраченным.
7. При полностью корректном результате оба token возвращаются одной транзакцией
   вместе с новым ratchet state и результатом обработки.

Если процесс завершился после шага 4, но до возврата token, попытка считается
неудачной. Это fail-closed поведение может уменьшить доступность, зато crash
loop не обходит лимит.

Если запись резервирования не удалась, decrypt не вызывается. Если запись
успешного состояния и возврата token не удалась, кандидатный ratchet
отбрасывается, plaintext не показывается, а зарезервированные token остаются
потраченными.

## Что считается успешной попыткой

Token возвращаются только после:

- успешного Olm decrypt;
- ограничения plaintext до 4155 байт;
- строгого внутреннего CBOR;
- совпадения всех защищаемых полей Envelope;
- проверки inner type и его полей;
- допустимого состояния контакта;
- успешной атомарной записи результата.

Корректное, но причинно раннее сообщение со статусом `state deferred` тоже
возвращает token. Его кандидатный ratchet отбрасывается, а повторная проверка
выполняется только после изменения локального состояния.

Успешное сообщение не добавляет token сверх зарезервированного. Оно только
возвращает один token в каждый bucket с ограничением по capacity. Нельзя
восстанавливать весь bucket после одного корректного сообщения.

## Что считается неудачной попыткой

Зарезервированные token остаются потраченными при:

- ошибке Account или Session decrypt;
- неверном MAC;
- превышении предела расшифрованного plaintext;
- повреждённом или неканоническом внутреннем CBOR;
- несовпадении внутренней копии Envelope;
- неверном contact secret, ID, generation или новом hint;
- запрещённом inner type для текущего Olm type;
- постоянной ошибке перехода состояния;
- сбое постоянной записи после decrypt.

`OlmMessage::from_parts()` вызывается до резервирования и может вернуть ошибку
без расхода token, потому что decrypt ещё не начался. Его вход уже ограничен
8183 байтами. Частый поток таких объектов относится к будущему общему лимиту
соседа между соединениями.

## Поведение при пустом bucket

Rate-limited Envelope не вызывает decrypt и не запускает активный цикл
повторов. Для контакта планируется не более одного следующего пробуждения на
ближайший refill period.

Когда token становится доступен, выбирается самый ранний ещё не проверенный
Envelope этого контакта. Постоянно отклонённый объект удаляется из активной
очереди и оставляет обычный ограниченный tombstone. Остальные ждут своей
очереди, TTL или вытеснения общей квотой.

На один контакт остаётся предел 32 отложенных crypto candidate из `PAYLOAD.md`.
Profile limit не создаёт отдельной неограниченной очереди.

Злоумышленник может постоянно присылать подделки раньше настоящего Envelope и
занять доступные token. Это отказ в обслуживании. Полностью устранить его
невозможно: Relay и так может удалить настоящий контейнер. Limiter не должен
маскировать этот риск заявлением о гарантированной доставке.

## Перезапуск и повреждение состояния

Token сохраняются в зашифрованном секретном хранилище после каждого
резервирования и возврата. Открытая SQLite-очередь и диагностическая база их не
содержат.

Логическая запись версии 1 содержит:

- `limiter_version = 1`;
- `profile_tokens` от 0 до 32;
- `pending_contact_tokens` от 0 до 4;
- не более 128 пар local contact ID и `contact_tokens` от 0 до 8.

Capacity и refill period являются константами реализации и не загружаются из
базы. Runtime anchor тоже не сериализуется. Так повреждённый файл не может
увеличить предел или записать время, которое даст лишний refill.

При перезапуске:

- загружаются последние committed integer values;
- offline refill не выполняется;
- runtime anchors создаются заново;
- незавершённое резервирование остаётся потраченным;
- новые попытки не начинаются до успешной загрузки всех bucket.

Неизвестная версия, лишний contact bucket, значение выше capacity, отсутствие
profile или pending bucket и противоречие с таблицей контактов блокируют crypto
операции. Повреждённое состояние нельзя заменять полными bucket по умолчанию.

Если contact bucket потерян при существующем контакте, он создаётся пустым, а
не полным. Восстановление начинается обычной скоростью. Это может временно
сорвать доставку, но повреждение базы не даёт дополнительные попытки.

## Оценка числа попыток

При постоянно работающем процессе profile bucket допускает не больше:

```text
32 + floor(runtime_seconds / 10)
```

неудачных decrypt за один непрерывный период атаки без учёта успешных
сообщений. За 365 суток это не больше 3 153 632 попыток.

Если идеализировать 64-битный MAC как независимое равномерное угадывание,
верхняя оценка вероятности хотя бы одного случайного успеха равна примерно
`N / 2^64`. Для указанного годового profile limit это около `1,71 * 10^-13`.

Это только простой верхний расчёт, а не доказательство безопасности Olm или
Lantern. Он не учитывает ошибки реализации, компрометацию ключа и другие пути
атаки. Даже с limiter граница остаётся 64-битной и должна быть пересмотрена до
beta или production.

## Диагностика и интерфейс

В безопасной диагностике разрешены только общие события:

- crypto attempt rejected;
- crypto attempt rate limited;
- crypto limiter storage failed;
- crypto limiter state invalid.

Событие не содержит:

- контакт или его внутренний ID;
- bucket type;
- точное число token;
- точный refill period или время следующей попытки;
- `message_id`, `recipient_hint`, ключ, ciphertext или plaintext;
- исходную ошибку хранилища или vodozemac.

Сетевому соседу не отправляется различимый ответ о неизвестном контакте, пустом
bucket, неверном MAC или внутренней ошибке. Локальный CLI может показать только
общее сообщение, что криптографическая обработка временно ограничена. Сбросить
bucket командой CLI в v0.1 нельзя.

## Обязательные тесты реализации

- точные capacity и refill period всех трёх bucket;
- границы нуля, одного token и полной ёмкости;
- burst 8, 4 и 32 без лишней попытки;
- совместное расходование contact и profile bucket;
- совместное расходование pending и profile bucket;
- отсутствие bucket для неизвестного hint;
- запрет decrypt до committed reservation;
- возврат ровно одного token после полного успеха;
- отсутствие возврата после каждой постоянной ошибки;
- возврат после `state deferred` без сохранения ratchet;
- сбой до reservation, после reservation и при success commit;
- crash и restart на каждом переходе;
- отсутствие offline refill после обычного и аварийного перезапуска;
- отсутствие refill при переводе настенных часов;
- точный дискретный refill и сохранение остатка периода;
- capacity не превышается после долгой работы;
- удаление контакта не заполняет profile bucket;
- новый QR не заполняет pending bucket;
- повреждённое, отсутствующее и превышающее capacity состояние;
- не более 130 bucket при 128 контактах;
- один crypto worker и отсутствие превышения при конкурентной подаче;
- отсутствие активного retry loop при пустом bucket;
- FIFO среди ожидающих Envelope одного контакта;
- TTL, tombstone и общая очередь продолжают ограничивать ожидание;
- property-тест длительных последовательностей reserve, refund, refill,
  restart и corruption;
- отсутствие contact ID, token, времени и секретов в `Debug`, ошибках и
  постоянной диагностике.

## Риски и ограничения

- Limiter создаёт возможность временно блокировать настоящий контакт
  поддельными Envelope.
- Profile bucket связывает доступность всех контактов одного профиля.
- Offline-время не восстанавливает token, поэтому после перезапуска ожидание
  может быть неожиданным для пользователя.
- Сбой записи после успешного decrypt расходует token без атаки.
- Парсер Olm message работает до reservation и требует отдельного общего лимита
  соседа.
- Точные параметры пока не проверены пользовательскими испытаниями.
- Хранение bucket зависит от ещё не выбранной защиты секретной базы.
- Математическая оценка не исправляет 64-битный MAC и не является аудитом.
- Реализации и тестов limiter пока нет.

Настоящий пользовательский текст остаётся запрещён до реализации limiter,
секретного хранилища и остальных условий ADR 0013.
