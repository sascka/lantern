# lantern-sync

Третий сетевой пакет Этапа 3. Crate реализует один ограниченный batch
синхронизации поверх любого `BoundedSession` и не зависит от TCP,
криптографического ядра или локальной базы.

Порядок одного направления:

```text
sender                         receiver
  |---- Offer: до 32 ID --------->|
  |<--- Request: только missing ---|
  |---- Transfer Envelope 1 ------>|
  |---- ... ---------------------->|
  |---- Done --------------------->|
```

Offer и Request содержат только отсортированные уникальные `message_id`. В них
нет `recipient_hint`, TTL, размеров, контактов, времени или tombstone.

Transfer содержит точную копию канонического Envelope и ограниченный route
grant: остаток TTL, число пройденных переходов и бюджет копий получателя. Его ID
повторяется в служебном заголовке и сверяется с ID внутри CBOR. Receiver
принимает Transfer только в ответ на Request текущего соединения.

## Ограничения

- один batch предлагает не больше 32 ID;
- один ID занимает 16 байт;
- sender обслуживает только ID из текущего Offer;
- receiver принимает Transfer строго в порядке своего Request;
- дубликаты и неотсортированные ID отклоняются;
- Transfer проходит строгий декодер `lantern-core`;
- route grant проверяется против общих границ и полей своего Envelope;
- любой sync error поглощает `BoundedSession`, поэтому ошибочное соединение
  нельзя случайно использовать дальше;
- одновременно в памяти декодируется только один входящий Envelope.

Для sender предусмотрен `EnvelopeSource`. Source вызывается только после
Request и готовит один Transfer за раз. Благодаря этому слой узла может
сохранить маршрутную reservation до отправки, не расходуя бюджет на простой
Offer.

Лимит 32 уменьшает раскрытие metadata и расход ресурсов, но не скрывает ID от
соседа. Случайный `message_id` не содержит имя контакта, однако позволяет
сопоставлять встречи, на которых предлагался один контейнер.

## Граница безопасности

- sync не аутентифицирует соседа и не проверяет E2EE;
- Relay видит предложенные ID, точные размеры кадров и сам зашифрованный
  Envelope;
- подмена байтов может быть замечена строгим декодером или криптографическим
  слоем, но сам sync не доказывает целостность сообщения;
- потеря, перестановка или лишний кадр завершают текущий batch, повтор выполняет
  следующий слой через новое соединение;
- абсолютное время устройства в sync-формат не входит;
- route grant не доказывает честный маршрут, локальные границы всё равно
  обязательны;
- ошибки и `Debug` не содержат ID или payload.

Кража локальной базы не относится к этому crate: он ничего не сохраняет. Риск
открытой транспортной базы и защита секретной SQLCipher-базы описаны отдельно в
`THREAT_MODEL.md` и `SECRET_STORAGE.md`.

## Ручная проверка

В первом терминале:

```bash
export PATH="$HOME/.cargo/bin:$PATH"
cargo run -p lantern-sync --example sync-probe --locked -- \
  listen 127.0.0.1:38384
```

Во втором терминале:

```bash
export PATH="$HOME/.cargo/bin:$PATH"
cargo run -p lantern-sync --example sync-probe --locked -- \
  connect 127.0.0.1:38384
```

Оба процесса должны вывести:

```text
One synthetic Lantern Envelope synchronized
```

Probe использует очевидный синтетический payload. Это не проверка E2EE и не
пользовательский интерфейс.

## Что пока не реализовано

- подтверждение сохранённого batch отправителю;
- передача Wait-копии подтверждённому конечному получателю;
- настоящие сообщения и CLI.
