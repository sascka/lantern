# lantern-node

Статус: жизненный цикл добавлен в девятом пакете, профиль - в десятом.

`lantern-node` связывает уже проверенные компоненты в один жизненный цикл узла.
Это библиотечный crate, а не готовое CLI-приложение.

## Что он делает

- читает время один раз на операцию, которая обслуживает или меняет состояние;
- открывает SQLite-хранилище и восстанавливает очередь;
- не позволяет второму процессу открыть тот же профиль;
- при откате часов завершает активные Envelope и очищает диагностику;
- удаляет достигшие TTL записи и сохраняет итог одной транзакцией;
- сохраняет созданный локально Envelope до ответа вызывающему коду;
- типизированно фиксирует запуск, восстановление, повторы, истечение и откат
  часов;
- создаёт транспортную сессию только через `BoundedSession`;
- по явному запросу восстанавливает отдельную базу безопасной диагностики;
- после ошибки часов, очереди, хранилища или диагностики переходит в `Failed` и
  не продолжает работу с возможно рассинхронизированным состоянием.

## Порядок операции

Обычная операция идёт в таком порядке:

1. Получить одно логическое время.
2. Обслужить TTL очереди и срок диагностики.
3. При необходимости изменить очередь.
4. Атомарно сохранить полный снимок и последнее наблюдавшееся время.
5. Записать только безопасную категорию результата в журнал.

Если SQLite-транзакция не завершилась, диск сохраняет предыдущий снимок. Текущий
объект узла переходит в `Failed`. Для продолжения нужен новый запуск и
восстановление последнего целого снимка.

## Профиль и диагностика

Обычные `start` и `start_with_clock` оставляют диагностику только в памяти.
Постоянный файл включается отдельными методами:

- `start_with_persistent_diagnostics`;
- `start_with_clock_and_persistent_diagnostics`.

База очереди и база диагностики должны иметь разные пути. Также нельзя
использовать SQLite-sidecar или lock-файл одного компонента как файл другого.

Рядом с базой очереди остаётся пустой файл с суффиксом `.lock`. При постоянной
диагностике такой же файл создаётся рядом с её базой. В них нет PID, пути или
времени. Эксклюзивные блокировки держат файловые дескрипторы процесса. Они
освобождаются при `stop`, внутренней ошибке или уничтожении `NodeRuntime`.

## Граница транспорта

Crate может создать ограниченную сессию, но не читает и не разбирает сетевые
кадры. Для приёма Envelope нужны ещё не зафиксированные маршрутные поля. Их формат
появится на Этапе 3, а не в этом пакете.

## Проверка на Arch Linux

Из корня репозитория:

```bash
export PATH="$HOME/.cargo/bin:$PATH"
cargo test -p lantern-node --all-targets --locked --offline
```

## Ограничения

- Это не запускаемый процесс и не сетевой узел.
- В коде нет сокетов, wire framing, рукопожатия и поиска соседей.
- В коде нет криптографии, ключей и открытого текста.
- Постоянная диагностика отключена по умолчанию.
- Файл диагностики раскрывает порядок событий и внутренние сроки удаления.
- База SQLite не зашифрована.
- Lock не защищает от вредоносного процесса с теми же правами на каталог.
- Процесс не пытается автоматически переоткрыть базу после ошибки.
- Падение между изменением in-memory очереди и коммитом SQLite оставит на диске
  предыдущий целый снимок. После перезапуска операцию можно повторить, а
  дедупликация не даст сохранить один ID дважды.
