# lantern-bridge

`lantern-bridge` связывает секретное криптографическое состояние с открытой
очередью узла. Ключи, пароль базы и открытый текст через этот слой в
`lantern-node` не передаются.

Исходящий Envelope сначала сохраняется вместе с новым состоянием сессии в
секретной базе. После этого мост переносит точную сохраненную копию в очередь и
удаляет запись outbox. Повторный запуск проверяет уже записанный Envelope и
завершает перенос без повторного шифрования.

Входящий Envelope сначала проверяется и расшифровывается криптографическим
слоем. Узел удаляет его из очереди только после сохранения нового состояния
сессии и истории. Если процесс остановился между этими действиями, повторный
запуск завершит удаление и не вернет текст второй раз.

Этот crate не устанавливает сетевые соединения и не делает Lantern безопасным
сам по себе. Независимого аудита пока не было.

Интеграционный тест `persistent_contacts` собирает два постоянных конечных
профиля через подписанные QR, SAS и два Olm-подтверждения. Relay в этом сценарии
имеет только открытую очередь. Пользовательский chat-текст пока не передаётся.

Тест `encrypted_three_node` создаёт один настоящий chat Envelope, проводит его
через две LAN-встречи и открывает профилем Bob. Relay получает только открытую
очередь и не имеет crypto API. Это happy path, а не полный негативный аудит.

Тест `public_api` меняет внешний ID, TTL, лимит переходов и ciphertext. Ни одна
ошибка не добавляет сообщение в историю и не сохраняет кандидатный ratchet.
Следующее корректное сообщение открывается, а его replay блокируется tombstone.

Окна остановки между двумя базами проверяются отдельно. Если очередь уже
содержит исходящий Envelope, повторный экспорт завершает outbox по точному
совпадению. Если история Bob уже закоммичена, повторная обработка возвращает
`Recovered` без текста и завершает создание tombstone.
