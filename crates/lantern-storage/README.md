# Lantern Storage

Статус: четвёртый рабочий пакет Этапа 2.

`lantern-storage` сохраняет полное состояние ограниченной очереди Lantern в
одном SQLite-файле. Crate зависит от `lantern-core`, но не знает ничего о сети,
интерфейсе пользователя и внутреннем устройстве будущей криптографии.

## Что хранится

В базе находятся три группы данных:

- канонический CBOR каждого Envelope;
- локальные поля маршрутизации, включая сроки, hop count и бюджет копий;
- ограниченные tombstone для защиты от быстрого возврата удалённых ID.

Также сохраняются версия схемы, лимиты очереди и последнее наблюдавшееся
настенное время. Лимиты являются частью формата базы. Открытие существующего
файла с другими лимитами завершается ошибкой.

`lantern-storage` не хранит журнал сообщений, контакты, приватные ключи и
пароли. В текущих тестах `protected_payload` содержит обычные заметные байты.
Это не шифротекст и не пользовательская переписка.

## Граница проверки

SQLite-ограничения отсекают очевидно неверные длины и диапазоны. После чтения
crate всё равно повторно проверяет данные через типы `lantern-core`:

- число активных записей и tombstone проверяется до создания больших списков;
- общий размер CBOR сверяется с локальной байтовой квотой;
- каждый Envelope проходит строгий CBOR-декодер;
- `message_id` в строке должен совпасть с ID внутри Envelope;
- маршрутный срок должен точно соответствовать `first_seen + remaining_ttl`;
- активная очередь не может содержать дубликаты и пересекаться с tombstone.

Повреждённая или неизвестная схема не исправляется автоматически. Операция
завершается безопасной категорией ошибки без вывода SQL, пути к файлу или
содержимого строки.

## Запись и восстановление

`save` заменяет предыдущий снимок одной SQLite-транзакцией. Если вставка или
commit завершаются ошибкой, SQLite откатывает транзакцию, а предыдущий снимок
остаётся доступен. Тест принудительно заполняет лимит страниц посередине записи
и проверяет этот результат после повторного открытия файла.

`load` читает и проверяет снимок, обслуживает TTL и tombstone, а затем
атомарно сохраняет результат обслуживания. Возвращённая очередь соответствует
уже сохранённому состоянию.

Сейчас база рассчитана на один процесс Lantern. Два процесса с отдельными
in-memory снимками могут последовательно перезаписать изменения друг друга.
Межпроцессная блокировка профиля появится вместе с процессом узла. До этого
один файл нельзя одновременно использовать в двух запущенных экземплярах.

## Настенное время и откат часов

Абсолютные секунды нужны только для восстановления после перезапуска. При
обычном ходе часов `load` удаляет записи, чей локальный срок уже наступил.

Если новое время меньше последнего сохранённого времени, определить честный
остаток TTL без доверенных часов нельзя. В этом случае crate:

1. сообщает `ClockRecovery::RollbackDetected`;
2. переводит все активные записи в `expired`;
3. создаёт для них ограниченные tombstone от нового текущего времени;
4. переносит существующим tombstone только срок, остававшийся на последней
   известной точке;
5. сохраняет пустую активную очередь и новую временную точку.

Это может удалить доставляемое сообщение слишком рано, то есть ухудшить
доступность. Зато откат часов не продлевает хранение на неопределённый срок.
Во время непрерывной работы будущий процесс узла должен использовать
монотонные часы. Этот crate не получает часы сам и не решает runtime-задачу.

## Настройки SQLite

Используется `rusqlite` 0.40.1 с bundled SQLite 3.53.2. Версии зафиксированы в
`Cargo.toml` и `Cargo.lock`. Системная SQLite не используется, чтобы сборки на
разных машинах не получили случайно разные возможности и исправления.

Основные ограничения:

- `STRICT`-таблицы и `application_id` Lantern;
- версия схемы `1` без автоматической миграции;
- основной файл не больше 128 MiB;
- одно SQLite-значение не больше 128 KiB;
- SQL-текст не больше 64 KiB;
- не больше 1000 активных Envelope и 2000 tombstone;
- режим журнала `DELETE`;
- `synchronous=EXTRA`;
- `secure_delete=ON`;
- `trusted_schema=OFF`;
- права `0600` на Unix.

Основной файл и уже существующие sidecar-файлы проверяются до открытия.
Символические ссылки отклоняются. Между проверкой пути и действиями SQLite
остаётся небольшое окно гонки. Локальный процесс с доступом от того же
пользователя всё ещё может вмешаться, поэтому это не sandbox и не защита
скомпрометированной учётной записи.

## Запуск на Arch Linux

```bash
sudo pacman -S --needed rustup base-devel

cd "/home/qual/Desktop/prog/Lantern proj"
export PATH="$HOME/.cargo/bin:$PATH"

rustup set profile minimal
rustup toolchain install 1.97.1 --component clippy,rustfmt

cargo fmt --all --check
cargo clippy --workspace --all-targets --all-features --locked -- -D warnings
cargo test --workspace --all-targets --all-features --locked
```

`base-devel` нужен для сборки bundled SQLite из исходного C-кода.

## Ограничения безопасности

Файл SQLite не зашифрован. Права `0600` мешают другим обычным пользователям
системы прочитать файл, но не защищают от владельца учётной записи, root,
вредоносной программы, резервной копии или физической кражи носителя.

Человек с копией текущей базы увидит Envelope, точные локальные сроки,
идентификаторы, размеры и tombstone. После будущего криптографического рубежа
payload должен стать шифротекстом, но маршрутные метаданные всё равно останутся
видимыми. Секретные поля потребуют отдельного решения о шифровании базы и
управлении ключом.

`secure_delete=ON` уменьшает количество удалённых значений в свободных
страницах обычного SQLite-файла. Это не гарантирует физическое стирание SSD,
снимков файловой системы, резервных копий и старых блоков журналирования.

Слой не аутентифицирует Envelope криптографически. Он обнаруживает структурные
повреждения и внутренние противоречия, но правильно сформированную замену
payload сможет обнаружить только будущий E2EE-компонент.

Решение, альтернативы и оставшиеся риски описаны в
[`docs/adr/0004-sqlite-queue-persistence.md`](../../docs/adr/0004-sqlite-queue-persistence.md).
