# Lantern Secret Storage

Статус: первый кодовый пакет секретного хранилища.

`lantern-secret-storage` пока выполняет только три задачи:

- создаёт и строго читает ограниченный `secrets.kdf`;
- проверяет парольную фразу до запуска KDF;
- получает 32-байтный ключ через Argon2id 1.3.

SQLCipher, Olm Account, Session, контакты, limiter, история и pending outbox в
crate ещё не добавлены. Полученный ключ остаётся закрытым типом и пока никуда не
передаётся.

## Формат

Заголовок является канонической CBOR map из семи полей, описанных в
[`SECRET_STORAGE.md`](../../SECRET_STORAGE.md). Декодер принимает только:

- версию формата 1;
- Argon2id 1.3;
- соль ровно 16 байт;
- 65536 KiB памяти;
- 3 прохода;
- 4 lanes;
- результат 32 байта.

Другие параметры не считаются настройкой пользователя. Они требуют новой
версии формата.

## Работа с секретами

`Passphrase` принимает от 16 до 128 скалярных значений Unicode и не больше 1024
байт UTF-8. Тип владеет строкой и обнуляет её байты при удалении. `DatabaseKey`
также обнуляет свой массив и не открывает байты через публичный API.

Argon2 запускается с явно выделенной памятью. После результата все 64 MiB
рабочих блоков обнуляются. Это уменьшает число остаточных копий, но не является
гарантией против swap, debugger, core dump или вредоносной программы.

Соль берётся через `getrandom` из генератора ОС. Ошибка генератора завершает
создание без небезопасного запасного варианта.

## Файл

На Unix файл создаётся через `create_new` с правами `0600`, `O_NOFOLLOW` и
`O_CLOEXEC`. Чтение отклоняет:

- символические ссылки;
- hard link;
- специальные и пустые файлы;
- доступ группы или остальных пользователей;
- файл больше 128 байт.

Проверяется уже открытый файловый дескриптор, а чтение дополнительно ограничено 129
байтами. Проверка каталога профиля и атомарное создание пары KDF + SQLCipher
остаются следующим уровнем.

Если запись или `fsync` завершаются ошибкой, функция возвращает ошибку и не
пытается удалить файл по имени. Такой файл считается следом незавершённого
создания и не должен открываться как готовый профиль.

## Почему SQLCipher ещё не подключён

`rusqlite 0.40.1` и `libsqlite3-sys 0.38.1` содержат SQLCipher 4.14.0. На 17
июля 2026 года upstream выпустил 4.17.0 с исправлениями обработки ошибок и
другими изменениями. Подключать 4.14.0 как будто это текущая проверенная версия
нельзя.

Мы не заменяем её самодельным шифрованием и не скачиваем непроверенный C-код из
`build.rs`. Следующий пакет должен выбрать воспроизводимый способ собрать и
проверить SQLCipher 4.17.0 либо более новую проверенную версию.

## Проверка на Arch Linux

```bash
sudo pacman -S --needed rustup base-devel

cd "/home/qual/Desktop/prog/Lantern proj"
export PATH="$HOME/.cargo/bin:$PATH"

cargo fmt --all --check
cargo clippy --workspace --all-targets --all-features --locked -- -D warnings
cargo test --workspace --all-targets --all-features --locked
```

Тесты включают официальный Argon2id-вектор RFC 9106, точный CBOR-вектор,
обрезанные и неканоничные header, границы пароля, права файла, symlink, hard
link и отсутствие секретов в `Debug` и ошибках.

## Ограничения

- Терминальный ввод пароля без отображения ещё не реализован.
- Каталог профиля и его права пока не проверяются этим crate.
- Сбой между записью заголовка и будущим созданием базы ещё не обрабатывается.
- Ключ пока не используется для шифрования данных.
- Обнуление памяти не является защитой от скомпрометированного процесса.
- Этот пакет не делает Lantern готовым к настоящей переписке.
