# lantern-time

`lantern-time` задаёт единые правила работы со временем во время запуска Lantern.
Компонент не хранит данные и не пишет события в журнал.

## Зачем нужен отдельный компонент

Очередь конвертов и диагностический журнал удаляют записи по TTL. Если каждый
компонент будет самостоятельно читать системные часы, один и тот же момент может
получить разные значения, а перевод часов назад способен продлить жизнь данным.

Компонент объединяет два источника из стандартной библиотеки Rust:

- `Instant` измеряет время от запуска процесса и не должен идти назад;
- `SystemTime` даёт календарное время, но может быть переведено назад или вперёд.

Результатом является логическая отметка в целых секундах, которая никогда не
уменьшается внутри одного процесса.

## Правило вычисления

Для каждого чтения выбирается максимум из трёх значений:

- последняя логическая опора плюс прошедшее от неё монотонное время;
- текущее системное время;
- предыдущее логическое время.

Если системные часы продвинули результат вперёд, это значение становится новой
опорой для `Instant`. Поэтому последующий возврат системных часов не замораживает
TTL.

Если наблюдаемое системное время стало меньше предыдущего, чтение получает статус
`WallClockRollbackDetected`. Оркестратор должен считать оставшийся TTL
неопределённым, завершить активные записи и очистить диагностический журнал.

## Границы

- Точность политики - одна секунда, как у текущих полей протокола и SQLite.
- Допустимы только значения от Unix epoch до `i64::MAX` секунд.
- Прыжок системных часов вперёд может удалить данные раньше срока.
- Поведение `Instant` во время сна устройства зависит от платформы. Обычно этот
  промежуток компенсирует `SystemTime`, но полностью решить проблему без
  платформенного источника времени нельзя.
- Компонент не делает Lantern безопасным и не заменяет тестирование всей системы.

