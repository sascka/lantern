# lantern-transport

`lantern-transport` задаёт общую ограниченную границу между Lantern и способами
доставки. Статус: восьмой рабочий пакет Этапа 2.

Crate работает только с непрозрачными кадрами. Он не знает структуру сообщения,
не получает ключи и не декодирует Envelope.

## Что реализовано

- один кадр ограничен 64 KiB;
- одно направление сессии ограничено 1024 кадрами и 64 MiB;
- меньшие локальные лимиты разрешены, увеличение выше границ v0.1 запрещено;
- вход читается только в буфер вызывающего кода размером не меньше 64 KiB;
- адаптер получает только остаток разрешённой байтовой квоты;
- пустой кадр, неверная длина, превышение квоты и ошибка адаптера завершают
  сессию;
- исходящий кадр проверяется до вызова адаптера;
- ошибки не содержат кадр, адрес соседа или текст ошибки ОС;
- точное число переданных байтов скрыто из `Debug`.

Лимиты применяются отдельно к приёму и отправке. Поэтому максимальная суммарная
передача одной сессии в обоих направлениях может достигать 128 MiB.

## Что должен сделать конкретный адаптер

`FrameTransport` получает или отправляет ровно один полный кадр за вызов.
Адаптер обязан проверить длину своего wire framing до чтения тела и не имеет
права выделять память по недоверенной длине, превышающей переданный буфер.

Текущий crate не может проверить внутренние выделения чужой реализации
адаптера. Конкретный `LanConnection` в `lantern-lan` уже реализует
`FrameTransport`: он проверяет 4-байтную длину до тела и читает только в
переданный ограниченный буфер.

## Что пока не реализовано

- служебные типы LAN-синхронизации;
- обнаружение соседей и сетевые адреса;
- список известных контейнеров и запрос недостающих;
- асинхронный runtime и отмена операций;
- квота по IP или физическому соседу, потому что транспорт не видит адрес;
- шифрование соединения между соседями.

Повторное подключение создаёт новую сессию и новую квоту. Поэтому этот crate сам
по себе не ограничивает частые подключения. `lantern-lan` дополняет его
ограниченным in-memory окном по IP на стороне listener.

## Запуск на Arch Linux

Property-тесты генерируют разные длины кадров и проверяют, что сессия не
выходит за квоты кадров и байтов. Входные данные в этих тестах не содержат
секретов.

Из корня проекта:

```bash
export PATH="$HOME/.cargo/bin:$PATH"
cargo test -p lantern-transport --all-targets --all-features --locked --offline
```
