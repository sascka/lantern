# Lantern Diagnostics

Статус: пятый рабочий пакет Этапа 2.

`lantern-diagnostics` предоставляет безопасный ограниченный журнал технических
событий. Он не зависит от сети, хранилища, криптографии и других crate Lantern.
Сторонних зависимостей у него нет.

## Зачем нужен отдельный тип

Обычный logger принимает строку. В такую строку легко случайно подставить текст
сообщения, полный ID, путь к базе или содержимое ошибки сторонней библиотеки.

В `DiagnosticEvent` строки и байтовые массивы отсутствуют. Событие содержит
только:

- один код из закрытого списка;
- одну общую категорию результата;
- ограниченный счётчик объектов;
- грубую категорию размера;
- грубую категорию длительности.

Точный размер и длительность превращаются в категорию при создании события и
после этого не сохраняются. Абсолютное время используется только внутри
журнала для удаления старых записей. `DiagnosticRecord`, итератор и `Debug` его
не возвращают.

## Ограничения журнала

Значения по умолчанию:

- не больше 2048 записей;
- не больше 64 KiB логического размера;
- хранение не дольше 7 суток.

Конфигурацию можно уменьшить. Общие верхние границы API:

- 10 000 записей;
- 320 000 логических байтов, то есть примерно 312,5 KiB;
- 7 суток хранения;
- 10 000 объектов в одном агрегированном событии.

Одна запись учитывается как 32 логических байта. Это стабильная квота API, а не
точное измерение памяти allocator. Внутренний `VecDeque` и служебные поля
занимают дополнительную ограниченную память.

При заполнении удаляется самая старая запись. На точной границе срока запись
уже считается истёкшей. События сохраняют только локальный последовательный
номер, поэтому виден порядок, но не абсолютное время.

Очистка по сроку выполняется лениво при `record`, `maintain` или `view`.
Публично прочитать записи можно только через `view(now)`, который сначала
обслуживает срок и откат часов. Пока процесс полностью бездействует, истёкшие
данные физически могут оставаться в памяти. Будущий runtime должен читать
`lantern-time` и вызывать `maintain` периодически.

## Неверные часы

Время передаёт вызывающий компонент. Сам crate не читает системные часы.
Одно логическое чтение `lantern-time` должно использоваться для журнала и
очереди в пределах общей операции обслуживания.

Если новое значение меньше предыдущего, весь журнал очищается. Очистка может
удалить полезную диагностику раньше срока, но откат часов не продлевает её
хранение. Результат обслуживания отдельно сообщает об обнаруженном откате,
даже если журнал был пуст.

Скачок часов вперёд приведёт к раннему удалению записей. Это допустимый отказ
доступности для необязательной диагностики.

## Чего этот пакет не делает

- не устанавливает глобальный logger;
- не пишет в stdout или stderr;
- не создаёт файлы;
- не отправляет данные по сети;
- не принимает произвольные строки и поля;
- не подключён автоматически к `lantern-core` или `lantern-storage`;
- не переживает завершение процесса.

Будущий процесс узла должен явно создать журнал и преобразовывать результаты
операций в разрешённые коды. Ошибки внешней библиотеки нельзя передавать в
журнал целиком.

Лимит будущего файла из `PRIVACY.md` остаётся отдельным и равен 10 MiB. Он не
используется как разрешение занять столько памяти.

Файловое логирование отложено. Перед ним нужно отдельно определить безопасный
формат, восстановление после обрыва записи, права файла и удаление по сроку. Оно
по-прежнему должно быть выключено по умолчанию.

## Ограничения безопасности

Закрытый тип уменьшает риск случайной утечки, но не мешает злонамеренному
вызывающему коду кодировать информацию последовательностью событий или
счётчиков. Скомпрометированный процесс также может прочитать память журнала.

Коды, порядок, счётчики и категории размера всё равно являются метаданными.
Они могут показать активность узла и примерный объём операций. Поэтому журнал
ограничен, не содержит личности и не должен передаваться автоматически.

Решение и альтернативы описаны в
[`docs/adr/0005-bounded-safe-diagnostics.md`](../../docs/adr/0005-bounded-safe-diagnostics.md).
