# Lantern Core

Статус: доменное ядро после восьмого рабочего пакета Этапа 2.

`lantern-core` содержит ограниченные доменные типы, которые не зависят от сети,
хранилища и криптографической библиотеки.

Сейчас crate умеет:

- проверять логические поля Envelope версии 1;
- ограничивать TTL, hop limit, priority и размер непрозрачного payload;
- хранить `message_id` и `recipient_hint` как отдельные типы фиксированного
  размера;
- создавать локальную маршрутную запись для источника и Relay;
- ограничивать недоверенные маршрутные счётчики локальными максимумами;
- проверять переходы состояния контейнера;
- скрывать полные идентификаторы, подсказку получателя, payload и точные
  локальные времена из `Debug`;
- детерминированно кодировать Envelope в CBOR;
- строго декодировать один Envelope из недоверенного среза байтов;
- отклонять неизвестные поля, повторы ключей, неканоничные значения,
  неопределённые длины и данные после Envelope;
- проверять общий лимит 64 KiB до разбора и копировать payload только после
  проверки структуры;
- хранить Envelope и локальные маршрутные записи в ограниченной in-memory
  очереди;
- одновременно ограничивать очередь 1000 записями и 64 MiB канонических
  CBOR-байтов;
- отклонять активные повторы без замены сохранённого объекта;
- детерминированно вытеснять старые записи по FIFO;
- хранить до 2000 ограниченных tombstone после открытия, истечения или
  вытеснения;
- возвращать удалённые записи и изменения tombstone слою хранения;
- строго восстанавливать уже проверенный снимок без скрытого вытеснения;
- консервативно перевести все активные записи в `expired` после обнаруженного
  отката часов.

## Что пока не реализовано

- генерация случайного `message_id`;
- создание `recipient_hint`;
- E2EE и работа с настоящим текстом;
- сетевой адаптер, wire framing и LAN.

Массивы в тестах являются очевидными тестовыми значениями. Они не являются
ключами, шифротекстом или защищённой пользовательской перепиской.

## Граница недоверенного ввода

`decode_envelope` сначала отклоняет пустой вход и вход больше 64 KiB. Затем он
проверяет фиксированную схему из семи полей, каноничность заголовков, типы,
длины, диапазоны и конец входа. Byte string читаются как ссылки на исходный
срез. Новый `Vec<u8>` для payload создаётся только после этих проверок.

Декодер не является общей реализацией произвольного CBOR. Он намеренно
принимает только формат Envelope версии 1 из `PROTOCOL.md`. Решение и
альтернативы описаны в
`docs/adr/0002-strict-cbor-codec.md`.

## Очередь и дедупликация

Очередь принимает только уже созданные типы `Envelope` и `LocalRouteRecord` в
состоянии `stored`. Она не разбирает сетевые байты повторно. Несовпадение
`message_id` между ними отклоняется.

Активный `message_id` и действующий tombstone проверяются до вытеснения.
Дубликат не меняет существующий Envelope, не расходует квоту и не продлевает
tombstone. Если места не хватает, удаляются самые старые записи по
`first_seen_at`, при равенстве используется `message_id`.

Байтовая квота считает полное каноническое CBOR-представление Envelope без
создания временной копии payload. Она не измеряет точный расход allocator.
Служебные структуры добавляют память сверх 64 MiB, но их число отдельно
ограничено.

Решение, альтернативы и ограничения описаны в
`docs/adr/0003-bounded-queue-and-deduplication.md`.

## Время

`first_seen_at` и `local_deadline` представлены локальными секундами в одной
шкале времени. При создании и восстановлении записи проверяются переполнение и
точное равенство `local_deadline = first_seen_at + remaining_ttl`.

Отдельный crate `lantern-storage` сохраняет эти значения вместе с последним
наблюдавшимся настенным временем. Обычный перезапуск обслуживает достигшие
сроки. При откате часов все активные записи удаляются раньше срока, чтобы не
допустить продления TTL. Отдельный `lantern-time` выдаёт неуменьшающееся время
для обслуживания во время работы и сообщает об откате настенных часов. Процесс
узла должен передавать одно чтение времени всем обслуживаемым компонентам.

## Запуск на Arch Linux

Если `rustup` ещё не установлен:

```bash
sudo pacman -S --needed rustup
```

Затем:

```bash
cd "/home/qual/Desktop/prog/Lantern proj"

export PATH="$HOME/.cargo/bin:$PATH"
rustup set profile minimal
rustup toolchain install 1.97.1 --component clippy,rustfmt

cargo fmt --all --check
cargo clippy --workspace --all-targets --all-features --locked -- -D warnings
cargo test --workspace --all-targets --all-features --locked
```

Файл `rust-toolchain.toml` фиксирует Rust 1.97.1. Собственная сторонняя
зависимость этого crate - `minicbor` 2.2.2 с feature `alloc`. Версия закреплена
точно, default features отключены, транзитивных зависимостей нет. Лицензия
зависимости - `BlueOak-1.0.0`, её полный текст сохранён в
`LICENSES/BlueOak-1.0.0.txt`. Зависимости отдельного слоя хранения перечислены
в корневом `THIRD_PARTY_NOTICES.md`.

## Ограничения безопасности

Этот пакет не обеспечивает конфиденциальность или целостность сообщения.
`protected_payload` является только непрозрачным массивом байтов. Relay с
доступом к памяти текущей реализации увидит весь Envelope и локальную
маршрутную запись.

Постоянная база реализована отдельным crate и не шифруется. Её кража раскрывает
Envelope и локальные маршрутные метаданные. Неканоничное или повреждённое
представление CBOR отклоняется, но изменение правильно сформированного payload
пока нельзя отличить от нового контейнера. Это сможет сделать только будущий
криптографический компонент.

Очередь ограничивает активные записи, логические CBOR-байты и tombstone. Это не
мешает спамеру с новыми ID вытеснять полезные сообщения и не устанавливает
лимит одного соседа. Неверное время может привести к раннему истечению. Сам
`lantern-core` часы не читает и зависит от правильного вызова оркестратором.
Отдельный `lantern-transport` ограничивает одну сессию, но не меняет правила
очереди и не декодирует Envelope.

В коде нет логов, сетевых обращений, `unsafe` и секретных данных. Декодер и
очередь покрыты точечными, негативными и property-тестами с фиксированным
seed. Fuzzing и независимый аудит ещё не проводились.
