# Результаты проверки vodozemac 0.10.0

Дата проверки: 17 июля 2026 года.

Окружение: Linux x86_64, Rust 1.97.1, `vodozemac 0.10.0`, default features
отключены.

## Результат тестов

Все восемь compatibility-тестов прошли.

- Alice создаёт сессию, пока Bob офлайн, используя опубликованный one-time key.
- Bob принимает первое pre-key сообщение и отвечает через ту же сессию.
- Изменение каждого отдельного байта нормального сообщения либо ломает формат,
  либо приводит к отказу расшифрования.
- После неудачной расшифровки исходное сообщение всё ещё принимается. Проверка
  не обнаружила изменения состояния получателя при неверном MAC.
- Повтор уже принятого сообщения отклоняется, а следующее новое сообщение
  принимается.
- 32 сообщения расшифровываются в полностью обратном порядке.
- После получения сообщения с индексом 41 сообщение с индексом 0 уже нельзя
  расшифровать. Оставшиеся 40 пропущенных сообщений принимаются.
- Зашифрованные Account и Session pickle восстанавливаются с правильным ключом
  и отклоняются с неправильным.

Эти проверки подтверждают наблюдаемое поведение одной версии библиотеки. Они не
являются доказательством корректности протокола и не заменяют опубликованные
тестовые векторы или аудит.

## Найденная граница Debug

`Session` не показал plaintext в проверенном `Debug`, а ошибки неверного MAC не
содержали plaintext или ciphertext.

`InboundCreationResult` устроен иначе. Он имеет автоматически созданный
`Debug`, а одно из его публичных полей содержит расшифрованный первый plaintext.
В результате форматирование `InboundCreationResult` через `{:?}` раскрывает
сообщение в виде массива байтов.

Это не криптографическая уязвимость Olm, но опасная граница API. Если кандидат
будет принят, результат создания входящей сессии нужно немедленно разделять на
Session и plaintext. Сам объект нельзя передавать в универсальные ошибки,
tracing spans или другие журналы.

## Зависимости и лицензии

`vodozemac` подключён только как dev-зависимость отдельного workspace.
Основные `Cargo.toml` и `Cargo.lock` Lantern не изменены.

Отдельный lock-файл содержит 81 сторонний пакет для всех записанных целевых
платформ. В тестовом дереве Linux активно 74 сторонних пакета. Повторяющихся
версий в активном дереве `cargo tree --duplicates` не нашёл.

В metadata нет пакетов с неизвестной лицензией. Используются разрешительные
варианты Apache-2.0, MIT, BSD, Unlicense, Unicode-3.0 и Apache-2.0 с
LLVM-exception. Видимого конфликта с MPL-2.0 у тестового crate нет, но это не
юридическое заключение. Перед переносом зависимости в рабочий crate понадобятся
полные copyright-уведомления и тексты всех выбранных лицензий.

Точные версии находятся в `Cargo.lock`. Дерево и сведения о лицензиях можно
повторно получить командами:

```bash
cargo tree --locked --edges normal,dev,build
cargo tree --locked --duplicates
cargo metadata --locked --format-version 1
```

## RustSec

`cargo-audit 0.22.2` загрузил базу из 1160 advisories и проверил отдельный
`Cargo.lock`. Известные уязвимости не найдены. Во время проверки Cargo не смог
дополнительно обновить локальный crates.io index, но advisory database была
получена перед сканированием.

Отсутствие записи в RustSec не означает, что библиотека безопасна. База содержит
только уже опубликованные advisories.

## Что эксперимент не проверяет

- подпись и проверку contact bundle;
- безопасное хранение ключа pickle;
- ограничение размера входящего ciphertext до вызова библиотеки;
- работу нескольких устройств одного контакта;
- обновление one-time key и fallback key;
- смену `recipient_hint`;
- совместимость с libolm или другим клиентом;
- поведение после кражи живого состояния процесса;
- forward secrecy и post-compromise recovery отдельными тестовыми векторами;
- устойчивость всей интеграции Lantern.

## Вывод

`vodozemac 0.10.0` технически подходит для дальнейшего прототипирования
личной асинхронной сессии. Окончательный выбор всё ещё нельзя сделать без
отдельного решения о 64-битном MAC Olm v1, окне в 40 ключей и правилах
аутентификации contact bundle.
