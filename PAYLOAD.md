# Защищённый payload Lantern v0.1

Статус: спецификация для `Lantern v0.1 experimental preview`.

Этот документ определяет двоичную оболочку Olm и plaintext, который Lantern
передаёт в Olm. Он не определяет сетевое соединение, пользовательский интерфейс
или способ хранения приватных ключей.

В v0.1 `protected_payload` используется только для:

- короткого личного сообщения;
- двух подтверждений добавления контакта;
- четырёх сообщений смены `recipient_hint`.

Вложений, реакций, групп, read receipt, typing indicator, команд удалённого
управления и расширяемых пользовательских полей в этом формате нет.

## Два уровня формата

`protected_payload` состоит из двух разных уровней:

1. Открытая бинарная оболочка хранит тип Olm message и полные байты, созданные
   `OlmMessage::to_parts()`.
2. Внутренний канонический CBOR целиком шифруется Olm и появляется только после
   успешного `decrypt`.

Relay видит оболочку, но не должен видеть внутренний тип сообщения, текст или
служебные значения. Тип Olm message остаётся открытым, потому что принимающая
сторона должна отличить первое pre-key message от обычного сообщения.

## Ограничения размеров

Для crypto-профиля v0.1 действуют более узкие пределы, чем общий предел
Envelope:

- полный `protected_payload` - от 1 байта до 8192 байт;
- поле `olm_message` внутри оболочки - от 1 байта до 8183 байт;
- расшифрованный внутренний CBOR - от 1 байта до 4155 байт;
- открытый текст пользователя - от 1 до 4096 байт UTF-8.

Общий Lantern Core по-прежнему принимает непрозрачный payload размером до 63
KiB. Crypto-слой обязан отклонить значение больше 8192 байт до вызова
`OlmMessage::from_parts()` или выделения дополнительных буферов.

Предел 8192 байта оставляет запас для заголовков Olm v1 и pre-key message при
максимальном пользовательском тексте. До реализации этот расчёт должен быть
подтверждён тестом на точной версии `vodozemac 0.10.0`. Увеличивать предел без
нового решения нельзя.

## Открытая оболочка protected_payload

Оболочка является каноническим CBOR map из трёх полей:

| Ключ | Поле | Тип | Ограничение |
| --- | --- | --- | --- |
| 0 | `protected_payload_version` | unsigned integer | только `1` |
| 1 | `olm_message_type` | unsigned integer | `0` или `1` |
| 2 | `olm_message` | byte string | от 1 до 8183 байт |

Значение `olm_message_type` берётся без преобразования из
`OlmMessage::to_parts()`:

| Значение | Вариант vodozemac | Разрешённое использование |
| --- | --- | --- |
| 0 | `OlmMessage::PreKey` | только `contact_initiator_confirm` |
| 1 | `OlmMessage::Normal` | все остальные внутренние типы |

В поле 2 записываются полные bytes из `to_parts()`, а не только результат
`OlmMessage::message()`. Потеря metadata Olm сделала бы расшифрование
невозможным.

Base64 и serde-оболочка `vodozemac` здесь не используются. Они увеличили бы
размер и добавили второй текстовый формат без пользы для бинарного Envelope.
При чтении вызывается `OlmMessage::from_parts(type, bytes)`.

Оболочка считается недоверенной. Декодер до vodozemac проверяет:

- общий предел 8192 байта;
- map точной длины 3;
- ключи `0`, `1`, `2` в возрастающем порядке;
- канонические заголовки и определённые длины;
- отсутствие неизвестных, повторных и пропущенных полей;
- отсутствие тегов, float, текста, массивов и вложенных map;
- отсутствие данных после map;
- разрешённые version, type и длину byte string.

Неизвестная версия отклоняется до попытки разобрать остальные поля как v1.

## Общие поля внутреннего сообщения

После расшифрования plaintext должен быть одним каноническим CBOR map. Первые
девять полей одинаковы для всех типов:

| Ключ | Поле | Тип | Ограничение |
| --- | --- | --- | --- |
| 0 | `inner_version` | unsigned integer | только `1` |
| 1 | `content_type` | unsigned integer | от `0` до `6` |
| 2 | `protected_payload_version` | unsigned integer | только `1` |
| 3 | `envelope_protocol_version` | unsigned integer | только `1` |
| 4 | `message_id` | byte string | ровно 16 байт |
| 5 | `recipient_hint` | byte string | ровно 16 байт |
| 6 | `ttl_seconds` | unsigned integer | от `60` до `604800` |
| 7 | `max_hops` | unsigned integer | от `1` до `16` |
| 8 | `priority` | unsigned integer | только `0` |

Поля 2-8 сравниваются с открытой оболочкой и Envelope:

- внутренний `protected_payload_version` равен версии оболочки;
- `envelope_protocol_version` равен `protocol_version` Envelope;
- `message_id`, `recipient_hint`, `ttl_seconds`, `max_hops` и `priority` должны
  побайтно или численно совпасть с Envelope.

Сравнение выполняется до показа текста, изменения контакта или постоянного
сохранения нового ratchet state. Несовпадение любого поля отклоняет весь
Envelope.

Во внутреннем сообщении нет имени отправителя, имени контакта, session ID,
публичного ключа, IP-адреса и абсолютного времени. Контакт определяется только
по локальной проверенной сессии, выбранной через `recipient_hint`.

## Типы содержимого

| Значение | Имя | Назначение |
| --- | --- | --- |
| 0 | `chat_message` | короткий текст пользователя |
| 1 | `contact_initiator_confirm` | первое подтверждение Bob |
| 2 | `contact_receiver_confirm` | ответ Alice |
| 3 | `hint_proposal` | предложение нового входящего hint |
| 4 | `hint_accepted` | принятие предложения |
| 5 | `hint_committed` | разрешение перейти на новый hint |
| 6 | `hint_final` | подтверждение работы нового hint |

Неизвестный тип не сохраняется для будущего разбора и не передаётся
пользовательскому коду. Для нового типа понадобится новая `inner_version`.

## chat_message

Map содержит десять полей: общие ключи `0`-`8` и поле:

| Ключ | Поле | Тип | Ограничение |
| --- | --- | --- | --- |
| 9 | `text` | text string | от 1 до 4096 байт UTF-8 |

Максимальная каноническая длина внутреннего `chat_message` равна 4155 байтам.

Текст не нормализуется и после расшифрования сохраняет точные UTF-8 bytes.
Декодер отклоняет Unicode control characters. В v0.1 сообщение является одной
логической строкой, поэтому переводы строк, tab, NUL и escape запрещены.

Текст остаётся недоверенным даже после успешной криптографической проверки.
CLI не выполняет его как команду, не разбирает как Markdown и не передаёт
напрямую терминалу без безопасного отображения.

Отправлять `chat_message` можно только активному проверенному контакту через
`OlmMessage::Normal`. Значения TTL и max hops выбираются в пределах Envelope и
копируются без изменений во внутренние поля.

## Подтверждения контакта

Оба подтверждения являются map из 13 полей. После общих ключей идут:

| Ключ | Поле | Тип | Ограничение |
| --- | --- | --- | --- |
| 9 | `invitation_id` | byte string | ровно 16 байт |
| 10 | `response_id` | byte string | ровно 16 байт |
| 11 | `invitation_secret` | byte string | ровно 32 байта |
| 12 | `response_secret` | byte string | ровно 32 байта |

Оба ID и оба secret должны точно совпасть с текущим проверенным обменом из
`CONTACTS.md`. Подтверждение с данными другого или отменённого обмена
отклоняется.

Для служебных подтверждений фиксируются внешние и внутренние значения:

- `ttl_seconds = 604800`;
- `max_hops = 2`;
- `priority = 0`.

`contact_initiator_confirm`:

- создаётся Bob после успешной SAS-проверки;
- адресуется на `inbound_recipient_hint` Alice из invite;
- является единственным разрешённым plaintext внутри `OlmMessage::PreKey`;
- принимается Alice только в текущем состоянии `sas_verified`;
- создаёт входящую Olm v1 session только с ожидаемым Curve25519 key Bob.

`contact_receiver_confirm`:

- создаётся Alice после полной проверки первого подтверждения;
- адресуется на `inbound_recipient_hint` Bob из response;
- всегда находится в `OlmMessage::Normal`;
- принимается Bob только для ожидающей исходящей сессии;
- переводит контакт Bob в `active` после атомарного сохранения.

Alice может считать локальный контакт активным после атомарного сохранения
своего ответа. Если следующее сообщение Alice придёт Bob раньше ответа из-за
перестановки, Bob временно откладывает его до обработки подтверждения.

Каждая сторона хранит только один неизменяемый Envelope подтверждения и
повторно передаёт те же bytes. После истечения его TTL контактный обмен
отменяется и начинается заново с новыми QR. Повторное шифрование подтверждения
в той же попытке запрещено, потому что оно без необходимости продвинуло бы
ratchet.

Максимальная каноническая длина подтверждения равна 161 байту.

## Сообщения смены recipient_hint

Все четыре типа являются map из 11 полей. После общих ключей идут:

| Ключ | Поле | Тип | Ограничение |
| --- | --- | --- | --- |
| 9 | `generation` | unsigned integer | от `1` до `u32::MAX` |
| 10 | `new_recipient_hint` | byte string | ровно 16 байт |

Для них фиксируются:

- `ttl_seconds = 604800`;
- `max_hops = 2`;
- `priority = 0`;
- `olm_message_type = 1`;
- только активная проверенная сессия;
- одно текущее поколение и одно активное предложение.

Значения generation и new hint повторяются в каждом шаге без изменения.
Получатель сверяет их с локальным состоянием до перехода:

1. `hint_proposal` адресуется на текущий входящий hint собеседника и предлагает
   следующий входящий hint отправителя.
2. `hint_accepted` адресуется на старый текущий hint владельца предложения и
   возвращает те же generation и new hint.
3. `hint_committed` снова адресуется на текущий hint собеседника и разрешает ему
   начать использовать новое значение.
4. `hint_final` адресуется уже на `new_recipient_hint`. Для этого типа ключи 5 и
   10 внутреннего map должны быть равны.

Первый корректный `chat_message` или другой разрешённый normal message на новом
hint также завершает переход. Правила срока старого hint и пределов состояния
находятся в `CONTACTS.md`.

Повтор того же шага с теми же generation и hint идемпотентен. То же поколение с
другим hint, пропуск поколения или сообщение из невозможного состояния
отклоняются. Одновременно существует только один Envelope текущего шага. После
его истечения разрешена одна замена с теми же generation и hint, как описано в
`CONTACTS.md`.

Максимальная каноническая длина сообщения смены hint равна 79 байтам.

## Порядок входящей обработки

Все входящие bytes считаются недоверенными. Обработка выполняется в таком
порядке:

1. Строгий Lantern Core проверяет Envelope и общий предел 64 KiB.
2. Crypto-слой отклоняет `protected_payload` больше 8192 байт.
3. Строгий декодер проверяет открытую оболочку без вызова Olm.
4. По локальной таблице hint выбирается не более одного контакта или текущего
   контактного обмена.
5. Проверяются состояние контакта, разрешённый Olm type и общий лимит
   криптографических попыток.
6. Из последнего сохранённого pickle создаётся отдельное кандидатное состояние
   Account или Session.
7. `OlmMessage::from_parts()` разбирает полные bytes, после чего кандидат
   выполняет decrypt.
8. Результат ограничивается 4155 байтами и проходит строгий внутренний CBOR
   decoder.
9. Все общие поля сравниваются с оболочкой и Envelope.
10. Проверяются type-specific поля и допустимый переход состояния.
11. Новое ratchet state, дедупликация и изменение контакта сохраняются одной
    атомарной операцией секретного хранилища.
12. Только после успешной записи текст можно передать безопасному интерфейсу, а
    служебное действие считать выполненным.

Если шаг после decrypt завершается ошибкой, кандидатное состояние уничтожается.
Сохранённые Account, Session и one-time key остаются без изменений. Нельзя
расшифровывать прямо на единственном живом объекте и пытаться вручную откатить
отдельные поля ratchet.

Это особенно важно для первого pre-key message. `create_inbound_session()`
удаляет использованный one-time key после успешной расшифровки. Relay может
изменить открытый `message_id`, оставив Olm bytes без изменения. Внутренняя
копия обнаружит подмену, а кандидатное состояние не позволит ей уничтожить
ключ для настоящего Envelope.

## Временно отложенное сообщение

Корректное сообщение может прийти раньше причинно связанного подтверждения. Оно
не считается повреждённым и не должно навсегда менять ratchet.

В таком случае:

- кандидатное состояние отбрасывается;
- plaintext не сохраняется и не показывается;
- Envelope остаётся в ограниченной очереди до следующего изменения состояния;
- повторная попытка не выполняется в бесконечном цикле;
- на один контакт планируется не более 32 отложенных криптографических
  кандидатов, что совпадает с проверяемым окном перестановки Olm v1;
- TTL и общая квота очереди продолжают действовать.

После подходящего перехода состояния отложенные Envelope проверяются повторно в
порядке локального приёма. Если состояние так и не появилось, они истекают
обычным образом.

## Порядок исходящей обработки

1. Проверяется активность контакта или ожидаемый служебный переход.
2. Генерируется случайный `message_id`, выбирается текущий outbound hint и
   создаются все поля Envelope.
3. Из тех же значений кодируется один канонический внутренний map.
4. От последнего сохранённого Session создаётся кандидат и выполняется encrypt.
5. Полные parts Olm кодируются в строгую открытую оболочку.
6. Проверяются пределы 4155, 8183 и 8192 байта.
7. Новый зашифрованный Session pickle и полный неизменяемый Envelope попадают в
   pending outbox секретного хранилища одной атомарной записью.
8. После commit Envelope идемпотентно копируется в открытую маршрутную очередь.

При сбое открытой очереди pending outbox сохраняет те же ciphertext и
`message_id` для повторной вставки. Нельзя снова вызывать encrypt для того же
сообщения. Атомарная граница между секретным хранилищем и существующей открытой
SQLite-очередью зафиксирована как transactional outbox в `SECRET_STORAGE.md` и
ADR 0017.

## Ошибки и диагностика

Ни один тип ошибки или `Debug` не содержит:

- внутренний или пользовательский plaintext;
- ciphertext или полные bytes оболочки;
- Account и Session pickle;
- ключи, secret, ID и `recipient_hint`;
- точный ratchet index;
- исходную ошибку vodozemac, если её поля раскрывают ключ или ciphertext.

Наружу возвращается закрытая категория: malformed wrapper, unknown contact,
rate limited, decryption rejected, inner rejected, state deferred или storage
failed. Категория для удалённого узла не должна позволять отличить неверный MAC
от правильного plaintext с несовпавшим внешним полем.

`OlmMessage`, `InboundCreationResult` и plaintext buffer запрещено форматировать
через `Debug`. Plaintext buffer очищается настолько, насколько это позволяет
выбранная реализация и её зависимости, но проект не обещает надёжное удаление
всех копий из памяти без отдельной проверки.

## Потеря, повтор и перестановка

- Транспорт повторяет уже сохранённый Envelope, а не шифрует сообщение заново.
- Повтор с тем же `message_id` отсекается очередью и tombstone до crypto при
  наличии записи.
- Повтор Olm ciphertext с новым внешним ID не должен продвинуть сохранённый
  ratchet при отказе внутреннего сравнения.
- Перестановка до 32 последовательных сообщений проверяется интеграционными
  тестами.
- Сообщение за пределами окна может быть потеряно, но не должно давать неверный
  принятый plaintext.
- Успешная расшифровка не становится видимой до атомарного сохранения ratchet и
  отметки о принятии.
- Сбой после commit может привести к повторной локальной обработке Envelope,
  но сохранённая дедупликация не позволяет второй раз показать текст.

## Время устройства

Во внутреннем формате нет timestamp. Неверные часы не участвуют в ключах,
проверке контакта или сравнении полей.

Внутренний TTL должен совпасть с внешним, а фактический локальный срок считает
существующая модель времени Lantern. Откат часов может консервативно удалить
Envelope или уходящий hint и сорвать доставку, но не продлевает криптографическое
состояние и не меняет plaintext.

## Обязательные тесты реализации

- точные канонические vectors всех семи внутренних типов;
- точные vectors оболочки для pre-key и normal Olm message;
- границы 4096, 4155, 8183 и 8192 байта;
- пустой текст, неверный UTF-8 и каждый запрещённый control character;
- неизвестные, повторные, пропущенные, переставленные и неканоничные поля;
- все обрезанные префиксы оболочки и внутренних maps;
- изменение каждого внешнего поля при неизменном ciphertext;
- несовпадение версии оболочки и внутренней версии;
- запрет normal для первого подтверждения и pre-key для остальных типов;
- неверные ID, secret, generation, hint и состояние контакта;
- точные фиксированные параметры служебных Envelope;
- rollback Account после успешного Olm decrypt и неуспешного inner validation;
- rollback Session после успешного decrypt и несовпадения Envelope;
- сбой каждой постоянной записи до показа plaintext;
- восстановление pending outbox без повторного encrypt;
- повтор, потеря и все проверяемые перестановки до 32 сообщений;
- квота отложенных кандидатов и отсутствие активного retry loop;
- отсутствие контрольных секретов во всех ошибках, `Debug`, stdout, stderr и
  файлах диагностики;
- property-тесты обоих строгих декодеров;
- fuzzing оболочки и внутреннего CBOR до пользовательской демонстрации.

## Риски и ограничения

- Olm v1 сохраняет известное ограничение 64-битного MAC из ADR 0013.
- Тип pre-key или normal и размер ciphertext видны Relay.
- Без padding размеры разных внутренних типов можно приблизительно различать.
- Изменённый внешний Envelope может вызвать дорогую, но отклонённую попытку
  decrypt. Постоянный limiter из `CRYPTO_LIMITS.md` ещё не реализован.
- Кандидатное состояние и pending outbox требуют зашифрованного атомарного
