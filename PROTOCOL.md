# Протокол Lantern v0.1

Статус: черновик Этапа 0.

Этот документ описывает логическую структуру контейнера и его локальные
состояния. Здесь пока нет сетевого рукопожатия и конкретной криптографической
схемы. Их нельзя фиксировать до отдельного выбора библиотек и модели ключей.

## Задача контейнера

Контейнер переносит одно защищённое сообщение между узлами. После создания он
не меняется. Alice, Relay и Bob передают одинаковый набор байтов.

Неизменяемый контейнер проще проверять и дедуплицировать. Счётчики конкретного
маршрута хранятся рядом с ним в локальной записи узла, а не переписываются
внутри контейнера.

```text
прикладное сообщение
        |
криптографическое ядро
        |
неизменяемый Envelope
        |
локальная запись маршрутизации
        |
LAN-транспорт
```

Криптографическое ядро не открывает секреты транспорту. Транспорт не знает
структуру прикладного сообщения.

## Формат сериализации

Для v0.1 выбираем CBOR с детерминированным представлением.

Почему CBOR подходит:

- он компактнее JSON;
- двоичные поля не нужно кодировать в Base64;
- формат имеет стандартные правила детерминированного кодирования;
- для Rust и Python есть готовые реализации;
- схему можно описать отдельно от конкретного языка.

Недостатки:

- схема не поставляется вместе с форматом, как в Protocol Buffers;
- библиотеку нужно настроить на строгий разбор;
- не все CBOR-кодировщики создают одинаковые байты по умолчанию.

Protocol Buffers был альтернативой. Он даёт хорошую генерацию типов, но добавит
отдельный компилятор схем и свои правила работы с неизвестными полями. Для
небольшого фиксированного контейнера v0.1 это лишняя сложность.

Мы не используем собственный бинарный формат. Самодельный парсер недоверенных
данных добавит слишком много мест для ошибок.

## Правила CBOR для v0.1

- верхний объект - map;
- ключи - только небольшие беззнаковые целые числа;
- длины map и byte string всегда указываются заранее;
- неопределённые длины запрещены;
- числа кодируются в самом коротком допустимом виде;
- ключи идут в числовом порядке;
- повторяющиеся ключи запрещены;
- теги, числа с плавающей точкой и текстовые ключи запрещены;
- неизвестное поле в версии 1 приводит к отказу;
- неизвестная версия приводит к отказу без попытки угадать формат;
- весь контейнер проверяется до передачи данных другим компонентам.

Строгий отказ выбран вместо молчаливого принятия неизвестных полей. Для
расширения формата будет создана новая версия протокола. Это проще и безопаснее
для первой реализации.

В текущей Rust-реализации используется `minicbor` 2.2.2 и отдельный строгий
слой проверки схемы. Версия библиотеки закреплена точно. Декодер сначала
проверяет общий лимит, читает byte string как ссылки на входной срез и копирует
payload только после проверки структуры. Подробности выбора находятся в
`docs/adr/0002-strict-cbor-codec.md`.

## Envelope версии 1

Логическая схема:

| Ключ | Поле | Тип | Ограничение |
| --- | --- | --- | --- |
| 0 | `protocol_version` | unsigned integer | только `1` |
| 1 | `message_id` | byte string | ровно 16 байт |
| 2 | `recipient_hint` | byte string | ровно 16 байт |
| 3 | `ttl_seconds` | unsigned integer | от `60` до `604800` |
| 4 | `max_hops` | unsigned integer | от `1` до `16` |
| 5 | `priority` | unsigned integer | только `0` в v0.1 |
| 6 | `protected_payload` | byte string | от `1` байта до `63 KiB` |

Полный сериализованный Envelope не может быть больше `64 KiB`.

### `protocol_version`

Версия определяет правила разбора всего объекта. Декодер v0.1 принимает только
значение `1`. Неизвестную версию нельзя пытаться разобрать как первую.

### `message_id`

Это случайные 16 байт, созданные криптографически стойким генератором случайных
чисел на устройстве отправителя. Идентификатор не является хешем текста и не
содержит время, имя пользователя или адрес устройства.

`message_id` нужен для дедупликации. Совпадение с уже известным значением
считается повтором. При редкой случайной коллизии второй контейнер будет
отклонён. Такой отказ безопаснее, чем замена уже принятого объекта.

### `recipient_hint`

Это непрозрачная подсказка фиксированной длины. Она помогает конечному узлу
понять, стоит ли передавать контейнер своей криптографической сессии. Транспорт
может только сравнивать байты и не знает, как подсказка получена.

Каждое устройство создаёт собственный входящий hint из 16 случайных байт.
Собеседник использует это значение во внешних Envelope, адресованных владельцу
hint. Значение не выводится из имени, публичного ключа, номера телефона, session
ID или времени.

Hint меняется внутри уже проверенной Olm-сессии после ограниченного числа
сообщений. Во время подтверждённого перехода старое и новое значения могут
приниматься одновременно, но их число всегда ограничено. Полные правила
создания, проверки и смены находятся в [CONTACTS.md](CONTACTS.md).

### `ttl_seconds`

Отправитель просит честные узлы хранить контейнер не дольше указанного времени.
Значение ограничено диапазоном от 60 секунд до 7 суток.

TTL не является криптографической гарантией удаления. Злонамеренный Relay может
сохранить копию навсегда. Поле ограничивает только поведение честной реализации.

### `max_hops`

Отправитель задаёт максимальное количество последовательных передач между
честными узлами. Значение не превышает 16.

Сам Envelope не меняется, поэтому фактическое число переходов хранится в
локальной записи маршрутизации и передаётся в служебном сообщении синхронизации.
Злонамеренный узел может соврать и сбросить этот счётчик. Значит, `max_hops`
уменьшает обычное распространение, но не является полной защитой от
враждебной сети.

### `priority`

В v0.1 существует только обычный приоритет со значением `0`. SOS и системных
приоритетов пока нет. Любое другое значение отклоняется.

Поле оставлено явным, чтобы правила очереди не зависели от случайного порядка
получения полей. Новый приоритет появится только вместе с новой версией
протокола и отдельной защитой от злоупотреблений.

### `protected_payload`

Это результат работы выбранного E2EE-протокола. Для транспорта и Relay поле
полностью непрозрачно.

Криптографическая защита должна связывать payload как минимум с этими полями:

- `protocol_version`;
- `message_id`;
- `recipient_hint`;
- `ttl_seconds`;
- `max_hops`;
- `priority`.

Выбранный для experimental MVP Olm v1 не принимает внешние associated data.
Поэтому эти значения входят во внутреннее строго закодированное сообщение,
которое целиком шифруется Olm. После расшифрования внутренние значения
сравниваются с внешним Envelope. Несовпадение приводит к отказу до показа
сообщения пользователю.

Точная внутренняя структура `protected_payload` ещё не зафиксирована. Следующий
пакет должен включить прикладное сообщение, подтверждение контакта и служебные
сообщения смены hint из [CONTACTS.md](CONTACTS.md). Транспортная спецификация
не будет её дублировать.

## Чего нет в открытом Envelope

В контейнер не записываются открытым текстом:

- имя или ник отправителя;
- имя получателя;
- публичный ключ контакта;
- текст сообщения;
- название группы;
- имя файла;
- координаты;
- IP-адрес исходного устройства;
- абсолютное время создания.

Размер, TTL, число разрешённых переходов и `recipient_hint` всё равно являются
метаданными. v0.1 не обещает их скрыть.

Отдельное поле `payload_size` не нужно. CBOR byte string уже содержит длину.
Дублирование создало бы два значения, которые могут противоречить друг другу.

## Локальная запись маршрутизации

Эта запись не входит в Envelope и не передаётся целиком другим узлам:

| Поле | Назначение |
| --- | --- |
| `message_id` | ссылка на сохранённый Envelope |
| `first_seen_at` | локальное время первого приёма |
| `local_deadline` | крайний локальный срок хранения |
| `remaining_ttl` | оставшееся время жизни в честном маршруте |
| `hops_taken` | известное этому узлу число переходов |
| `copies_left` | локальный бюджет копий для Spray-and-Wait |
| `state` | текущее состояние контейнера |

Для v0.1 выбрана политика Binary Spray-and-Wait с начальным бюджетом 32 копии.
`copies_left` остаётся внутренней деталью маршрутизатора и не меняет Envelope.
Полученное от соседа значение не может быть больше 32. Злонамеренный узел может
соврать о бюджете, поэтому ограничение действует между честными узлами и не
заменяет локальную квоту.

Значения, полученные от соседа, всегда ограничиваются локальными максимумами.
Они не считаются доказательством честно пройденного маршрута.

## Работа TTL и времени

Часам отправителя доверять нельзя, поэтому абсолютного `expires_at` в Envelope
нет. Вместо него честные узлы передают в служебном сообщении остаток TTL. Это
маршрутное значение не меняет сам Envelope.

При первом приёме узел:

1. проверяет диапазон `ttl_seconds`;
2. проверяет полученный `remaining_ttl` и ограничивает его значением из
   Envelope;
3. записывает своё текущее время;
4. вычисляет `local_deadline` по меньшему из двух значений - принятому остатку
   TTL и локальному максимуму;
5. сохраняет запись вместе с контейнером одной атомарной операцией;
6. во время работы уменьшает остаток по монотонным часам;
7. при передаче сообщает следующему узлу уже уменьшенное значение.

На устройстве отправителя `remaining_ttl` начинается со значения `ttl_seconds`.
Так время хранения у честных узлов не запускается заново на каждом переходе.
Злонамеренный Relay может увеличить служебный счётчик, поэтому следующий узел
всё равно ограничивает его исходным TTL и своим локальным максимумом. Полностью
доказать общий срок хранения в недоверенном маршруте без доверенного времени
нельзя.

Для восстановления после перезапуска узел атомарно сохраняет локальные сроки и
последнее наблюдавшееся значение настенных часов. Если новые часы не меньше
сохранённых, достигшие `local_deadline` записи истекают обычным способом.

Если настенные часы после перезапуска ушли назад, срок нельзя автоматически
продлевать на разницу. Для v0.1 выбрано консервативное правило: все активные
Envelope локально переходят в `expired`, а их ID попадают в ограниченное
tombstone-хранилище от нового текущего времени. Существующим tombstone
переносится только срок, остававшийся на последней известной временной точке.
Уже истёкшие записи удаляются, после чего новое значение часов становится
началом следующего интервала. Это может удалить сообщение раньше срока, но не
позволяет откату часов продлить хранение.

Во время непрерывной работы `lantern-time` объединяет прошедшее время `Instant`,
текущее `SystemTime` и предыдущее логическое значение. Результат не уменьшается
внутри процесса. Один результат чтения передаётся очереди, диагностике и
хранилищу в пределах одной операции обслуживания.

Если наблюдаемое `SystemTime` стало меньше предыдущего, чтение получает статус
отката. Процесс узла должен консервативно завершить все активные Envelope и
явно очистить диагностический журнал. Простого обслуживания журнала логическим
временем недостаточно, потому что оно не уменьшается. При перезапуске остаётся
обязательной отдельная проверка сохранённой временной точки в
`lantern-storage`.

Неверное время может привести к раннему удалению сообщения. Это допустимый
отказ доступности. Бесконечное продление хранения недопустимо.

## Состояния контейнера

### `draft`

Прикладное сообщение ещё не защищено. Оно существует только на устройстве
отправителя и никогда не передаётся транспорту.

### `sealed`

Криптографическое ядро создало Envelope, но он ещё не записан в постоянную
очередь.

### `stored`

Envelope прошёл проверку формата и атомарно сохранён. Из этого состояния его
можно предложить соседу.

### `opened`

Конечный получатель успешно проверил и открыл сообщение. Повторный Envelope с
тем же `message_id` не создаёт новое сообщение.

### `expired`

Локальный срок закончился. Контейнер удаляется и больше не предлагается
соседям.

### `evicted`

Контейнер удалён из-за заполнения квоты. Это нормальный ограниченный отказ, а не
подтверждение доставки.

### `rejected`

Вход не прошёл проверку размера, версии, структуры или криптографии. Сам
контейнер не сохраняется. Можно записать только безопасный код причины.

## Переходы состояний

```text
draft -> sealed -> stored -> opened
                    |  |
                    |  +-> expired
                    +----> evicted

untrusted input -> rejected
```

Передача соседу не является отдельным состоянием. Один и тот же сохранённый
контейнер можно передать несколько раз разным узлам. Для этого в журнал пишется
безопасное техническое событие без содержимого сообщения.

Событие содержит только закрытый код операции, общую категорию результата,
ограниченный счётчик и необязательные грубые категории размера и длительности.
В нём нет Envelope, `message_id`, `recipient_hint`, адреса соседа, строки ошибки
и абсолютного времени. Последовательный номер показывает порядок локальных
событий, но не является сетевым полем и не передаётся соседу.

В v0.1 нет подтверждения прочтения и сквозного подтверждения доставки. Поэтому
Alice не может считать сообщение доставленным только из-за передачи Relay.

## Общая транспортная граница

`lantern-transport` работает с одним полным непрозрачным кадром. Он не разбирает
CBOR, не получает ключи и не знает структуру `protected_payload`.

Для одного направления одной сессии действуют начальные границы v0.1:

- размер кадра - от 1 байта до 64 KiB;
- не больше 1024 кадров;
- не больше 64 MiB кадров.

Приём и отправка учитываются отдельно. Конфигурация может уменьшить эти
значения, но не увеличить их через публичный API. При исчерпании входящей квоты
сессия завершается до следующего чтения.

Конкретный адаптер обязан проверить длину своего wire framing до чтения тела.
Он получает заранее выделенный ограниченный буфер и не должен выделять память
по большей недоверенной длине. TCP framing, рукопожатие, согласование версии и
синхронизация списков появятся отдельно на Этапе 3.

Сессионная квота не является идентичностью соседа. Повторное соединение создаёт
новую квоту, поэтому общий rate limit одного соседа остаётся отдельным сетевым
решением.

## Порядок проверки входящего Envelope

Узел обрабатывает вход в таком порядке:

1. ограничивает число байтов транспортного кадра;
2. проверяет общий размер Envelope;
3. строго разбирает CBOR без больших предварительных выделений;
4. проверяет наличие всех полей и отсутствие лишних;
5. проверяет тип и длину каждого поля;
6. проверяет версию, TTL, max hops и priority;
7. ищет `message_id` в ограниченном хранилище дедупликации;
8. проверяет локальную квоту;
9. только на конечном устройстве передаёт подходящий payload криптографическому
   ядру;
10. атомарно сохраняет допустимый контейнер и его локальную запись.

Нельзя сначала выделить память по длине из входного поля, а потом проверять
общий лимит. Текст ошибки не должен содержать входной payload.

## Ограниченная очередь

Стандартная очередь чужих контейнеров v0.1 имеет две одновременные квоты:

- не больше 1000 активных Envelope;
- не больше 64 MiB их канонического CBOR-представления.

Локальная конфигурация может уменьшить эти значения, но API Lantern Core не
позволяет увеличить их выше границ v0.1. Нулевые квоты запрещены.

Байтовая квота относится к полному сериализованному Envelope, а не только к
`protected_payload`. Она не является точным измерением памяти процесса:
маршрутная запись, индекс и накладные расходы allocator занимают дополнительное
место. Их рост ограничен квотой числа записей, а фактический расход памяти будет
измерен перед сетевой реализацией.

Перед добавлением Envelope очередь:

1. удаляет локально просроченные активные записи;
2. удаляет просроченные tombstone;
3. проверяет активный ID и tombstone;
4. отклоняет отдельно не помещающийся Envelope без вытеснения;
5. при необходимости удаляет старые записи по FIFO;
6. добавляет новый Envelope только после соблюдения обеих квот.

FIFO использует `first_seen_at`, а при равенстве - `message_id`. Порядок
детерминирован и не зависит от внутреннего обхода map. Один новый Envelope
может потребовать удаления нескольких старых записей по байтовой квоте.

FIFO не определяет полезность сообщения. Спамер с новыми идентификаторами может
вытеснить старые полезные записи. Отклонение всех новых объектов оставило бы
обратную проблему: первый заполнивший очередь сосед мог бы удерживать её до
истечения TTL. Для v0.1 выбран проверенный в симуляторе простой вариант, а
сессионная квота транспорта ограничивает только одно соединение. Глобальный
лимит одного соседа остаётся отдельным сетевым решением.

Текущая очередь работает в памяти процесса, а `lantern-storage` атомарно
сохраняет её снимок и tombstone в SQLite. Процесс узла ещё должен связать одну
операцию очереди с сохранением снимка и обработать ошибку без частичного
подтверждения сетевому соседу.

## Дедупликация

Основной ключ дедупликации - `message_id`. Запись о недавно удалённом сообщении
нужно хранить отдельно ограниченное время, иначе старый контейнер сразу вернётся
в очередь.

Начальное ограничение v0.1 - 2000 tombstone-записей на узел со сроком хранения
7 суток. Оно в два раза больше стандартной очереди из 1000 контейнеров, но
всё равно имеет жёсткую квоту. Фактический размер на диске будет проверен после
выбора хранилища. Это предварительная граница, а не результат нагрузочного
измерения. Когда старая запись вытеснена, защита опирается на TTL и
криптографическое состояние получателя.

Tombstone создаётся после `opened`, `expired` и `evicted`. При заполнении
tombstone-хранилища удаляется самая старая запись, а при равном времени -
меньший `message_id`. На границе `expires_at` запись уже не считается активной.
Повтор не продлевает её срок.

Совпадение с активным ID или tombstone проверяется до применения квоты. Новый
Envelope с тем же ID никогда не заменяет уже сохранённый, даже если остальные
поля отличаются.

Дедупликация экономит ресурсы, но сама по себе не доказывает подлинность.
Случайный новый `message_id` не превращает поддельный payload в сообщение Alice.

## Поведение при повреждении и потере

- повреждённый Envelope отклоняется целиком;
- неизвестная версия не разбирается как версия 1;
- дубликат не добавляется в очередь повторно;
- перестановка контейнеров не должна приводить к панике;
- обрыв передачи оставляет либо старое состояние, либо полностью записанный
  объект;
- частичный объект не становится `stored`;
- потерянный объект можно запросить при следующей встрече;
- превышение квоты приводит к отказу или документированному вытеснению, а не к
  неограниченному росту.

## Что ещё не определено

До реализации соответствующих Rust-компонентов нужно выбрать и проверить:

- внутренний формат `protected_payload`;
- формат служебных сообщений синхронизации;
- правило расчёта и передачи `hops_taken`;
- глобальные лимиты одного соседа между повторными подключениями.

Формат контактных bundle и жизненный цикл `recipient_hint` уже определены в
[CONTACTS.md](CONTACTS.md) и ADR 0014, но ещё не реализованы.

Открытый вопрос не разрешает временно ослабить проверку или придумать
криптографическую схему.

## Тесты реализации

Строгий CBOR-кодек уже проверяет:

- точный минимальный вектор;
- максимальный логический Envelope;
- общий лимит входа;
- короткие идентификаторы и payload неправильного размера;
- границы TTL и max hops;
- неизвестные version и priority;
- отсутствующие, лишние, повторяющиеся и переставленные ключи;
- неопределённые длины и неканоничные заголовки;
- неверные типы, теги, float, вложенные значения и данные после Envelope;
- все обрезанные префиксы минимального вектора;
- безопасные тексты ошибок без входного payload;
- детерминированный набор повреждённых входов длиной до 512 байт.

In-memory очередь уже проверяет:

- обе стандартные квоты и запрет их увеличения через API;
- активный повтор и повтор через tombstone;
- границу срока tombstone;
- FIFO и детерминированное разрешение равного времени;
- вытеснение нескольких Envelope по байтовой квоте;
- отказ отдельно не помещающемуся Envelope без очистки очереди;
- истечение `local_deadline` перед применением квоты;
- переполнение срока tombstone без удаления активной записи;
- ограничение tombstone и вытеснение самой старой записи;
- сохранение всех квот в детерминированной последовательности из 500 операций;
- отсутствие payload, полных ID и точного времени в `Debug` и ошибках.

Ограниченный диагностический журнал уже проверяет:

- отсутствие строк и байтовых полей в публичном событии;
- точные границы категорий размера и длительности;
- квоты числа записей и логических байтов;
- FIFO-вытеснение при заполнении;
- удаление на точной границе срока;
- очистку при откате часов;
- отсутствие точного времени и исходных чисел в `Debug`;
- сохранение квот в последовательности из 1000 событий.

Строгий CBOR-декодер дополнительно проверяет изменение каждого отдельного байта
канонического тестового Envelope. Изменение либо отклоняется, либо даёт другой
корректный канонический Envelope без паники. Второй случай ожидаем для байтов
payload: на Этапе 2 нет криптографической проверки целостности.

Общая транспортная граница уже проверяет:

- пустые и превышающие размер исходящие кадры до вызова адаптера;
- буфер вызывающего кода и уменьшение доступной части до остатка квоты;
- точные границы числа кадров и байтов в обоих направлениях;
- завершение сессии до нового вызова адаптера после исчерпания квоты;
- пустой вход, неверную длину, закрытие соединения и безопасную категорию
  транспортной ошибки;
- отсутствие точных байтов и состояния адаптера в `Debug`;
- совпадение лимита кадра с `lantern-core`;
- передачу допустимого и повреждённого непрозрачного кадра строгому CBOR
  декодеру без разбора внутри транспорта.

Отдельные property-тесты с фиксированными seed генерируют входные байты,
корректные Envelope, цепочки операций очереди, скачки времени и длины кадров.
Они проверяют общие свойства, а точные границы по-прежнему остаются в unit-тестах.

Общий жизненный цикл узла уже проверяет:

- одно чтение времени на операцию очереди и диагностики;
- восстановление Envelope после обычного перезапуска;
- консервативное истечение очереди при откате часов между запусками;
- очистку диагностики и истечение очереди при откате часов во время работы;
- атомарное сохранение локального Envelope и дедупликацию повтора;
- запрет новых операций после ошибки часов;
- отсутствие пути, ID, payload, точного времени и точного размера очереди в
  `Debug` и ошибках.

Постоянная диагностика и профиль уже проверяют:

- восстановление типизированных событий после перезапуска;
- удаление на границе срока и полную очистку при откате часов;
- квоты записей, логического размера и файла до 10 MiB;
- отказ при неизвестном коде, увеличенном сроке, другой версии и других
  лимитах;
- сохранение предыдущего снимка в памяти и на диске при ошибке транзакции;
- права `0600`, запрет символических и жёстких ссылок на Unix;
- отказ второго процесса до чтения часов;
- запрет двум профилям одновременно писать в одну базу диагностики;
- освобождение блокировки после `stop`, ошибки и уничтожения объекта;
- отказ при пересечении путей очереди, SQLite-sidecar, lock и диагностики;
- отказ при числе строк выше сохранённой квоты до загрузки записей;
- отсутствие тестового payload, полного `message_id`, `recipient_hint` и пути
  очереди в байтах постоянной базы диагностики.

Для следующих компонентов остаются тесты:

- криптографическое отклонение изменения каждого защищённого байта после выбора
  E2EE-библиотеки;
- перезапуск между сетевым приёмом и повторной доставкой;
- неверные часы, потеря, дублирование и перестановка;
- отсутствие секретных тестовых строк в выводе будущего CLI узла и его
  постоянных логах.

Наличие текущих тестов не заменяет fuzzing, проверку зависимостей и независимый
аудит.

## Ограничения решения

- `recipient_hint` и размеры контейнеров раскрывают часть метаданных, а Relay
  может коррелировать смену hint по времени трафика;
- злонамеренный узел может хранить контейнер после TTL;
- злонамеренный узел может соврать о числе переходов и копий;
- Relay может удалить сообщение и полностью сорвать доставку;
- `vodozemac 0.10.0` выбран только условно и ещё не подключён к Lantern;
- контактные QR, SAS и смена hint пока существуют только в спецификации;
- строгий CBOR-кодек ещё не проходил fuzzing и независимый аудит;
- защита от повторов ограничена размером локального хранилища;
- базы очереди и диагностики не зашифрованы;
- блокировка профиля не защищает от вредоносного процесса с теми же правами на
  каталог;
- FIFO позволяет потоку новых ID вытеснять полезные старые контейнеры;
- спецификация ещё изменится до первой сетевой реализации.

Пока эти решения не реализованы и не протестированы, протокол нельзя называть
безопасным или готовым для реального использования.
