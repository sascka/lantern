# Протокол Lantern v0.1

Статус: черновик Этапа 0.

Этот документ описывает логическую структуру контейнера и его локальные
состояния. Здесь пока нет сетевого рукопожатия и конкретной криптографической
схемы. Их нельзя фиксировать до отдельного выбора библиотек и модели ключей.

## Задача контейнера

Контейнер переносит одно защищённое сообщение между узлами. После создания он
не меняется. Alice, Relay и Bob передают одинаковый набор байтов.

Неизменяемый контейнер проще проверять и дедуплицировать. Счётчики конкретного
маршрута хранятся рядом с ним в локальной записи узла, а не переписываются
внутри контейнера.

```text
прикладное сообщение
        |
криптографическое ядро
        |
неизменяемый Envelope
        |
локальная запись маршрутизации
        |
LAN-транспорт
```

Криптографическое ядро не открывает секреты транспорту. Транспорт не знает
структуру прикладного сообщения.

## Формат сериализации

Для v0.1 выбираем CBOR с детерминированным представлением.

Почему CBOR подходит:

- он компактнее JSON;
- двоичные поля не нужно кодировать в Base64;
- формат имеет стандартные правила детерминированного кодирования;
- для Rust и Python есть готовые реализации;
- схему можно описать отдельно от конкретного языка.

Недостатки:

- схема не поставляется вместе с форматом, как в Protocol Buffers;
- библиотеку нужно настроить на строгий разбор;
- не все CBOR-кодировщики создают одинаковые байты по умолчанию.

Protocol Buffers был альтернативой. Он даёт хорошую генерацию типов, но добавит
отдельный компилятор схем и свои правила работы с неизвестными полями. Для
небольшого фиксированного контейнера v0.1 это лишняя сложность.

Мы не используем собственный бинарный формат. Самодельный парсер недоверенных
данных добавит слишком много мест для ошибок.

## Правила CBOR для v0.1

- верхний объект - map;
- ключи - только небольшие беззнаковые целые числа;
- длины map и byte string всегда указываются заранее;
- неопределённые длины запрещены;
- числа кодируются в самом коротком допустимом виде;
- ключи идут в числовом порядке;
- повторяющиеся ключи запрещены;
- теги, числа с плавающей точкой и текстовые ключи запрещены;
- неизвестное поле в версии 1 приводит к отказу;
- неизвестная версия приводит к отказу без попытки угадать формат;
- весь контейнер проверяется до передачи данных другим компонентам.

Строгий отказ выбран вместо молчаливого принятия неизвестных полей. Для
расширения формата будет создана новая версия протокола. Это проще и безопаснее
для первой реализации.

## Envelope версии 1

Логическая схема:

| Ключ | Поле | Тип | Ограничение |
| --- | --- | --- | --- |
| 0 | `protocol_version` | unsigned integer | только `1` |
| 1 | `message_id` | byte string | ровно 16 байт |
| 2 | `recipient_hint` | byte string | ровно 16 байт |
| 3 | `ttl_seconds` | unsigned integer | от `60` до `604800` |
| 4 | `max_hops` | unsigned integer | от `1` до `16` |
| 5 | `priority` | unsigned integer | только `0` в v0.1 |
| 6 | `protected_payload` | byte string | от `1` байта до `63 KiB` |

Полный сериализованный Envelope не может быть больше `64 KiB`.

### `protocol_version`

Версия определяет правила разбора всего объекта. Декодер v0.1 принимает только
значение `1`. Неизвестную версию нельзя пытаться разобрать как первую.

### `message_id`

Это случайные 16 байт, созданные криптографически стойким генератором случайных
чисел на устройстве отправителя. Идентификатор не является хешем текста и не
содержит время, имя пользователя или адрес устройства.

`message_id` нужен для дедупликации. Совпадение с уже известным значением
считается повтором. При редкой случайной коллизии второй контейнер будет
отклонён. Такой отказ безопаснее, чем замена уже принятого объекта.

### `recipient_hint`

Это непрозрачная подсказка фиксированной длины. Она помогает конечному узлу
понять, стоит ли передавать контейнер своей криптографической сессии. Транспорт
может только сравнивать байты и не знает, как подсказка получена.

Правило создания и смены `recipient_hint` определит выбранный E2EE-протокол.
Нельзя делать его из имени, постоянного публичного ключа, номера телефона или
другого легко узнаваемого значения. Одна подсказка не должна использоваться
бессрочно, иначе она станет меткой для наблюдения.

Если подходящий E2EE-протокол не позволяет безопасно получить такую подсказку,
поле будет пересмотрено до начала сетевой реализации. Временную самодельную
схему для него использовать нельзя.

### `ttl_seconds`

Отправитель просит честные узлы хранить контейнер не дольше указанного времени.
Значение ограничено диапазоном от 60 секунд до 7 суток.

TTL не является криптографической гарантией удаления. Злонамеренный Relay может
сохранить копию навсегда. Поле ограничивает только поведение честной реализации.

### `max_hops`

Отправитель задаёт максимальное количество последовательных передач между
честными узлами. Значение не превышает 16.

Сам Envelope не меняется, поэтому фактическое число переходов хранится в
локальной записи маршрутизации и передаётся в служебном сообщении синхронизации.
Злонамеренный узел может соврать и сбросить этот счётчик. Значит, `max_hops`
уменьшает обычное распространение, но не является полной защитой от
враждебной сети.

### `priority`

В v0.1 существует только обычный приоритет со значением `0`. SOS и системных
приоритетов пока нет. Любое другое значение отклоняется.

Поле оставлено явным, чтобы правила очереди не зависели от случайного порядка
получения полей. Новый приоритет появится только вместе с новой версией
протокола и отдельной защитой от злоупотреблений.

### `protected_payload`

Это результат работы выбранного E2EE-протокола. Для транспорта и Relay поле
полностью непрозрачно.

Криптографическая защита должна связывать payload как минимум с этими полями:

- `protocol_version`;
- `message_id`;
- `recipient_hint`;
- `ttl_seconds`;
- `max_hops`;
- `priority`.

Их детерминированное CBOR-представление передаётся криптографической библиотеке
как associated data, если выбранный протокол поддерживает такую модель. Если
протокол работает иначе, он всё равно должен обнаруживать замену этих полей до
показа сообщения пользователю.

Точная внутренняя структура `protected_payload` появится только после выбора
E2EE. Транспортная спецификация не будет её дублировать.

## Чего нет в открытом Envelope

В контейнер не записываются открытым текстом:

- имя или ник отправителя;
- имя получателя;
- публичный ключ контакта;
- текст сообщения;
- название группы;
- имя файла;
- координаты;
- IP-адрес исходного устройства;
- абсолютное время создания.

Размер, TTL, число разрешённых переходов и `recipient_hint` всё равно являются
метаданными. v0.1 не обещает их скрыть.

Отдельное поле `payload_size` не нужно. CBOR byte string уже содержит длину.
Дублирование создало бы два значения, которые могут противоречить друг другу.

## Локальная запись маршрутизации

Эта запись не входит в Envelope и не передаётся целиком другим узлам:

| Поле | Назначение |
| --- | --- |
| `message_id` | ссылка на сохранённый Envelope |
| `first_seen_at` | локальное время первого приёма |
| `local_deadline` | крайний локальный срок хранения |
| `remaining_ttl` | оставшееся время жизни в честном маршруте |
| `hops_taken` | известное этому узлу число переходов |
| `copies_left` | локальный бюджет копий для Spray-and-Wait |
| `state` | текущее состояние контейнера |

`copies_left` используется только после выбора Spray-and-Wait. Для других
алгоритмов поле остаётся внутренней деталью маршрутизатора и не меняет Envelope.

Значения, полученные от соседа, всегда ограничиваются локальными максимумами.
Они не считаются доказательством честно пройденного маршрута.

## Работа TTL и времени

Часам отправителя доверять нельзя, поэтому абсолютного `expires_at` в Envelope
нет. Вместо него честные узлы передают в служебном сообщении остаток TTL. Это
маршрутное значение не меняет сам Envelope.

При первом приёме узел:

1. проверяет диапазон `ttl_seconds`;
2. проверяет полученный `remaining_ttl` и ограничивает его значением из
   Envelope;
3. записывает своё текущее время;
4. вычисляет `local_deadline` по меньшему из двух значений - принятому остатку
   TTL и локальному максимуму;
5. сохраняет запись вместе с контейнером одной атомарной операцией;
6. во время работы уменьшает остаток по монотонным часам;
7. при передаче сообщает следующему узлу уже уменьшенное значение.

На устройстве отправителя `remaining_ttl` начинается со значения `ttl_seconds`.
Так время хранения у честных узлов не запускается заново на каждом переходе.
Злонамеренный Relay может увеличить служебный счётчик, поэтому следующий узел
всё равно ограничивает его исходным TTL и своим локальным максимумом. Полностью
доказать общий срок хранения в недоверенном маршруте без доверенного времени
нельзя.

Если настенные часы после перезапуска ушли назад, срок нельзя автоматически
продлевать на разницу. Точный способ хранения устойчивого срока зависит от
выбранной базы и будет проверен отдельными тестами.

Неверное время может привести к раннему удалению сообщения. Это допустимый
отказ доступности. Бесконечное продление хранения недопустимо.

## Состояния контейнера

### `draft`

Прикладное сообщение ещё не защищено. Оно существует только на устройстве
отправителя и никогда не передаётся транспорту.

### `sealed`

Криптографическое ядро создало Envelope, но он ещё не записан в постоянную
очередь.

### `stored`

Envelope прошёл проверку формата и атомарно сохранён. Из этого состояния его
можно предложить соседу.

### `opened`

Конечный получатель успешно проверил и открыл сообщение. Повторный Envelope с
тем же `message_id` не создаёт новое сообщение.

### `expired`

Локальный срок закончился. Контейнер удаляется и больше не предлагается
соседям.

### `evicted`

Контейнер удалён из-за заполнения квоты. Это нормальный ограниченный отказ, а не
подтверждение доставки.

### `rejected`

Вход не прошёл проверку размера, версии, структуры или криптографии. Сам
контейнер не сохраняется. Можно записать только безопасный код причины.

## Переходы состояний

```text
draft -> sealed -> stored -> opened
                    |  |
                    |  +-> expired
                    +----> evicted

untrusted input -> rejected
```

Передача соседу не является отдельным состоянием. Один и тот же сохранённый
контейнер можно передать несколько раз разным узлам. Для этого в журнал пишется
безопасное техническое событие без содержимого сообщения.

В v0.1 нет подтверждения прочтения и сквозного подтверждения доставки. Поэтому
Alice не может считать сообщение доставленным только из-за передачи Relay.

## Порядок проверки входящего Envelope

Узел обрабатывает вход в таком порядке:

1. ограничивает число байтов транспортного кадра;
2. проверяет общий размер Envelope;
3. строго разбирает CBOR без больших предварительных выделений;
4. проверяет наличие всех полей и отсутствие лишних;
5. проверяет тип и длину каждого поля;
6. проверяет версию, TTL, max hops и priority;
7. ищет `message_id` в ограниченном хранилище дедупликации;
8. проверяет локальную квоту;
9. только на конечном устройстве передаёт подходящий payload криптографическому
   ядру;
10. атомарно сохраняет допустимый контейнер и его локальную запись.

Нельзя сначала выделить память по длине из входного поля, а потом проверять
общий лимит. Текст ошибки не должен содержать входной payload.

## Дедупликация

Основной ключ дедупликации - `message_id`. Запись о недавно удалённом сообщении
нужно хранить отдельно ограниченное время, иначе старый контейнер сразу вернётся
в очередь.

Точный размер и срок tombstone-хранилища определим после симуляции. Оно тоже
имеет жёсткую квоту. Когда старая запись вытеснена, защита опирается на TTL и
криптографическое состояние получателя.

Дедупликация экономит ресурсы, но сама по себе не доказывает подлинность.
Случайный новый `message_id` не превращает поддельный payload в сообщение Alice.

## Поведение при повреждении и потере

- повреждённый Envelope отклоняется целиком;
- неизвестная версия не разбирается как версия 1;
- дубликат не добавляется в очередь повторно;
- перестановка контейнеров не должна приводить к панике;
- обрыв передачи оставляет либо старое состояние, либо полностью записанный
  объект;
- частичный объект не становится `stored`;
- потерянный объект можно запросить при следующей встрече;
- превышение квоты приводит к отказу или документированному вытеснению, а не к
  неограниченному росту.

## Что ещё не определено

До начала Rust-реализации нужно выбрать и проверить:

- конкретную CBOR-библиотеку и её строгий режим;
- точные тестовые байты для каждого допустимого Envelope;
- E2EE-протокол и способ получения `recipient_hint`;
- внутренний формат `protected_payload`;
- формат служебных сообщений синхронизации;
- правило расчёта и передачи `hops_taken`;
- начальный бюджет `copies_left` после результатов симулятора;
- размер и срок хранения записей дедупликации;
- способ хранения `local_deadline` после перезапуска.

Открытый вопрос не разрешает временно ослабить проверку или придумать
криптографическую схему.

## Тесты для будущей реализации

- минимальный корректный Envelope;
- Envelope максимального допустимого размера;
- превышение общего размера на один байт;
- короткий и длинный `message_id`;
- короткий и длинный `recipient_hint`;
- TTL на границах и за пределами диапазона;
- max hops на границах и за пределами диапазона;
- неизвестный priority и protocol version;
- отсутствующее, лишнее и повторяющееся поле;
- неопределённая длина CBOR;
- неканоничное представление числа;
- повреждение каждого байта заголовка и payload;
- повтор одного `message_id`;
- перезапуск между сохранением и повторной доставкой;
- неверные часы, потеря, дублирование и перестановка;
- заполнение очереди и tombstone-хранилища;
- отсутствие секретных тестовых строк в логах.

Результаты этих тестов появятся только после реализации. Этот документ сам по
себе не означает, что проверки уже пройдены.

## Ограничения решения

- `recipient_hint` и размеры контейнеров раскрывают часть метаданных;
- злонамеренный узел может хранить контейнер после TTL;
- злонамеренный узел может соврать о числе переходов и копий;
- Relay может удалить сообщение и полностью сорвать доставку;
- CBOR-парсер и криптографическая библиотека пока не выбраны;
- защита от повторов ограничена размером локального хранилища;
- спецификация ещё изменится до первой сетевой реализации.

Пока эти решения не реализованы и не протестированы, протокол нельзя называть
безопасным или готовым для реального использования.
