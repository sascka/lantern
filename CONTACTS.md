# Контакты Lantern v0.1

Статус: спецификация для `Lantern v0.1 experimental preview`.

Этот документ определяет добавление одного контакта при личной встрече,
проверку ключей и смену `recipient_hint`. Он не определяет экран камеры, формат
сообщений чата или сетевую доставку.

В v0.1 нет каталога пользователей, номера телефона, электронной почты,
облачного аккаунта и автоматического поиска контактов. Два человека добавляют
друг друга только при личной встрече и проверяют один и тот же короткий код на
обоих устройствах.

## Границы первой версии

- один профиль хранит не более 128 контактов;
- у контакта есть только одно устройство;
- одновременно профиль выполняет не более одного добавления контакта;
- приглашение и ответ используются один раз;
- для добавления нужны два QR-кода и ручная проверка SAS;
- приглашение содержит один Olm one-time key;
- ответ не содержит второй one-time key;
- Bob создаёт исходящую Olm-сессию, Alice принимает первое сообщение;
- fallback key, удалённая проверка и автоматическое восстановление контакта не
  входят в v0.1.

Имена Alice и Bob обозначают роли одного обмена. Они не являются именами
пользователей и не записываются в QR-код.

## Что хранится в QR-коде

QR содержит строку ASCII:

```text
lantern-contact-v1:<base64url>
```

После префикса находится канонический CBOR, закодированный Base64URL без
символов `=`. В строке запрещены пробелы, переносы и обычные символы Base64
`+` и `/`.

Ограничения проверяются до выделения памяти под весь объект:

- полная строка - не более 704 байт ASCII;
- декодированный CBOR - не более 512 байт;
- только один объект CBOR без данных после него;
- map имеет известную длину и числовые ключи в возрастающем порядке;
- запрещены неизвестные, повторяющиеся и пропущенные поля;
- запрещены неопределённые длины и неканоничные числа;
- повторное кодирование Base64URL должно дать исходную строку.

Такие ограничения не делают QR доверенным. Они только не дают повреждённому
или злонамеренному вводу бесконтрольно расходовать память.

## Приглашение Alice

Приглашение является CBOR map из 13 полей:

| Ключ | Поле | Тип | Ограничение |
| --- | --- | --- | --- |
| 0 | `bundle_version` | unsigned integer | только `1` |
| 1 | `kind` | unsigned integer | `0`, то есть invite |
| 2 | `envelope_versions` | array | только `[1]` в v0.1 |
| 3 | `inner_versions` | array | только `[1]` в v0.1 |
| 4 | `olm_versions` | array | только `[1]` в v0.1 |
| 5 | `invitation_id` | byte string | ровно 16 случайных байт |
| 6 | `signing_identity_key` | byte string | Ed25519, ровно 32 байта |
| 7 | `curve_identity_key` | byte string | Curve25519, ровно 32 байта |
| 8 | `one_time_key` | byte string | Curve25519, ровно 32 байта |
| 9 | `sas_key` | byte string | Curve25519, ровно 32 байта |
| 10 | `invitation_secret` | byte string | ровно 32 случайных байта |
| 11 | `inbound_recipient_hint` | byte string | ровно 16 случайных байт |
| 12 | `signature` | byte string | Ed25519, ровно 64 байта |

Массивы версий нужны для будущего согласования, но декодер v0.1 принимает
только точное значение `[1]`. В будущей версии массив не сможет содержать
больше восьми строго возрастающих значений.

`invitation_id`, `invitation_secret` и `inbound_recipient_hint` создаются
криптографически стойким генератором случайных чисел. Они не выводятся из
публичного ключа, имени или времени.

Канонический invite v1 имеет длину ровно 292 байта. Вместе с префиксом его
строка QR имеет длину 409 байт.

## Ответ Bob

Ответ является CBOR map из 13 полей:

| Ключ | Поле | Тип | Ограничение |
| --- | --- | --- | --- |
| 0 | `bundle_version` | unsigned integer | только `1` |
| 1 | `kind` | unsigned integer | `1`, то есть response |
| 2 | `envelope_version` | unsigned integer | только `1` в v0.1 |
| 3 | `inner_version` | unsigned integer | только `1` в v0.1 |
| 4 | `olm_version` | unsigned integer | только `1` в v0.1 |
| 5 | `response_id` | byte string | ровно 16 случайных байт |
| 6 | `invitation_id` | byte string | ID принятого приглашения |
| 7 | `signing_identity_key` | byte string | Ed25519, ровно 32 байта |
| 8 | `curve_identity_key` | byte string | Curve25519, ровно 32 байта |
| 9 | `sas_key` | byte string | Curve25519, ровно 32 байта |
| 10 | `response_secret` | byte string | ровно 32 случайных байта |
| 11 | `inbound_recipient_hint` | byte string | ровно 16 случайных байт |
| 12 | `signature` | byte string | Ed25519, ровно 64 байта |

Выбранные версии должны присутствовать в приглашении Alice. Ответ с другим
`invitation_id` не относится к текущему обмену и отклоняется.

В ответе нет one-time key. После проверки Bob использует ключ из приглашения
Alice и становится инициатором одной двусторонней Olm-сессии. Симметричный
обмен двумя one-time key здесь не нужен и оставил бы один неиспользованный ключ.

Канонический response v1 имеет длину ровно 272 байта. Вместе с префиксом его
строка QR имеет длину 382 байта.

## Подпись bundle

Оба bundle подписываются собственным Ed25519 identity key. Подпись охватывает
канонический CBOR всех полей, кроме поля 12, и отдельный домен:

```text
Lantern contact invite signature v1\0
Lantern contact response signature v1\0
```

Байт `\0` входит в строку домена. Для подписи используется точная
последовательность:

```text
domain || canonical_unsigned_bundle
```

Нельзя проверять подпись после повторной сборки объекта с другим порядком или
другим кодированием. Декодер сначала проверяет каноническую форму исходных
байтов, затем подпись.

Самоподпись связывает Ed25519 key с остальными полями и обнаруживает изменение
bundle после его создания. Она сама по себе не доказывает, что ключ принадлежит
нужному человеку. Для этого нужна очная SAS-проверка.

## Проверка SAS

Lantern использует готовый SAS API из `vodozemac 0.10.0`. Собственный алгоритм
отпечатка или выбора эмодзи не создаётся.

Каждая сторона создаёт временную пару Curve25519 через `vodozemac::sas::Sas` и
передаёт только публичный `sas_key`. Приватная часть не сериализуется и не
попадает в журнал. Публичный API `vodozemac 0.10.0` не позволяет восстановить
незавершённый `Sas` после перезапуска. Поэтому перезапуск до ручного
подтверждения отменяет обмен. Нужно создать новые QR и сравнить новый SAS.

После получения двух bundle стороны выполняют Diffie-Hellman через API
библиотеки. `EstablishedSas::bytes(info)` принимает только UTF-8 строку, поэтому
произвольный бинарный CBOR передаётся через однозначный Base64URL без `=`. Для
обеих сторон используется одинаковый `info`:

```text
"Lantern contact SAS v1\0"
|| hex4(invite_length) || ":"
|| base64url(full_signed_invite_cbor) || ":"
|| hex4(response_length) || ":"
|| base64url(full_signed_response_cbor)
```

Порядок всегда один: сначала invite, затем response. `hex4` содержит четыре
строчные шестнадцатеричные ASCII-цифры. Обе длины относятся к каноническому CBOR
до Base64URL и помещаются в `u16` благодаря пределу 512 байт. Такое кодирование
связывает с SAS те же полные подписанные bytes и не добавляет собственный KDF.

Основное представление SAS - семь индексов эмодзи, которые возвращает
`SasBytes::emoji_indices()`. Интерфейс показывает эмодзи вместе с их текстовыми
названиями из таблицы Matrix SAS. Пользователи должны сравнить все семь позиций
и явно подтвердить совпадение на обоих устройствах.

Если терминал не может надёжно показать эмодзи, обе стороны переключаются на
три числа из `SasBytes::decimals()`. Нельзя сравнивать эмодзи на одном
устройстве с числами на другом. Числа являются другим отображением того же SAS,
а не дополнительной защитой.

При несовпадении хотя бы одной позиции обмен прекращается. Bundle, временные
SAS-ключи, one-time key и случайные значения этого обмена больше не
используются. Автоматический повтор с теми же данными запрещён.

SAS имеет ограниченную длину. Он защищает только при внимательном сравнении и
новых случайных данных для каждой попытки. Интерфейс не должен предлагать
пропустить проверку или автоматически считать её успешной.

## Постоянный отпечаток контакта

Для страницы контакта публичный Ed25519 key показывается как Base32 по RFC 4648
без `=`. Перед строкой ставится `L1-`, а символы делятся на группы по четыре:

```text
L1-XXXX-XXXX-XXXX-XXXX-XXXX-XXXX-XXXX-XXXX-XXXX-XXXX-XXXX-XXXX-XXXX
```

Это не отдельный хеш и не замена SAS. Отпечаток только даёт человеку
стабильное представление уже проверенного identity key. Curve25519 identity key
связан с ним подписью bundle.

## Порядок добавления контакта

1. Alice создаёт новое приглашение и показывает первый QR.
2. Bob сканирует его, строго разбирает CBOR и проверяет подпись Alice.
3. Bob создаёт ответ на это приглашение и показывает второй QR.
4. Alice сканирует ответ, строго разбирает его и проверяет подпись Bob.
5. Оба устройства вычисляют SAS по двум полным подписанным bundle.
6. Alice и Bob сравнивают семь эмодзи или три числа и подтверждают совпадение.
7. Bob создаёт исходящую Olm v1 session с identity key и one-time key Alice.
8. Bob отправляет первое зашифрованное подтверждение на входящий hint Alice.
9. Alice создаёт входящую сессию только для ожидаемого Curve25519 key Bob,
   проверяет подтверждение и отвечает на входящий hint Bob.
10. Контакт становится активным только после проверки обоих зашифрованных
    подтверждений.

Первое подтверждение Bob содержит внутри Olm:

- версию служебного сообщения;
- `invitation_id`;
- `response_id`;
- `invitation_secret`;
- `response_secret`;
- тип `contact_initiator_confirm`.

Ответ Alice содержит те же значения и тип `contact_receiver_confirm`. Точный
CBOR обоих подтверждений находится в `PAYLOAD.md`. Менять смысл или порядок
проверки при его реализации нельзя.

`invitation_secret` и `response_secret` связывают первую сессию с конкретной
очной встречей. Они не заменяют SAS и не доказывают личность: человек с камерой
мог увидеть QR. Их нельзя показывать в диагностике или экспортировать отдельно.

## Состояния добавления

Локальный профиль использует конечный набор состояний:

```text
empty
invite_created
response_created
sas_verified
session_pending
active
identity_changed
removed
```

Из сети нельзя сразу перейти в `active`. Каждый переход проверяет текущее
состояние, роль и оба ID. Повтор уже выполненного перехода обрабатывается
идемпотентно либо отклоняется без создания второго контакта.

Профиль хранит не более одного незавершённого обмена. Создать новый можно
только после завершения или явной отмены старого. Отмена уничтожает временный
SAS private key, случайные значения и неиспользованный one-time key. Повторное
использование сохранённого QR после отмены не возобновляет обмен.

Абсолютного времени в bundle нет. Неверные часы устройства не должны менять
ключи или позволять принять чужой ответ. Старое приглашение может дойти до шага
сравнения, но Alice примет ответ только для своего текущего активного
`invitation_id`.

## Изменение identity key

Проверенный Ed25519 key является постоянной идентичностью контакта в v0.1.
Bundle с другим ключом нельзя автоматически связать со старым контактом, даже
если имя совпадает.

При изменении ключа контакт переходит в `identity_changed`. Отправка и приём
прикладных сообщений блокируются. Пользователи должны снова встретиться,
обменяться двумя новыми QR и проверить новый SAS. В v0.1 нет автоматического
переноса доверия, отзыва ключа и восстановления после кражи устройства.

## Назначение recipient_hint

Каждое устройство случайно создаёт собственный входящий `recipient_hint`.
Собеседник ставит это значение во внешний Envelope, когда отправляет сообщение
владельцу hint.

Значение:

- имеет длину 16 байт;
- создаётся криптографически стойким генератором;
- проверяется на совпадение с другими активными локальными hint;
- не выводится из ключа, session ID, имени, телефона или времени;
- никогда не записывается целиком в логи;
- хранится вместе с контактом только в будущем секретном хранилище.

Внешний hint видят Relay и соседние узлы. Без локальной таблицы они не должны
узнать имя контакта, но могут связывать несколько Envelope с одним и тем же
hint. Поэтому hint нужно ограниченно менять.

## Смена recipient_hint

Каждое направление меняется независимо. Владелец входящего hint начинает
смену после 32 успешно расшифрованных прикладных сообщений, принятых на текущий
hint. Служебные сообщения в этот счётчик не входят.

Номер поколения имеет тип `u32`. Первое значение равно `0`, при смене оно
увеличивается на единицу. Переполнение блокирует дальнейшую отправку до новой
очной проверки, а не возвращает поколение к нулю.

Смена состоит из четырёх зашифрованных служебных сообщений:

1. Владелец создаёт новый случайный hint и отправляет `hint_proposal` с новым
   поколением и значением.
2. Собеседник сохраняет его как ожидаемый, продолжает использовать старый и
   отвечает `hint_accepted`.
3. Владелец начинает принимать старый и новый hint и отправляет
   `hint_committed`.
4. Собеседник переключает исходящие Envelope на новый hint и отправляет через
   него `hint_final`.

Первое корректное сообщение на новом hint также считается `hint_final`. После
этого новый hint становится текущим, а старый переходит в состояние
`retiring`.

Старый hint принимается ещё не более семи суток локального логического времени.
Этот срок равен максимальному TTL Envelope. После срока он удаляется, даже если
сеть всё ещё хранит старую копию. При обнаруженном откате часов старый hint
удаляется консервативно вместе с истекающей активной очередью. Скачок часов
вперёд может удалить его раньше и сорвать доставку старого Envelope.

Для одного входящего направления одновременно существуют не более трёх
значений:

- текущий hint;
- один предложенный hint;
- один уходящий hint.

Новая смена не начинается, пока уходящий hint не удалён. В каждый момент есть
не более одного предложения. Повтор служебного сообщения с тем же поколением и
значением идемпотентен. То же поколение с другим значением, пропуск поколения и
сообщение из неверного состояния отклоняются.

Потерянное служебное сообщение остаётся обычным Envelope и повторно передаётся
транспортом. После истечения его TTL можно создать новый Envelope с тем же
типом, поколением и hint, но нельзя создавать несколько одновременно. Потеря и
перестановка могут задержать смену, но не должны создавать неограниченное число
hint или разрывать уже работающий старый маршрут.

Точные внутренние поля четырёх служебных сообщений и их связь с Envelope
зафиксированы в `PAYLOAD.md`.

Relay не видит содержимое служебного сообщения и не получает явную пару старого
и нового hint. По времени, размерам и направлению трафика он всё равно может
предположить связь между ними. Ротация уменьшает постоянство метки, но не
обеспечивает анонимность.

## Хранение

До появления зашифрованного секретного хранилища реализация этого протокола не
может создавать настоящие identity key, invitation secret, SAS private key,
Olm Account или Session.

Секретное хранилище должно атомарно сохранять:

- локальные private identity key;
- Olm Account и Session pickle;
- незавершённое приглашение или ответ;
- подтверждение успешного SAS;
- таблицу контактов и соответствие hint контакту;
- состояние и поколение смены hint;
- новые ratchet state до показа или отправки сообщения.

Открытая SQLite-очередь хранит только Envelope и маршрутные метаданные. Она не
получает plaintext, приватные ключи, секреты приглашения или таблицу контактов.

## Обработка недоверенных данных

Bundle отклоняется до изменения состояния, если:

- превышен предел строки или CBOR;
- нарушена каноническая форма;
- неизвестны версия, тип или поле;
- длина любого поля неверна;
- подпись неверна;
- Curve25519 key не проходит проверку библиотеки;
- identity key обеих сторон совпадает;
- ответ не ссылается на текущее приглашение;
- выбранная версия не была предложена;
- профиль уже выполняет другой обмен;
- достигнут предел контактов;
- `recipient_hint` сталкивается с локальным активным значением.

Текст ошибки не содержит исходный QR, ключ, подпись, ID, secret или hint.
Разрешены только закрытая категория ошибки и общий ограниченный счётчик.

## Проверки реализации

До включения настоящих сообщений нужны unit-, integration-, property- и
fuzz-тесты:

- точные канонические векторы invite и response;
- подпись и отказ после изменения каждого поля;
- все обрезанные префиксы и входы больше лимита;
- повторные, неизвестные, переставленные и неканоничные поля;
- неверные размеры ключей, ID, secret, signature и hint;
- неподдерживаемые и пониженные версии;
- неверная ссылка response на invitation;
- отказ от неконтрибутивного Curve25519 key;
- одинаковый SAS у двух честных сторон;
- фиксированные подменённые bundle дают ожидаемое несовпадение SAS;
- отмена на каждом переходе и запрет повторного one-time key;
- потеря, повтор и перестановка каждого служебного сообщения;
- границы 32 сообщений, трёх hint, семи суток и `u32`;
- восстановление каждого состояния после перезапуска;
- атомарный отказ при сбое записи;
- отсутствие контрольных секретов в `Debug`, stdout, stderr и журналах.

Тест совпадения SAS не является независимым аудитом всей схемы. Перед beta
нужны отдельный криптографический обзор реализации и проверка пользовательского
сценария, в котором люди действительно замечают несовпадение.

## Риски и ограничения

- Человек может невнимательно сравнить SAS или подтвердить несовпадение.
- Камера, снимок экрана или вредоносное приложение могут скопировать QR.
- Утечка identity key и hint раскрывает часть социального графа и метаданных.
- Взлом Relay позволяет удалять, задерживать, повторять и переставлять
  Envelope, но не должен позволять создать корректное подтверждение контакта.
- Кража незашифрованной очереди раскрывает ciphertext, hint, размеры и сроки.
- Кража разблокированного секретного хранилища раскрывает контакты, ключи и
  сессии. Хранилище выбрано в `SECRET_STORAGE.md`, но пока не реализовано.
- Повреждение bundle, подтверждения или сообщения должно привести к отказу, а
  не к изменённому принятому plaintext.
- Старый QR можно показать повторно. Без текущего состояния и новой SAS-проверки
  он не активирует контакт, но может потратить время пользователя.
- Пределы QR, одного обмена, 128 контактов и трёх hint сдерживают память, но
  могут привести к отказу при достижении квоты.
- `recipient_hint` не скрывает размеры и время трафика. Relay может
  коррелировать старый и новый hint.
- Неверные часы могут преждевременно удалить уходящий hint и сорвать доставку.
- Потеря и перестановка могут оставить смену hint незавершённой. Старый маршрут
  остаётся рабочим, пока действует его ограниченный срок.
- Документ пока не подкреплён реализацией, fuzzing или независимым аудитом.

Lantern v0.1 остаётся `experimental preview`. Эта схема не даёт оснований
называть проект безопасным для реальной опасной переписки.
