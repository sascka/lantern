# Участие в разработке Lantern

Статус: правила разработки `experimental preview`.

Спасибо за интерес к Lantern. Проект работает с недоверенными сетевыми данными,
секретными ключами и личными сообщениями, поэтому любое изменение должно быть
понятным, ограниченным и проверяемым.

Этапы 0-4 завершены. В репозитории есть спецификация, воспроизводимый
симулятор, Rust-библиотеки, постоянное хранилище, LAN-транспорт,
криптографический контур и минимальный Linux CLI. Текущая версия остаётся
`experimental preview` и не считается прошедшей аудит безопасности.

## Перед началом работы

Прочитайте документы в таком порядке:

1. [VISION.md](VISION.md);
2. [MVP.md](MVP.md);
3. [ROADMAP.md](ROADMAP.md);
4. [THREAT_MODEL.md](THREAT_MODEL.md);
5. [PROTOCOL.md](PROTOCOL.md);
6. [PRIVACY.md](PRIVACY.md);
7. [SECURITY.md](SECURITY.md).

Изменение не должно противоречить этим документам. Если противоречие уже есть,
сначала исправляется спецификация. Нельзя молча выбирать в коде более удобный
вариант.

## Размер изменения

Один pull request решает одну связанную задачу. Нормальный размер изменения
позволяет:

- понять цель без длинного устного объяснения;
- проверить все изменённые ветви поведения;
- откатить исправление без потери несвязанной работы;
- описать риск и результат тестов.

Не нужно дробить одну небольшую функцию на десятки формальных коммитов. Но и
объединять новый протокол, интерфейс, рефакторинг и обновление зависимостей в
один pull request нельзя.

## Работа строго по этапам

Текущий этап указан в `ROADMAP.md`.

Допустимы:

- задачи текущего этапа;
- минимальные исправления найденных ошибок;
- тесты для уже существующего поведения;
- документация принятых решений.

Не принимаются:

- функции из будущего этапа;
- собственные криптографические алгоритмы;
- скрытая телеметрия;
- обязательная идентичность по телефону или email;
- изменения ключевых целей без обсуждения;
- крупный рефакторинг вместе с исправлением одной ошибки.

Если идея полезна, но относится к будущему этапу, её можно записать для
обсуждения. Реализация ждёт своего места в плане.

## Предложение архитектурного решения

Изменение протокола, криптографии, хранения, маршрутизации или границы доверия
сначала описывается текстом.

Нужно указать:

1. проблему;
2. ограничения;
3. выбранное решение;
4. альтернативы;
5. плюсы и минусы;
6. влияние на приватность и модель угроз;
7. способ проверки;
8. план совместимости со старыми данными, если они уже существуют.

Аргумент «так проще» недостаточен, если упрощение снимает проверку безопасности
или ломает разделение компонентов.

## Ветки и коммиты

Для отдельной задачи создаётся отдельная ветка от актуальной `main`.

Рекомендуемые имена:

```text
docs/privacy-clarification
feat/simulator-direct-delivery
fix/envelope-size-check
test/queue-eviction
```

Коммит должен описывать результат, а не процесс работы. Используем короткие
префиксы:

```text
docs: clarify relay retention
feat: add direct delivery simulation
fix: reject oversized envelope
test: cover queue eviction
refactor: separate routing state
chore: update development tooling
```

Не добавляйте в историю временные ключи, базы, дампы памяти, результаты сборки
и настройки редактора.

## Требования к документации

- Текст пишется простым и точным языком.
- Для обычного тире используется символ `-`.
- Обещания отделяются от уже проверенных результатов.
- Новый термин объясняется при первом использовании.
- Команда должна быть пригодна для копирования на Arch Linux.
- Ссылка ведёт на официальный или первичный источник, если он существует.
- Результат теста не указывается без настоящего запуска.
- Изменение требований отражается во всех затронутых документах.

Документы проекта сейчас пишутся на русском языке. Перевод добавляется
отдельным файлом и не заменяет исходную спецификацию без обсуждения.

## Требования к Rust

Когда начнётся Этап 2, Rust-код должен:

- собираться стабильным зафиксированным toolchain;
- использовать `Result` для ожидаемых ошибок;
- не применять `unwrap` и `expect` на недоверенном пути;
- проверять размеры до выделения памяти;
- не писать секреты в ошибки и логи;
- не выполнять скрытые сетевые запросы;
- иметь понятные имена и минимальные зависимости;
- проходить `cargo fmt --check`;
- проходить строгий `cargo clippy`;
- проходить `cargo test`.

Собственный `unsafe` по умолчанию запрещён. Исключение оформляется по правилам
из `SECURITY.md`.

## Требования к Python

Симулятор Этапа 1 использует Python 3.12 или совместимую актуальную версию.

Код должен:

- иметь type hints;
- использовать dataclass для простых моделей данных, где это уместно;
- иметь фиксированный seed в воспроизводимых экспериментах;
- тестироваться через `pytest`;
- отделять модель от сбора метрик и вывода;
- не обращаться к сети;
- не выдавать случайный результат за подтверждённый вывод.

## Тесты

Pull request с кодом содержит тест, если изменение можно проверить
автоматически. Исправление ошибки сначала воспроизводит ошибку тестом, а затем
исправляет её минимальным изменением.

В описании указываются реально запущенные команды и результат. Например:

```text
cargo fmt --all --check
cargo clippy --workspace --all-targets --all-features --locked -- -D warnings
cargo test --workspace --all-targets --all-features --locked
python -m ruff format --check lantern_sim tests
python -m ruff check lantern_sim tests
python -m pytest
```

Точные дополнительные команды и зависимости среды находятся в README каждого
компонента. Rust workspace уже создан, поэтому проверки запускаются из корня
репозитория и охватывают все его crate.

Фраза «тесты должны пройти» не заменяет результат запуска.

## Проверка безопасности

Для каждого изменённого компонента нужно ответить:

- какие данные секретны;
- кто видит данные;
- что даст взлом Relay;
- что раскроет украденная база;
- можно ли изменить или повторить сообщение;
- можно ли заполнить память или диск;
- можно ли продлить распространение бесконечно;
- какие метаданные раскрывают связи пользователей;
- что попадёт в логи;
- что произойдёт при неверном времени;
- что произойдёт при потере и перестановке.

Если ответ изменился, обновляются `THREAT_MODEL.md`, `PRIVACY.md` или
`SECURITY.md`.

## Зависимости

Новая зависимость требует объяснения, почему без неё нельзя обойтись. В pull
request указываются:

- точная версия;
- официальный источник;
- лицензия;
- транзитивные зависимости;
- известные сетевые обращения и телеметрия;
- история сопровождения и уязвимостей;
- функции, которые действительно включены.

Правила совместимости лицензий находятся в `LICENSES.md`. Обновление версии
зависимости рассматривается как отдельное изменение, если оно влияет на
безопасность или формат данных.

## Описание pull request

Описание отвечает на пять вопросов:

1. Что изменилось?
2. Почему это относится к текущему этапу?
3. Как это проверено?
4. Какие риски остались?
5. Какие документы затронуты?

Для изменения интерфейса или формата добавляется короткий пример. Для
производительности указываются воспроизводимые измерения, а не субъективное
ощущение скорости.

## Проверка и объединение

Автор сначала самостоятельно просматривает итоговый diff. Проверяющий обращает
внимание на границы задачи, обработку ошибок, лимиты, тесты, зависимости и
утечки данных.

Изменение криптографии или границы доверия требует отдельного внимательного
обзора. Если второго подготовленного разработчика пока нет, изменение не
объявляется независимо проверенным.

Нельзя объединять изменение только ради соблюдения срока демонстрации, если
известна неисправленная ошибка безопасности.

## Сообщения об уязвимостях

Рабочий эксплойт, приватный ключ и реальные сообщения нельзя публиковать в
обычном issue. Текущий порядок описан в `SECURITY.md`.

Приватный канал ещё не настроен, поэтому публичный приём внешних изменений не
должен открываться до его появления.

## Лицензия вклада

Вклад принимается под лицензией изменяемого файла. Отправляя изменение, автор
подтверждает, что создал его сам или имеет право передать на этих условиях.

Нельзя копировать код, текст, изображение или тестовый вектор без проверки
происхождения и лицензии. Подробная схема находится в `LICENSES.md`.
