# Обзор E2EE для личных сообщений Lantern

Статус: первый обзор завершён 17 июля 2026 года.

## Задача

Для личных сообщений Lantern нужна существующая реализация, которая:

- работает асинхронно, когда Bob находится офлайн;
- меняет ключ сообщения после каждой отправки;
- допускает ограниченную потерю и перестановку;
- обнаруживает изменение ciphertext до показа текста;
- поддерживает одноразовые ключи для начала сессии;
- имеет документированный протокол, Rust API и понятную лицензию;
- не зависит от LAN, Relay или другого транспорта;
- не требует от Lantern самостоятельно собирать криптографическую схему из
  отдельных примитивов.

Телефон, электронная почта и центральный сервер не должны быть частью
криптографической идентичности.

## Короткий результат

Ни один проверенный вариант пока не выполняет все требования без существенной
оговорки.

| Вариант | Что подходит | Почему не подключаем сейчас |
| --- | --- | --- |
| официальный `libsignal` | асинхронный 1:1, PQXDH и Double Ratchet, правильная модель для Lantern | Rust crate не выпускается как обычная стабильная зависимость, API предназначен прежде всего для клиентов Signal, репозиторий использует собственный workspace и AGPL-3.0-only |
| `vodozemac 0.10.0`, Olm v1 | опубликован на crates.io, чистый Rust, Apache-2.0, высокий уровень API, одноразовые ключи, Double Ratchet, независимый аудит | стабильный Olm v1 использует 64-битный MAC, окно перестановки жёстко ограничено 40 ключами на цепочку, а аутентификация опубликованных ключей остаётся задачей приложения |
| `OpenMLS 0.8.1` | RFC 9420, MIT, поддерживаемый Rust, тестовые векторы, свежий аудит | это групповая модель с эпохами, Commit, Authentication Service и Delivery Service; для личного MVP она заметно увеличивает объём состояния и не соответствует утверждённому разделению личных сообщений и будущих групп |
| Noise, HPKE или отдельный AEAD | хорошие стандартные строительные блоки | сами по себе не дают готовую асинхронную сессию, Double Ratchet, одноразовые prekey и защиту от повторного показа; недостающие части пришлось бы проектировать самостоятельно |

## Официальный Signal Protocol

PQXDH создан для ситуации, когда получатель офлайн, но заранее опубликовал
набор ключей. Double Ratchet поддерживает потерянные и переставленные сообщения
через номера сообщений и ограниченное хранение пропущенных ключей. Это лучше
всего совпадает с личным чатом Lantern.

Проблема находится не в спецификации, а в доступной интеграции. Официальный
репозиторий `signalapp/libsignal` содержит реализацию на Rust, но публикует
поддерживаемые интерфейсы для Java, Swift и TypeScript. Внутренний
`libsignal-protocol` имеет версию `0.1.0`, живёт внутри большого workspace и не
предлагается как обычный стабильный crate для стороннего Rust-приложения.
Привязка к случайному Git commit сделала бы обновление и аудит Lantern
непредсказуемыми.

Весь репозиторий лицензирован под `AGPL-3.0-only`. Это не автоматический запрет
для открытого Lantern, но потребует отдельного решения о границе с MPL-crate и
повторной юридической проверки.

## Olm и vodozemac

Olm предназначен для асинхронного канала один-к-одному и использует 3DH,
одноразовые ключи и Double Ratchet. `vodozemac` является поддерживаемой Rust
реализацией Matrix и может использоваться напрямую вне Matrix.

Опубликованная версия `0.10.0` имеет следующие свойства:

- лицензия Apache-2.0;
- минимальный Rust 1.85;
- собственный код запрещает `unsafe`;
- обычный Olm API не зависит от сети;
- X25519 с неконтрибутивным результатом отклоняется;
- строгая проверка Ed25519 включена всегда;
- экспериментальная версия Olm v2 выключена по умолчанию;
- стабильная версия Olm v1 использует усечённый 64-битный HMAC-SHA256;
- на одну receiving chain сохраняются не более 40 пропущенных message key;
- одна сессия хранит не более пяти receiving chain.

Независимый аудит Least Authority 2022 года назвал 64-битный MAC уровнем ниже
обычной современной цели в 128 бит. Замечание осталось открытым, потому что
увеличение тега меняет протокол Olm v1. Аудит также отдельно отметил
неизменяемый предел в 40 пропущенных ключей.

Версия `0.10.0` исправила опубликованную в феврале 2026 года проблему с
неконтрибутивными X25519-ключами. Это важное исправление, но оно не меняет длину
MAC в Olm v1.

Экспериментальный Olm v2 использует полный MAC. Подключать его нельзя: формат
ещё не является стабильной спецификацией, feature прямо назван
`experimental-session-config`, а совместимых production-клиентов пока нет.
Самостоятельно менять длину тега Olm v1 тоже нельзя, потому что это будет новая
несовместимая криптографическая схема Lantern.

## OpenMLS

MLS описан в RFC 9420 и умеет защищать сообщения при недоверенной Delivery
Service. Application message содержит эпоху и generation, поэтому стандарт
допускает ограниченную перестановку. OpenMLS `0.8.1` поддерживается, использует
MIT и прошёл аудит SRLabs в 2026 году. Найденные проблемы высокой и средней
важности исправлены в ветках `0.8.1` и `0.7.3`; одно замечание низкой важности
на момент публикации отчёта ещё исправлялось.

Для Lantern Diploma MVP это неправильный уровень сложности. Даже группа из
двух устройств требует group state, KeyPackage, Welcome, эпохи, Commit,
разрешение конфликтов и отдельную политику Authentication Service. Исходный
план оставляет MLS будущим закрытым группам, а не личному чату v0.1.

## Аутентификация контакта

Даже готовая ratchet-библиотека не решает, кому принадлежит публичный ключ.
Для первой версии безопаснее всего оставить утверждённый очный обмен:

1. устройства показывают QR с версией, публичными identity key и одноразовым
   приглашением;
2. оба пользователя сравнивают короткое представление отпечатка;
3. контакт сохраняется только после подтверждения;
4. последующие новые key bundle принимаются только с подписью уже проверенного
   identity key;
5. изменение identity key всегда требует явного предупреждения и повторной
   проверки.

На момент обзора точный формат QR, подписи bundle и хранения приватных ключей
ещё не был выбран. Формат двух QR и SAS-проверка позже зафиксированы в
`CONTACTS.md` и ADR 0014. Способ хранения приватных ключей остаётся открытым.

## recipient_hint

`recipient_hint` нельзя вычислять напрямую из постоянного публичного ключа.
Такой hint был бы стабильным идентификатором и позволил бы разным Relay
связывать сообщения одного получателя.

Предварительная модель для будущей проверки:

- случайные 16 байт на конкретную пару устройств;
- первое значение передаётся внутри лично проверенного contact bundle;
- следующее значение передаётся только внутри аутентифицированного E2EE;
- получатель некоторое время принимает текущее и предыдущее значения;
- старое значение удаляется не раньше максимального TTL и срока локальной
  дедупликации;
- Relay не получает связь между старым и новым значением от протокола.

Эта предварительная модель позже уточнена в `CONTACTS.md` и ADR 0014. Перед
реализацией всё ещё нужны тесты потери, перестановки и отсутствия ответа.

## Что нужно проверить до окончательного выбора

Для `vodozemac` можно сделать изолированный совместимый эксперимент, не
подключая его к `lantern-node`:

- закрепить ровно версию `0.10.0` и отключить default features;
- построить сессию через `Account`, one-time key и `SessionConfig::version_1()`;
- проверить положительный обмен Alice и Bob;
- изменить каждый байт сообщения и убедиться в полном отказе без изменения
  состояния;
- проверить повтор уже принятого ciphertext;
- проверить потерю и все перестановки в окне до 32 сообщений;
- проверить поведение за пределом 40 сохранённых ключей;
- сохранить и восстановить зашифрованные account и session pickle;
- проверить, что ошибки и `Debug` не раскрывают plaintext и приватные ключи;
- получить точное дерево зависимостей, лицензии и результаты `cargo audit`;
- отдельно решить, допустим ли 64-битный MAC для experimental diploma MVP.

Этот эксперимент не означает принятие Olm для production. Если 64-битная
граница будет отклонена, код эксперимента не должен попасть в основной путь
Lantern.

Эксперимент завершён 17 июля 2026 года. Все восемь compatibility-тестов прошли,
Проверка подтвердила окно в 40 ключей и обнаружила, что
`InboundCreationResult` раскрывает первый plaintext через `Debug`. По итогам
`vodozemac 0.10.0` условно принят только для `Lantern v0.1 experimental
preview`. Обязательные условия записаны в
`docs/adr/0013-use-vodozemac-for-experimental-mvp.md`, полный результат - в
`experiments/vodozemac-compat/RESULTS.md`.
