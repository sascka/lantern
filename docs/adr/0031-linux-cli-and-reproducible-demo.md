# ADR 0031: Linux CLI и воспроизводимая демонстрация

Статус: принято и реализовано 18 июля 2026 года.

## Контекст

Библиотечный маршрут Alice -> Relay -> Bob уже работал, но пользователь не мог
повторить его обычными процессами из терминала. Последнему пакету Этапа 4 нужны
были минимальный Linux CLI, реальная граница вывода и одна документированная
демонстрация без функций за пределами MVP.

Временный объект `SasHandle` из vodozemac нельзя сериализовать. Поэтому обмен
контактами нельзя безопасно разбить на независимые команды с сохранением
промежуточного состояния.

## Решение

Добавлено приложение `lantern-cli` под `AGPL-3.0-or-later`. Оно только собирает
существующие библиотечные границы:

- секретный профиль остаётся в `lantern-secret-storage`;
- шифрование выполняет `lantern-crypto`;
- зашифрованный outbox переносит `lantern-bridge`;
- открытая очередь принадлежит `lantern-node`;
- сокет и одна встреча остаются в `lantern-lan`.

Контакт добавляют два одновременно работающих процесса. Они передают два
подписанных QR и два зашифрованных подтверждения через приватные файлы. Оба
показывают SAS и требуют точную строку `MATCH`. При остановке процесс не
сохраняет временный SAS: обмен начинается заново.

Парольная фраза и сообщение читаются только из обычных файлов `0600` с одной
жёсткой ссылкой. Каталоги профиля, узла и обмена имеют права `0700`. Размеры
проверяются до чтения. CLI не отражает путь, ввод или секрет в тексте ошибки.

Встроенный SQLCipher собирается с Clang. GCC 16.1.1 в текущем Arch Linux
создаёт для его thread-local генератора некорректные `R_X86_64_DTPOFF32`
relocation при оптимизации. Debug-сборка скрывала проблему, но release linker
её правильно отклонил. `.cargo/config.toml` фиксирует только C-компилятор, не
меняя исходники SQLCipher или параметры криптографии.

Команды `receive` и `inbox` являются единственными местами, где открытый текст
намеренно появляется в stdout. Диагностика выводит закрытые коды, категории и
ограниченные счётчики без ID, сетевого адреса и текста.

## Проверка

Интеграционный тест запускает отдельные процессы Alice, Relay и Bob. Он:

- создаёт два постоянных профиля и три узла;
- выполняет полный контактный обмен и сравнивает показанный SAS;
- создаёт один настоящий зашифрованный Envelope;
- соединяет Alice только с Relay;
- завершает первую встречу до соединения Relay только с Bob;
- открывает сообщение у Bob один раз;
- проверяет вывод и файлы Relay на тестовые секреты и plaintext.

Тест вводит `MATCH` автоматически и сравнивает обе строки SAS. Это проверяет
протокол и CLI, но не человеческое внимание при ручной сверке.

Ручная последовательность записана в `DEMO.md`. Скрипт
`scripts/verify-v0.1.sh` запускает форматирование, clippy, тесты Rust, ruff и
pytest на чистом коммите и печатает его хеш.

## Последствия

Плюсы:

- главный сценарий можно повторить без тестового API;
- Relay запускается без секретной базы и функции decrypt;
- криптография не зависит от файлового обмена и TCP;
- вывод имеет небольшую понятную границу;
- CLI не добавляет сторонних зависимостей workspace.

Цена и ограничения:

- четыре файла контактного обмена неудобны и существуют только для MVP;
- сбой между локальными commit контакта может оставить стороны в разных
  состояниях; v0.1 требует пересоздать оба демонстрационных профиля;
- для сборки на текущем Arch Linux нужен Clang;
- два процесса должны оставаться запущенными не больше двух минут;
- нет камеры, QR-изображения, обнаружения соседей и фонового демона;
- ручной адрес раскрывает IP соседу и не аутентифицирует транспорт;
- кража разблокированного профиля раскрывает ключи, контакты и историю;
- кража открытой очереди раскрывает зашифрованные контейнеры и метаданные;
- тест на loopback не заменяет испытание трёх физических устройств;
- реализация не проходила fuzzing CLI, независимый аудит и проверку
  специалистом по удобству SAS.
