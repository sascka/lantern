# ADR 0005: ограниченная безопасная диагностика

Статус: принято 17 июля 2026 года. Постоянный режим позже добавлен в ADR 0010.

## Контекст

MVP требует технический журнал без текста сообщений и секретов. Обычный
строковый logger слишком широк для этой границы: форматирование ошибки или
структуры может случайно вывести payload, ID, путь к базе, сетевой адрес или
точное время.

На этом этапе ещё нет процесса узла, CLI и транспорта. Значит, нам нужен не
готовый файл логов, а безопасный язык событий, который будущие компоненты не
смогут случайно наполнить произвольными данными.

## Решение

Создаётся независимый crate `lantern-diagnostics` без сторонних зависимостей.
Он не подключает глобальный logger и ничего не выводит самостоятельно.

`DiagnosticEvent` содержит только:

- `EventCode` из закрытого списка технических этапов;
- `EventOutcome` из закрытого списка общих результатов;
- `object_count` от 0 до 10 000;
- `SizeBucket` вместо точного размера;
- `DurationBucket` вместо точной длительности.

API не принимает строки, пути, IP-адреса, `message_id`, `recipient_hint`,
Envelope, payload и внешние объекты ошибок. Добавление нового поля потребует
изменения публичного типа и отдельной проверки.

## Журнал

`DiagnosticJournal` хранит записи в `VecDeque`. По умолчанию разрешено 2048
записей и 64 KiB логических данных. Максимальная конфигурация API ограничена
10 000 записями и 320 000 логическими байтами. Одна запись учитывается как 32
логических байта.

При заполнении удаляется самая старая запись. Каждая запись получает только
последовательный номер. Абсолютное время и срок истечения находятся во
внутренней структуре и не возвращаются через публичную запись или `Debug`.

Срок можно уменьшить, но нельзя увеличить выше 7 суток. На точной границе
запись удаляется.

Обслуживание ленивое и запускается при записи, явном `maintain` или создании
`view`. Прочитать записи без обслуживания времени публичный API не позволяет.
При полном бездействии процесса истёкшая запись может физически оставаться в
памяти до следующего вызова. Будущий runtime должен вызывать обслуживание
периодически.

Логическая байтовая квота не равна точному расходу heap. `VecDeque`,
`expires_at` и выравнивание добавляют память. Число объектов ограничено
отдельно, поэтому эта разница не приводит к неограниченному росту.

## Работа времени

Журнал не получает системное время сам. Вызывающий компонент передаёт локальное
значение секунд.

Если время уменьшилось, все существующие записи удаляются. Результат сообщает
об откате даже для пустого журнала. Последовательный номер не сбрасывается.

Такой подход выбирает раннюю потерю диагностики вместо неопределённого
продления срока. Скачок часов вперёд также может удалить записи раньше. Для
необязательного журнала это безопаснее бессрочного хранения метаданных.

## Почему в этом пакете не было файла

Постоянные диагностические файлы по умолчанию запрещены политикой приватности.
Пользователь сможет включить их только явно. Предельные 10 MiB из политики
относятся к будущему файлу, а не к in-memory журналу.

Добавление файла требовало отдельно определить:

- формат и строгий декодер;
- атомарность и восстановление после частичной записи;
- ротацию и физический предел 10 MiB;
- права и защиту пути;
- срок после перезапуска и отката часов;
- данные, видимые при краже файла.

Без процесса узла этот код нельзя было правильно подключить или проверить в
реальном сценарии. Поэтому пятый пакет завершил типизированный in-memory журнал.
Явно включаемый постоянный режим появился после жизненного цикла узла и описан
в `docs/adr/0010-persistent-diagnostics-and-profile-lock.md`.

## Рассмотренные альтернативы

### `log` или `tracing`

Эти библиотеки подходят обычным приложениям, но их API разрешает строки и
произвольные structured fields. Фильтр после форматирования слишком поздний:
секрет уже мог попасть в буфер или обработчик.

Их можно будет использовать как внешний backend только через адаптер, который
получает готовый `DiagnosticRecord` и не принимает исходные объекты Lantern.
Сейчас зависимость не нужна.

### Записывать только счётчики

Агрегаты раскрывают меньше порядка событий, но плохо объясняют переходы
состояния и ошибки восстановления. Ограниченная последовательность кодов даёт
больше пользы для отладки без идентификаторов.

### Сохранять точное время

Точная временная линия заметно облегчает сопоставление журнала с сетевыми
встречами. Для MVP достаточно порядка событий и грубой длительности операции.
Поэтому абсолютное время скрыто.

### Писать журнал в базу очереди

Это позволило бы переиспользовать SQLite, но смешало бы обязательное состояние
доставки с необязательной диагностикой. Диагностика по умолчанию должна быть
отключаемой и удаляемой независимо.

## Проверка угроз

- Секретные данные не имеют представления в публичном типе события.
- Взломанный Relay увидит коды, порядок, счётчики и грубые категории активности.
- Кража локальной базы очереди не раскрывает журнал, потому что он туда не
  записывается.
- Изменение сообщения не проверяется журналом и остаётся задачей E2EE.
- Повтор пакета может создать повтор технического события, но объём журнала
  ограничен.
- Поток новых событий вытесняет старые и не увеличивает память бесконечно.
- Журнал не инициирует передачу и не может вызвать распространение пакетов.
- Порядок и объём событий частично раскрывают активность, но не личности и
  непосредственные связи.
- В лог не попадают секреты, пока вызывающий код использует только этот API.
- Откат часов очищает журнал, скачок вперёд удаляет записи раньше.
- Потеря и перестановка пакетов могут изменить порядок кодов, но не нарушают
  инварианты журнала.
- Аварийное завершение теряет in-memory журнал. Постоянный режим рассматривается
  отдельно в ADR 0010.

## Последствия

Плюсы:

- случайно записать строку, ID или Envelope через API нельзя;
- базовый режим не создаёт глобальное состояние, stdout, stderr, файлы или
  телеметрию;
- память, срок и счётчики имеют жёсткие границы;
- точные размеры и длительности отбрасываются сразу;
- абсолютное время не выдаётся потребителю;
- откат часов не продлевает хранение;
- для базового режима новая цепочка зависимостей не появляется;
- в собственном коде нет `unsafe`.

Минусы и риски:

- in-memory журнал исчезает при завершении процесса и не помогает исследовать
  crash;
- в пятом пакете компоненты ещё не были подключены к журналу автоматически;
- точный расход heap выше логической квоты;
- последовательность кодов и счётчиков всё равно показывает активность;
- злонамеренный вызывающий код может построить скрытый канал из кодов и чисел;
- неверные часы могут удалить диагностику раньше срока;
- без периодического вызова `maintain` физическое удаление может запоздать;
- в пятом пакете файловая диагностика и её восстановление ещё не были
  реализованы;
- независимый аудит не проводился.
