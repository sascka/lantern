# ADR 0015: версионированный Olm payload и кандидатный ratchet

Статус: принято 17 июля 2026 года.

## Контекст

Envelope Lantern содержит открытые маршрутные поля и непрозрачный
`protected_payload`. Olm v1 защищает plaintext, но его API не принимает
external associated data. Если зашифровать только текст, Relay сможет изменить
`message_id`, `recipient_hint`, TTL или hop limit без криптографического
обнаружения.

Кроме этого, транспорт должен отличать `OlmMessage::PreKey` от
`OlmMessage::Normal`. `vodozemac` возвращает тип и полные двоичные bytes через
`OlmMessage::to_parts()` и восстанавливает сообщение через `from_parts()`.

Успешная расшифровка меняет Session. Успешное создание входящей сессии также
удаляет использованный one-time key из Account. Значит, проверять копию полей
Envelope только после необратимого изменения единственного живого состояния
нельзя.

## Решение

Использовать два строго версионированных уровня из `PAYLOAD.md`:

1. Каноническая открытая CBOR-оболочка содержит версию, Olm type и полные bytes
   из `to_parts()`.
2. Канонический внутренний CBOR содержит тип сообщения, копию защищаемых полей
   Envelope и type-specific данные. Он целиком шифруется Olm v1.

Внутренние `protocol_version`, `message_id`, `recipient_hint`, `ttl_seconds`,
`max_hops` и `priority` должны точно совпасть с Envelope. Внутренняя версия
оболочки также должна совпасть с открытой.

В v0.1 есть только семь внутренних типов: пользовательский текст, два
подтверждения контакта и четыре шага смены hint. Неизвестные поля и типы
отклоняются. Механизма произвольных extensions нет.

## Ограничения размера

Для crypto-профиля v0.1 полный `protected_payload` ограничен 8192 байтами,
несмотря на общий предел Lantern Core в 63 KiB. Расшифрованный внутренний CBOR
не может быть больше 4155 байт, а пользовательский UTF-8 text - больше 4096
байт.

Более узкий предел уменьшает стоимость недоверенного crypto-входа. Запас до 8
KiB нужен для metadata, MAC, padding и pre-key заголовков Olm. Перед
реализацией граница подтверждается тестом на точной версии `vodozemac 0.10.0`.

## Транзакционная обработка ratchet

Account или Session не расшифровывает вход прямо в единственном сохранённом
состоянии. Crypto-слой создаёт отдельного кандидата из последнего committed
encrypted pickle.

Кандидат выполняет decrypt, после чего Lantern проверяет:

- предел plaintext;
- строгий внутренний CBOR;
- совпадение всех внешних полей;
- разрешённый inner type;
- ID и secret контактного обмена;
- generation и состояние смены hint;
- допустимость локального перехода.

Только после всех проверок новый pickle, дедупликация и изменение контакта
сохраняются атомарно. При любой ошибке кандидат уничтожается, а committed
Account или Session остаётся прежним.

Это защищает доступность первого контакта. Relay может поменять открытый
`message_id` у настоящего pre-key Envelope. Olm plaintext останется корректным,
но внутренняя копия ID обнаружит подмену. Без кандидата библиотека уже удалила
бы one-time key и оригинал позже нельзя было бы принять.

## Исходящий pending outbox

Encrypt тоже изменяет Session. Новый ratchet state нельзя сохранить отдельно
от результата encrypt, иначе сбой создаст пропущенный ключ или повторное
шифрование одного сообщения.

Будущее секретное хранилище одной транзакцией записывает:

- новый encrypted Session pickle;
- полный неизменяемый Envelope;
- состояние исходящего служебного перехода, если оно изменилось;
- статус pending outbox.

После commit тот же Envelope идемпотентно переносится в открытую маршрутную
очередь. Сбой переноса не вызывает новый encrypt. Точная схема восстановления
между двумя хранилищами принимается вместе с решением о защите локальных
секретов.

## Потеря и перестановка

Один логический исходящий объект шифруется один раз. Повторная доставка всегда
использует сохранённые bytes Envelope.

Если корректный plaintext пришёл раньше необходимого локального перехода,
кандидатный ratchet не сохраняется, plaintext не записывается, а Envelope
откладывается до изменения состояния. На контакт планируется не больше 32
таких кандидатов. Это не расширяет окно Olm, которое в experimental MVP
проверяется только на перестановке 32 сообщений.

## Почему не используем associated data

Olm v1 не предоставляет такой параметр. Добавлять собственный MAC поверх Olm
означало бы проектировать новый криптографический протокол и управлять ещё одним
ключом.

Копия полей внутри зашифрованного канонического CBOR даёт нужное связывание без
нового примитива. Цена решения - расшифрование выполняется до обнаружения
изменённого внешнего поля. Поэтому обязательны предел входа, кандидатный ratchet
и будущий общий rate limit.

## Почему не используем serde и Base64

Стандартная serde-форма `OlmMessage` предназначена для Matrix JSON и содержит
числовой type и Base64 body. Envelope Lantern уже является бинарным CBOR.
Дополнительный текстовый слой увеличил бы размер, число зависимостей и варианты
ошибок.

`to_parts()` и `from_parts()` сохраняют ровно те же полные Olm bytes и явно
оставляют транспортный type рядом с ними.

## Почему не храним только внутренний ciphertext

Метод `OlmMessage::message()` возвращает только зашифрованную часть вложенного
Olm message. Для ratchet и первого 3DH нужны также version, chain index,
ratchet key и pre-key metadata. Поэтому Lantern хранит полный результат
`to_parts()`.

## Почему нет padding и compression

Compression для 4 KiB текста почти не помогает MVP, но добавляет сложный
декодер и риск чрезмерного распакованного размера. Padding уменьшил бы часть
утечки размеров, но увеличил трафик и потребовал отдельной политики классов
размера.

Оба механизма не входят в v0.1. Relay видит размер и может приблизительно
отличать короткие служебные сообщения от длинного текста. Это явно остаётся в
модели угроз.

## Почему text является одной строкой

Linux CLI не должен выводить расшифрованные terminal control sequences. В v0.1
декодер принимает от 1 до 4096 UTF-8 bytes и отклоняет control characters.
Текст не нормализуется, не разбирается как Markdown и не влияет на
идентификаторы или ключи.

Многострочный редактор и богатое форматирование не входят в MVP.

## Фиксированные параметры служебных Envelope

Подтверждения контакта и смена hint используют:

- TTL 7 суток;
- max hops 2;
- обычный priority 0.

Два hop достаточны для сценария Alice -> Relay -> Bob. Фиксированные значения
убирают лишние варианты состояния и не дают служебному сообщению жить дольше
максимального TTL. Для другой топологии понадобится новое решение.

## Отклонённые варианты

### Шифровать только текст

Открытые маршрутные поля можно было бы менять независимо от ciphertext. Это
нарушает связь сообщения с его Envelope.

### Подписывать Envelope отдельным identity key

Подпись раскрыла бы постоянный идентификатор отправителя или потребовала бы
скрывать и передавать её внутри Olm, где она уже не добавляет нужного
связывания. Она также не решает атомарное состояние ratchet.

### Доверять успешному Olm decrypt до inner validation

Olm подтверждает целостность своего plaintext, но не знает внешний Envelope и
состояние контакта Lantern. Успешной расшифровки недостаточно.

### Откатывать поля Session вручную

Double Ratchet имеет связанные sending chain, receiving chain и skipped keys.
Ручной откат внутренностей библиотеки потребовал бы low-level API и легко
повредил бы состояние. Восстановление отдельного кандидата из committed pickle
сохраняет границу библиотеки.

### Сразу использовать предел 63 KiB

MVP разрешает только 4 KiB plaintext. Большой crypto-вход увеличивает память и
стоимость подделанных попыток без пользовательской пользы.

## Последствия для модели угроз

- Взлом Relay раскрывает Olm type, ciphertext size и маршрутные поля, но не
  должен раскрывать inner type или plaintext.
- Кража открытой очереди раскрывает те же metadata и ciphertext. Pickle и
  plaintext там не хранятся.
- Изменение любого защищаемого поля Envelope обнаруживается после decrypt и не
  сохраняет кандидатный ratchet.
- Повтор не должен второй раз показать текст или повторно изменить контакт.
- Пределы 8192 и 4155 байт ограничивают память одного crypto-входа.
- TTL, max hops, очередь и один сохранённый Envelope на служебный шаг
  ограничивают распространение.
- Размер и время сообщений всё ещё помогают анализировать социальный граф.
- Секреты, ciphertext и точные ratchet counters запрещены в диагностике.
- Неверное время не входит во внутренний формат и не используется для ключей.
- Потеря и перестановка могут задержать или сорвать доставку, но не разрешают
  принять изменённый plaintext.

## Риски

- У Olm v1 остаётся 64-битный MAC.
- Изменённый внешний Envelope всё равно может вызвать decrypt до обнаружения
  несовпадения.
- Общий постоянный лимит неудачных попыток пока не выбран.
- Кандидатное состояние и pending outbox зависят от ещё не выбранного
  зашифрованного хранилища.
- Отложенные Envelope добавляют состояние и требуют тестов после перезапуска.
- 8 KiB предел пока обоснован расчётом и должен быть подтверждён работающим
  тестом.
- Документ не заменяет fuzzing или независимый аудит реализации.
