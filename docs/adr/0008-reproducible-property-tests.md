# ADR 0008: воспроизводимые property-тесты

Статус: принято 17 июля 2026 года.

## Контекст

Точечные unit-тесты уже проверяют известные граничные случаи, но не
покрывают все сочетания длин, порядка операций, скачков времени и
заполнения квот. Для этого нужны генерируемые наборы данных с явно
записанными свойствами.

## Решение

Для тестов выбран `proptest` 1.11.0. Версия закреплена точно, default features
отключены, включена только feature `std`. Зависимость добавлена в
`dev-dependencies` и не входит в обычную сборку Lantern.

Каждая группа тестов использует ChaCha и свой фиксированный seed. Обычный
повторный запуск тех же тестов получает ту же последовательность данных.
При ошибке `proptest` сокращает исходный пример, чтобы причину было проще
повторить и разобрать.

Первый набор проверяет:

- строгий декодер на произвольных байтах;
- каноническое кодирование корректных Envelope;
- квоты и внутренние связи очереди после длинных цепочек операций;
- неуменьшение логического времени при скачках системных часов;
- неизменность состояния после отклонённого отката `Instant`;
- раздельные квоты кадров и байтов транспортной сессии.

## Отклонённые варианты

### Только ручные табличные тесты

Они просты и дают точно заданные примеры, но плохо исследуют сочетания
операций. Точечные тесты остаются рядом с property-тестами.

### Написать свой генератор

Свой цикл случайных тестов добавит код для генерации, сокращения и сохранения
найденных примеров. Эта инфраструктура не является целью Lantern.

### Сразу перейти к fuzzing

Fuzzing нужен перед публичным релизом, но это отдельный долгоживущий процесс.
Он не заменяет быстрые воспроизводимые тесты, которые запускаются вместе с `cargo test`.

## Последствия и ограничения

- Тестовое дерево зависимостей стало больше.
- Фиксированный seed даёт воспроизводимость, но исследует одни и те же цепочки.
- Переменные окружения `PROPTEST_*` могут намеренно изменить seed, число
  примеров и другие настройки локального прогона.
- Генеративные тесты не доказывают отсутствие ошибок.
- Произвольные байты ограничены 8 KiB, а генерируемый payload - 4 KiB, чтобы
  полная проверка оставалась быстрой. Точные границы 63 KiB и 64 KiB по-прежнему
  проверяются unit-тестами.
- Этот пакет не заменяет fuzzing, аудит и тесты реальной сети.
