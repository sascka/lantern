# ADR 0002: строгий CBOR-кодек Envelope

Статус: принято 17 июля 2026 года.

## Контекст

`PROTOCOL.md` требует одно детерминированное представление Envelope и строгий
разбор недоверенных байтов. До создания очереди ядро должно уметь отклонять
слишком большой, неканоничный или неоднозначный контейнер.

Обычного успешного вызова CBOR-декодера недостаточно. Некоторые библиотеки
принимают неопределённые длины, повторяющиеся ключи, неизвестные поля и разные
представления одного числа. Такое поведение мешает сравнивать контейнеры и
расширяет поверхность для атак на разбор.

## Решение

Для `lantern-core` выбрана библиотека `minicbor` версии 2.2.2. Версия закреплена
точно. Отключены стандартные features, включён только `alloc`:

```toml
minicbor = { version = "=2.2.2", default-features = false, features = ["alloc"] }
```

У этой конфигурации нет транзитивных зависимостей. Библиотека используется для
чтения и записи элементов CBOR, но правила схемы проверяет отдельный код
Lantern.

Декодер применяет проверки в следующем порядке:

1. отклоняет пустой вход и вход больше 64 KiB;
2. принимает только map с заранее указанной длиной 7;
3. требует ключи `0`-`6` строго по порядку;
4. принимает только типы, указанные в схеме;
5. сравнивает заголовки map, чисел и byte string с самым коротким
   представлением, созданным `minicbor`;
6. проверяет длины и диапазоны до копирования payload;
7. требует конец входа сразу после Envelope;
8. только после всех проверок копирует `protected_payload` в принадлежащий
   объект.

Зафиксированный порядок ключей одновременно запрещает пропуски, перестановку,
повторы и неизвестные поля. Неопределённые длины, теги, float, текстовые ключи
и вложенные значения отклоняются.

Ошибки содержат только безопасную категорию и имя поля. В них нет входных
байтов, идентификаторов или payload. В кодеке нет логирования.

## Почему `minicbor`

`minicbor` даёт низкоуровневый декодер над заимствованным срезом, сообщает тип
следующего значения и текущую позицию, а definite byte string возвращает без
копирования. Это позволяет проверить размер и каноничность до выделения памяти
под payload.

Проверены документация версии 2.2.2, исходный репозиторий, состав features,
дерево зависимостей и лицензия. Библиотека распространяется под
`BlueOak-1.0.0`. Полный текст находится в
`LICENSES/BlueOak-1.0.0.txt`.

Поиск опубликованных advisory на дату решения не нашёл записи для `minicbor`.
Это не доказывает отсутствие уязвимостей и не заменяет аудит исходного кода.

## Рассмотренные альтернативы

### `ciborium` 0.2.2

Библиотека работает через Serde и намеренно допускает более свободное
декодирование. Для обычного обмена данными это удобно, но строгой границе
Lantern пришлось бы отдельно доказывать обработку повторов, порядка и
неканоничных значений.

### `serde_cbor` 0.11.2

Эта версия отмечена RustSec как неподдерживаемая. Подключать неподдерживаемый
парсер недоверенных данных к новому проекту нельзя.

### Собственный CBOR-парсер

Такой вариант дал бы полный контроль, но добавил бы собственную реализацию
сложного внешнего формата. Это лишний риск и противоречит правилу не создавать
самодельные безопасностно-критичные примитивы без необходимости.

### Автоматический derive для всей структуры

Derive уменьшил бы объём кода, но скрыл бы порядок проверок и мог выделить
память под payload раньше проверки всей схемы. Ручной обход фиксированных семи
полей здесь нужен именно как граница недоверенного ввода.

## Последствия

Плюсы:

- один Envelope имеет одно принимаемое CBOR-представление;
- общий лимит проверяется до разбора;
- payload копируется только после проверки структуры и диапазонов;
- транспорт сможет передавать ядру только ограниченный срез байтов;
- кодек не зависит от сети, базы и криптографии.

Минусы и риски:

- строгая версия 1 не допускает неизвестных полей для постепенного расширения;
- корректность границы зависит и от `minicbor`, и от нашего проверяющего кода;
- детерминированный негативный набор тестов не заменяет fuzzing;
- независимый аудит не проводился;
- лицензия BlueOak менее знакома многим разработчикам, поэтому её полный текст
  хранится рядом с остальными лицензиями.

## Следующий пакет

После этого решения можно реализовать ограниченную очередь и дедупликацию по
`message_id`. Они должны принимать уже проверенный Envelope и не разбирать CBOR
повторно.
