# ADR 0012: не подключать E2EE после первого обзора

Статус: заменено ADR 0013 17 июля 2026 года.

## Контекст

После завершения Lantern Core нужно выбрать готовую криптографическую
реализацию для личного асинхронного чата. Полный обзор находится в
`docs/research/CRYPTOGRAPHY_REVIEW.md`.

Официальный Signal Protocol лучше всего соответствует требуемой модели, но его
Rust-реализация не выпускается как обычный стабильный crate для сторонних
приложений. OpenMLS имеет стандартизованный и поддерживаемый Rust API, но
решает более сложную задачу групп и оставлен будущей версии Lantern.

Ближайший технический кандидат - Olm через `vodozemac 0.10.0`. Он опубликован,
поддерживается, прошёл независимый аудит и предоставляет асинхронную сессию
один-к-одному с Double Ratchet. При этом стабильный Olm v1 использует
64-битный MAC, а окно сохранённых пропущенных ключей ограничено 40 сообщениями
на receiving chain.

## Решение

Не добавлять криптографическую зависимость в рабочий путь Lantern по результату
одного документального обзора.

Разрешён следующий изолированный шаг - compatibility-эксперимент с точно
закреплённым `vodozemac 0.10.0`, отключёнными default features и только
стабильным `SessionConfig::version_1()`. Эксперимент не подключается к
`lantern-node`, хранилищу очереди или транспорту и не обрабатывает реальные
сообщения.

Эксперимент должен закончиться отдельным решением: принять ограничение Olm v1
для diploma MVP или отказаться от кандидата. Экспериментальный Olm v2, fork
`vodozemac`, собственная длина MAC и самостоятельная реализация Signal
Protocol запрещены.

## Почему не подключаем сразу

- Независимый аудит считает 64-битный MAC ниже современной 128-битной цели.
- Жёсткое окно в 40 ключей нужно проверить на модели store-and-forward.
- Аутентификация contact bundle находится вне Olm session API.
- Полное дерево зависимостей и лицензий ещё не зафиксировано в Lantern.
- Нужно проверить, что ошибка расшифрования не меняет сохранённое состояние.
- Официальные положительные и отрицательные векторы для нашей границы API ещё
  не определены.

## Последствия

Криптографический рубеж остаётся незавершённым. Этап 3 можно разрабатывать
только на очевидно тестовом непрозрачном payload. Настоящий текст, ключи и
пользовательские данные в транспорт не передаются.

В workspace не появляется новая runtime-зависимость, `Cargo.lock` и
`THIRD_PARTY_NOTICES.md` не меняются.

## Риски

- Подходящий стабильный Rust API Signal Protocol может не появиться вовремя для
  diploma MVP.
- Результат эксперимента может заставить отказаться от `vodozemac`.
- Даже успешный эксперимент не заменяет аудит всей интеграции Lantern.
- Выбор Olm v1 потребует честно зафиксировать 64-битную границу аутентификации и
  более узкое окно перестановки, чем общая очередь Envelope.
