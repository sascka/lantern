# ADR 0019: фиксированный LAN Hello и ручной адрес

Статус: принято и реализовано 17 июля 2026 года.

## Контекст

Этап 2 оставил `lantern-transport` независимым от сокетов. Первый шаг Этапа 3
должен соединить два Linux-процесса и отклонить неизвестную версию до чтения
`Envelope`. На этом шаге ещё не нужны обнаружение соседей, список контейнеров
или wire framing переменной длины.

## Решение

Добавлен отдельный crate `lantern-lan`. Он использует блокирующий TCP только с
числовым ручным адресом. Общая ограниченная сессия `lantern-transport` при этом
не получает зависимости от TCP.

Обе стороны сначала отправляют одинаковый `Hello` длиной 8 байт, а затем
читают ответ. Формат содержит magic, kind, version и два нулевых служебных
поля. Любое неизвестное значение отклоняется. Один общий монотонный deadline 5
секунд покрывает отправку и чтение полного `Hello`.

Разрешены loopback и числовые адреса локальных диапазонов. Публичные, wildcard,
multicast и DNS-адреса отклоняются до сетевого вызова. Ошибки имеют закрытые
категории и не сохраняют IP, порт или строку ошибки ОС.

В crate входит временный `lantern-lan-probe`. Он позволяет проверить один
handshake двумя процессами и сразу закрывает соединение. Это инструмент
разработки, а не пользовательский CLI Lantern.

## Почему не переменный CBOR Hello

В первой версии нет набора возможностей для согласования. CBOR потребовал бы
длины, декодера и дополнительных правил до того, как установлен формат кадров.
Фиксированные 8 байт проще проверить полностью и нельзя раздуть недоверенным
полем длины.

Когда реально понадобится согласование нескольких версий или возможностей,
будет определена новая версия handshake. Ненулевые flags версии 1 для этого не
используются.

## Почему пока без async runtime

Текущий шаг обслуживает одно соединение и проверяет только границу протокола.
Tokio или другой runtime добавил бы большую зависимость до появления
параллельных сессий, отмены и фонового обнаружения. Блокирующий probe не
определяет архитектуру будущего процесса узла.

## Риски

- `Hello` не аутентифицирует соседа и передаётся открыто.
- Ограничение диапазона IP не доказывает физическую близость устройства.
- IP и время соединения видны операционной системе и наблюдателю сети.
- Один принятый узел может занимать probe до 5 секунд.
- Общего rate limit по IP и между повторными соединениями пока нет.
- В первом пакете API ещё не передавал кадры и не проверял обрыв посередине
  кадра. Этот риск закрыт следующим решением ADR 0020.

## Следующий шаг

согласованного TCP-соединения.
