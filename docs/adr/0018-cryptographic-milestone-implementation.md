# ADR 0018: граница рабочего криптографического контура v0.1

Статус: принято и реализовано 17 июля 2026 года.

## Контекст

ADR 0013-0017 выбрали Olm v1, два контактных QR, строгий внутренний формат,
постоянный limiter и SQLCipher. До Этапа 3 нужно было проверить, что эти
решения складываются в один контур, а транспорт по-прежнему видит только
Envelope.

## Решение

Добавлен отдельный crate `lantern-crypto`. Он зависит от `lantern-core`,
`lantern-secret-storage` и точно зафиксированного `vodozemac 0.10.0`, но не
зависит от `lantern-transport`, `lantern-node` или сетевого API.

Crate реализует:

- подписанные invite и response точного размера;
- Base64URL QR с пределом до декодирования;
- очную SAS-проверку через публичный API `vodozemac`;
- два зашифрованных подтверждения и одну двустороннюю Olm-сессию;
- строгую Olm-оболочку и семь внутренних типов;
- сверку защищённой копии полей с внешним Envelope;
- кандидатный Account и Session при входящей обработке;
- исходящий chat с transactional outbox;
- входящий chat с reservation до decrypt и commit до выдачи текста.

Секретное хранилище отдельно фиксирует ratchet, историю, limiter и outbox.
Открытая очередь остаётся в `lantern-storage`. Между базами нет общей
транзакции: один и тот же готовый Envelope повторно копируется из pending
outbox, но второй encrypt не выполняется.

## Поправки к ранней спецификации

Публичный SAS API принимает `&str`, а не произвольные bytes. Полные подписанные
bundle поэтому входят в SAS-контекст через Base64URL с ASCII-длинами. Это
однозначное кодирование входа для штатного HKDF библиотеки, а не новый KDF.

Приватная часть `Sas` не сериализуется публичным API. Если процесс завершился
до ручного подтверждения, текущий обмен отменяется и начинается с новых QR.
Завершённые контакты и Olm-сессии восстанавливаются из SQLCipher.

Отменённый invite больше не выбирается приложением по hint и ID. Публичный API
Account не позволяет удалить один конкретный неиспользованный one-time key.
Такой ключ может оставаться внутри зашифрованного Account до вытеснения
следующими ключами, но сеть не получает состояния, в котором его можно принять.
Это ограничение доступности и очистки состояния, а не разрешение повторно
использовать QR.

## Проверки

Зафиксированы точные vectors всех семи внутренних типов и обоих типов
Olm-оболочки. Тесты меняют внешний `message_id` при неизменном ciphertext,
проверяют откат кандидата и затем принимают исходный Envelope. Отдельно
проверяются 32 сообщения в обратном порядке, replay, постоянный limiter,
неверный ключ SQLCipher, pending outbox после перезапуска и произвольный
ограниченный ввод строгих декодеров.

`experiments/sqlcipher-compat` остаётся отдельным стендом. Он не входит в
runtime и фиксирует точную native-версию до сборки основной базы.

На дату решения `cargo audit` проверил основной `Cargo.lock` и lock-файлы обоих
compatibility-экспериментов. В актуальной базе RustSec известных уязвимостей не
найдено. Это результат одного прогона, а не постоянная гарантия.

## Риски

- Olm v1 использует 64-битный MAC и остаётся только решением для experimental
  preview.
- SQLCipher и OpenSSL добавляют native-код, который не покрывается запретом
  `unsafe` для собственного Rust-кода.
- Наблюдатель видит hint, тип pre-key или normal, размеры и время передачи.
- Limiter позволяет подделкам временно мешать настоящему контакту.
- Property-тесты не заменяют coverage-guided fuzzing.
- Проект не проходил независимый аудит и не называется безопасным.

## Следующий шаг

Этап 3 может реализовать ограниченный LAN-транспорт на искусственном
непрозрачном payload. Подключение настоящего chat-пути к сетевому узлу остаётся
частью трёхустройственной демонстрации Этапа 4.
