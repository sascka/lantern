# ADR 0009: ограниченный жизненный цикл узла

Статус: принято 17 июля 2026 года.

## Контекст

Очередь, SQLite-хранилище, время, диагностика и транспортные квоты уже
проверены по отдельности. Без общего владельца вызывающий код может
прочитать часы несколько раз, забыть сохранить очередь или продолжить работу
после частичной ошибки.

## Решение

Добавлен crate `lantern-node`. Он владеет одним источником времени, очередью,
SQLite-хранилищем и in-memory журналом. Каждая операция, которая обслуживает или
меняет состояние узла, читает время один раз. Одно и то же значение используется
для очереди, диагностики и сохранения `last_observed_wall_seconds`.

При запуске узел:

1. Читает время.
2. Открывает базу.
3. Восстанавливает и повторно проверяет очередь.
4. При откате часов консервативно завершает активные записи.
5. Сохраняет итог в той же SQLite-базе.
6. Создаёт ограниченные диагностические события без пути, ID, payload, точного
   времени и точного размера очереди.

После внутренней ошибки состояние меняется на `Failed`. Повторные операции через
этот объект запрещены. После нового запуска SQLite вернёт последний целый
снимок. Это проще и надёжнее, чем пытаться угадать, какая часть операции успела
завершиться.

`NodeClock` оставлен маленьким интерфейсом. В обычной работе его реализует
`SystemRuntimeClock`. В тестах используется `ClockTracker` с явной
последовательностью наблюдений.

## Транспортная граница

Работающий узел создаёт `BoundedSession` с уже проверенными `SessionLimits`.
Узел в состоянии `Failed` или `Stopped` не может начать новую сессию.

Кадры в этом пакете не читаются. Без wire-формата нельзя безопасно передать
остаток TTL, число переходов и бюджет копий. Этот формат относится к Этапу 3.

## Альтернативы

### Вызывать все crate из CLI

Так будет одним слоем меньше, но порядок вызовов станет неявным и будет
дублироваться в каждом будущем способе запуска.

### Автоматически повторять ошибки SQLite

Повтор без понимания причины может долго удерживать файл и замаскировать
повреждение. Для v0.1 ошибка останавливает текущий объект узла.

### Сохранять данные в `Drop`

Неявный дисковый ввод-вывод в `Drop` не может вернуть ошибку. Вместо этого есть
явный метод `stop`, а каждая меняющая очередь операция сохраняет свой
результат сама.

## Риски и ограничения

- Кража базы раскроет Envelope и локальные маршрутные метаданные.
- Текущий payload не защищён криптографически.
- Повтор блокируется только пока ID есть в очереди или ограниченном
  tombstone-наборе.
- Поток новых ID может вытеснять полезные Envelope в пределах квоты.
- Неверное время может удалить Envelope раньше срока.
- Потеря и перестановка пакетов ещё не относятся к этому crate, потому что
  сетевого обмена нет.
- Постоянный журнал и блокировка профиля остаются следующим пакетом Этапа 2.
