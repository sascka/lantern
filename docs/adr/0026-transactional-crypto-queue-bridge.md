# ADR 0026: транзакционный мост crypto и очереди

Статус: принято и реализовано 18 июля 2026 года.

## Контекст

`lantern-crypto` уже создаёт зашифрованный Envelope и одной транзакцией
сохраняет новое состояние Olm-сессии вместе с точными CBOR-байтами в pending
outbox. `lantern-node` отдельно хранит открытую транспортную очередь. Между
этими базами нельзя сделать общую SQLite-транзакцию.

Если сначала изменить ratchet, а потом потерять Envelope, повторное шифрование
создаст другое состояние и может потерять сообщение. Если удалить входящий
Envelope до сохранения результата decrypt, авария потеряет единственную копию.
Простое последовательное соединение двух API поэтому недостаточно.

## Решение

Добавлен отдельный crate `lantern-bridge`. Он зависит от криптографического
контура и от узла, но не реализует сеть и не передаёт узлу пароль, ключи или
открытый текст.

Исходящий перенос выполняется так:

1. `lantern-crypto` заранее сохраняет новое состояние сессии и канонический
   Envelope в секретном outbox.
2. Мост читает не больше 32 записей за один вызов и снова строго декодирует
   сохранённые байты.
3. ID строки должен совпасть с ID внутри Envelope.
4. Узел атомарно сохраняет точный Envelope в своей очереди.
5. Только после этого запись удаляется из секретного outbox.

Перед чтением очереди мост требует рабочий `NodeRuntime` и сохраняет его
текущее состояние. Объект в состоянии `Failed` или `Stopped` не может
подтвердить outbox по одной несохранённой копии в памяти.

Если процесс остановился между шагами 4 и 5, повторный вызов сравнит два полных
Envelope. При точном совпадении он завершит удаление outbox без нового encrypt.
Одинаковый ID с другими полями считается конфликтом. При tombstone, истечении
или заполненной очереди outbox не подтверждается.

Входящий путь выполняется в другом порядке:

1. Мост берёт копию Envelope из очереди узла.
2. Активный контакт ищется по текущему, предложенному или уходящему входному
   `recipient_hint`.
3. Неизвестный hint не запускает decrypt и оставляет Envelope для Relay.
4. Криптографический слой проверяет payload и атомарно сохраняет новый ratchet
   и запись истории.
5. Только после успешного commit узел переводит Envelope в `Opened`, удаляет
   его из активной очереди и сохраняет tombstone.

Если процесс остановился между шагами 4 и 5, наличие принятого ID в секретной
истории позволяет завершить очистку очереди без второго decrypt и без второго
возврата текста вызывающему коду.

## Почему базы остаются раздельными

Одна общая база дала бы настоящую транзакцию, но смешала бы открытые данные
Relay с ключами, контактами и историей. Тогда сетевому жизненному циклу пришлось
бы получать доступ к секретному хранилищу. Раздельные базы сохраняют границу
компонентов, а идемпотентный протокол закрывает два известных окна аварии.

Другой вариант - шифровать заново при каждом запуске. Он несовместим с
состоянием ratchet и поэтому отклонён. Распределённая транзакция для двух
локальных SQLite-баз добавила бы намного больше состояний отказа без пользы для
MVP.

## Граница доверия

`lantern-node` по-прежнему не умеет расшифровывать Envelope. Его метод
`complete_opened` только сохраняет переход очереди и сам не доказывает, что
decrypt был успешен. Rust не поддерживает закрытый API между двумя отдельными
crate. Поэтому приложение должно выполнять этот переход только через
`lantern-bridge`; прямой вызов остаётся внутренним контрактом, который нужно
проверить при сборке CLI.

Мост видит `ReceivedChat` на короткое время и возвращает его конечному
приложению. Он ничего не логирует. `Debug` результата показывает только длину
текста через уже существующий закрытый тип.

## Проверки

Интеграционные тесты воспроизводят:

- перезапуск после сохранения очереди и до очистки outbox;
- конфликт двух разных Envelope с одним ID;
- отказ подтверждать outbox через остановленный узел;
- успешный decrypt, tombstone и replay;
- перезапуск после commit секретной истории и до очистки очереди;
- повторное восстановление без второго показа текста;
- неизвестный `recipient_hint` с непрозрачным недоверенным payload.

Все входы остаются ограничены: сохранённый Envelope не больше 64 KiB, один
вызов экспортирует не больше 32 записей, а очередь и обе базы имеют собственные
квоты.

## Риски и ограничения

- Тесты используют обычное завершение объекта, а не отключение питания во
  время файловой операции.
- Заблокированная первая запись outbox останавливает текущий FIFO batch. Это
  сохраняет порядок, но может задержать более новые сообщения.
- Ошибка сохранения после изменения очереди переводит текущий `NodeRuntime` в
  `Failed`. Продолжать можно только после нового запуска с целого снимка.
- Неизвестный hint не раскрывает контакт, но Relay всё равно видит hint, размер,
  TTL и сетевые метаданные.
- Кража открытой базы узла раскрывает Envelope и маршрутные данные. Кража
  секретной базы остаётся отдельной угрозой, описанной в модели угроз.
- Неверное время может удалить Envelope по TTL до decrypt. Мост не обходит
  решение часов узла.
- Потеря и перестановка сетевых пакетов не меняют порядок локальных commit, но
  полный трёхузловой зашифрованный сценарий ещё не собран.
- Публичный API и тесты не являются независимым аудитом безопасности.

## Следующий шаг

Собрать постоянные профили Alice и Bob с уже проверенными контактами так, чтобы
Relay имел только открытую очередь и не мог открыть их секретные базы.
