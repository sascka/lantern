# ADR 0024: ограниченная двусторонняя LAN-встреча

Статус: принято и реализовано 17 июля 2026 года.

## Контекст

Однонаправленные Offer, Request, Transfer и Done уже работали через настоящую
LAN-сессию. Для завершения Этапа 3 двум узлам нужно обменяться своими Envelope
в обоих направлениях, продолжить после обрыва и не выдавать новую полную квоту
одному IP при каждом reconnect.

Транспорт не знает очередь и криптографию, а sync не должен открывать TCP или
выбирать сетевой адрес. Поэтому порядок встречи остаётся в `lantern-node`, а
ограничение принятых соединений находится в `lantern-lan`.

## Решение

Для одной encounter-сессии определены две роли:

- `Initiator` открывает исходящее соединение, сначала отправляет batch, затем
  принимает batch;
- `Responder` принимает соединение, сначала принимает batch, затем отправляет
  batch.

Оба batch используют одну `BoundedSession`. В ней уже раздельно ограничены
входящие и исходящие кадры и байты. После ошибки сессия потребляется и не может
быть продолжена. Reconnect создаёт новое соединение и новый вызов
`run_encounter`. Автоматического цикла повторов нет.

Каждый `LanListener` хранит ограниченную таблицу IP:

- одна активная принятая сессия на IP;
- не больше 8 попыток за 60 секунд;
- не больше 256 IP одновременно в таблице;
- неактивная запись удаляется после окончания окна.

Lease активной сессии освобождается вместе с `LanConnection`. IP используется
только как ключ в памяти listener. Он не входит в ошибку, `Debug` или
диагностический журнал.

`NodeRuntime` записывает успешный Offer, завершённый Transfer и отказ только
через закрытые `EventCode`, `EventOutcome` и ограниченный `object_count`. В
событии нет адреса, ID, Envelope, payload, точного размера или строки ошибки.

## Почему reconnect не автоматический

Автоматический retry потребовал бы политики задержек, отмены, выбора адреса и
фонового runtime. Ошибка конфигурации могла бы создать бесконечный цикл
соединений. Для v0.1 вызывающий код явно решает, когда открыть следующую
сессию. Один вызов всегда выполняет конечное число сетевых операций.

## Проверка

Интеграционные тесты проверяют:

- разные Envelope в обоих направлениях одной loopback-сессии;
- обрыв после Request до сохранения Transfer;
- новое соединение и успешное постоянное сохранение;
- третье соединение, на котором принятый ID больше не запрашивается;
- отказ второй активной сессии с того же IP;
- разрешённый reconnect после закрытия первой сессии;
- предел попыток и предел таблицы peer.

## Риски и ограничения

- IP не является идентичностью и может принадлежать нескольким устройствам;
- атакующий может менять IP;
- окно и таблица сбрасываются при перезапуске listener;
- исходящие соединения ограничивает вызывающий код, а не listener;
- один blocking listener не обслуживает несколько handshake параллельно;
- неверно назначенные роли ждут до сетевого deadline;
- подтверждения конечного получателя для Wait-фазы ещё нет;
- TCP не аутентифицирован и не зашифрован транспортным слоем;
- реализация не проходила fuzzing или независимый аудит.
