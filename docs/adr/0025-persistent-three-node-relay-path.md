# ADR 0025: постоянный трёхузловой маршрут

Статус: принято и реализовано 18 июля 2026 года.

## Контекст

Этап 3 проверил одну встречу двух узлов. Главный сценарий Lantern требует двух
разных встреч: сначала Alice с Relay, затем Relay с Bob. Между ними Alice уже
не должна быть доступна, а Relay должен пережить перезапуск.

Подключать настоящий текст сразу нельзя. Хотя криптографический контур уже
существует, его транзакции ещё не связаны с открытой транспортной очередью.
Первый пакет Этапа 4 поэтому использует только очевидный синтетический payload.

## Решение

Новый интеграционный тест запускает три отдельных `NodeRuntime` с тремя
SQLite-базами:

1. Alice создаёт один синтетический Envelope с `max_hops = 2`.
2. Alice открывает encounter-сессию только с Relay.
3. Relay атомарно сохраняет Envelope и разрывает соединение с Alice.
4. Процесс Relay завершается.
5. Новый `NodeRuntime` Relay восстанавливает очередь из той же базы.
6. Relay открывает новую encounter-сессию только с Bob.
7. Bob сохраняет Envelope с `hops_taken = 2`.
8. Повторная встреча Relay и Bob передаёт ноль объектов.
9. Все три базы открываются снова, а сумма copy budget остаётся равна 32.

В тесте нет общего объекта соединения Alice и Bob. Единственный путь Envelope
состоит из двух последовательных TCP-сессий через постоянную очередь Relay.

Второй тест использует `max_hops = 1`. Alice ещё может отдать часть бюджета
Relay, но Relay уже не имеет права предлагать Envelope Bob. Это проверяет
жёсткую остановку распространения на локальной маршрутной записи.

## Почему пока используется синтетический payload

Relay должен хранить весь Envelope и поэтому видит его открытые поля и
`protected_payload` как байты. Конфиденциальность появляется только если Alice
до очереди создала корректный E2EE payload, а Bob открыл его через проверенную
сессию. Этот пакет не делает такого утверждения и не содержит ключей.

Следующий пакет должен определить транзакционный мост между secret outbox и
открытой очередью. До этого нельзя передавать пользовательский текст через
трёхузловой тест.

## Риски и ограничения

- сценарий работает только на loopback в автоматическом тесте;
- payload синтетический и не проверяет конфиденциальность;
- Relay видит все открытые поля Envelope и сетевые метаданные;
- конечный получатель ещё не распознаётся транспортной очередью;
- последний hop может быть потрачен на Relay, который уже не сможет передать
  копию дальше;
- Wait-копия с одним токеном не доставляется Bob;
- базы транспортной очереди не зашифрованы;
- тест не является длительным fault-injection испытанием;
- независимый аудит не проводился.

## Следующий шаг

Определить минимальный транзакционный API, который переносит зашифрованный
outbox Envelope в транспортную очередь и передаёт принятый Envelope
криптографическому ядру только на конечном устройстве.
