# Симулятор Lantern

Статус: седьмой рабочий блок Этапа 1.

Симулятор проверяет маршрутизацию сообщений между узлами без настоящей сети и
криптографии. Payload представлен только размером. Текст сообщения, ключи и
другие секреты не создаются.

## Что уже работает

- детерминированные идентификаторы с фиксированным seed;
- узлы, сообщения и встречи;
- создание сообщения в заданный момент времени;
- Direct Delivery;
- Epidemic routing;
- Binary Spray-and-Wait;
- сценарий Alice -> Relay -> Bob;
- дедупликация при повторной встрече;
- TTL для каждой локальной копии;
- ограничение числа переходов сообщения;
- явные причины удаления и запрета передачи;
- проверяемый бюджет копий для Spray-and-Wait;
- детерминированная потеря передач;
- дублирование с проверкой дедупликации;
- перестановка контейнеров внутри встречи;
- отдельные метрики сетевых попыток и сохранённых передач;
- настраиваемая квота числа копий и байтов на узел;
- детерминированное FIFO-вытеснение при заполнении квоты;
- явные причины вытеснения и отказа сохранения;
- ограниченное tombstone-хранилище удалённых `message_id`;
- отклонение немедленного возврата вытесненной или просроченной копии;
- метрики истечения и вытеснения tombstone-записей;
- ограниченный генератор сценария на 4-50 узлах;
- отдельные воспроизводимые потоки для сообщений и встреч;
- одинаковый след встреч для сравнения разных политик;
- параметры сценария в JSON-результате;
- количество доставленных сообщений;
- задержка доставки;
- число и объём передач;
- пиковое число и объём сохранённых копий.

## Чего пока нет

- реалистичное движение узлов;
- пакетные эксперименты;
- графики.

Эти функции будут добавляться отдельными проверяемыми блоками Этапа 1.

## Ограничения безопасности модели

Идентификаторы создаются через `random.Random`. Это правильно для
воспроизводимого эксперимента, но недопустимо для настоящего протокола.
Production-идентификаторы позже будут создаваться системным криптографически
стойким генератором случайных чисел.

Симулятор не доказывает безопасность шифрования, сети или хранилища. Он измеряет
только поведение алгоритмов маршрутизации.

## Время жизни и число переходов

`Message` содержит неизменяемые параметры сообщения. В том числе исходный TTL
и максимальное число переходов. Эти значения одинаковы для всех копий.

`StoredMessage` описывает конкретную копию на конкретном узле. В ней хранится
оставшийся TTL, момент получения и уже пройденное число переходов. При передаче
TTL уменьшается на время, прошедшее с момента получения копии, а счётчик
переходов увеличивается на один. Поэтому Relay не может случайно начать отсчёт
TTL заново.

Допустимый TTL - от 60 секунд до 7 суток. Допустимое число переходов - от 1 до
16. Значения по умолчанию - 300 секунд и 16 переходов.

Копия с истёкшим TTL удаляется перед обработкой следующего события. В результате
записывается причина `ttl_expired`. Если передача потратит последний разрешённый
переход, но не доставит сообщение получателю, она не выполняется. Причина
`hop_limit_before_destination` записывается отдельно, а копия остаётся у
отправителя. Он ещё может встретить конечного получателя.

## Binary Spray-and-Wait

При создании сообщения политика получает ограниченное число токенов копирования
`copies_left`. При встрече с новым Relay отправитель отдаёт ему половину своих
токенов, а остальные оставляет себе. Если число нечётное, больший остаток
остаётся у отправителя.

Когда у локальной копии остаётся один токен, начинается фаза Wait. Такую копию
можно передать только конечному получателю. После успешной передачи токены
перераспределяются, поэтому их общая сумма не увеличивается.

Binary Spray-and-Wait распространяет доступные копии быстрее, чем вариант, где
их может создавать только исходный узел. Цена этого решения - в распространении
участвует больше Relay. Окончательный алгоритм и начальный бюджет для Lantern
будут выбраны только после пакетных экспериментов.

Симулятор принимает бюджет от 1 до 64 токенов. Это техническая граница
эксперимента, а не утверждённое значение протокола. Значение по умолчанию - 4.

## Потеря, дублирование и перестановка

Модель канала работает отдельно от политики маршрутизации. Политика выбирает
контейнер для отправки, после чего `NetworkConditions` определяет результат
попытки.

Потерянная попытка записывается как `lost`, но не создаёт копию у получателя и
не уменьшает бюджет Spray-and-Wait. При следующей встрече сообщение можно
попробовать отправить снова.

При дублировании первая попытка сохраняет копию, а повтор той же передачи
проходит через обычную дедупликацию и получает результат `duplicate_ignored`.
Повтор увеличивает сетевой трафик, но не число сохранённых сообщений и не
расходует дополнительный токен копирования.

Перестановка разворачивает порядок контейнеров, выбранных для одного направления
одной встречи. Она не изменяет время встречи и не создаёт искусственную связь
между узлами. Для сценария с одним сообщением перестановка незаметна, поэтому
она отдельно проверяется сценарием с двумя сообщениями.

Процент потерь и дубликатов задаётся целым числом от 0 до 100. Решение для каждой
попытки зависит от seed, номера встречи, направления и `message_id`. Поэтому
одна и та же возможная передача получает одинаковый результат при сравнении
разных политик.

`attempt_count` и `bytes_attempted` учитывают весь смоделированный сетевой
трафик, включая потери и дубликаты. `transmission_count` и `bytes_transmitted`
учитывают только копии, которые были сохранены новым узлом.

## Квота хранения и вытеснение

Каждый узел получает одинаковую квоту `StorageQuota`. По умолчанию она повторяет
начальные ограничения MVP - не больше 1000 копий и 64 MiB. Для небольших тестов
оба значения можно уменьшить.

Если новая копия не помещается, узел удаляет копии в порядке FIFO. Сначала
выбирается самое раннее локальное `received_at`. Если несколько копий получены
одновременно, используется меньший `message_id`. Он нужен только как стабильное
правило сравнения и не даёт сообщению дополнительного приоритета.

Перед удалением узел сначала вычисляет полный список кандидатов. Если входящая
копия сама больше байтовой квоты, она отклоняется без изменения существующей
очереди. Успешное вытеснение записывается как `quota_evicted`, а слишком большая
копия - как `item_exceeds_byte_quota`.

FIFO выбран только для симулятора. Он использует локальное время приёма и не
доверяет присланным TTL или priority. При этом непрерывный поток новых сообщений
может вытеснять полезные старые копии. До Rust-реализации FIFO нужно сравнить с
другими правилами на больших сценариях.

Если несколько контейнеров одновременно претендуют на последнее место,
перестановка может изменить, какой из них останется. Это ожидаемый эффект
ограниченной очереди, поэтому сравнивать политики нужно на нескольких seed, а не
по одному удобному запуску.

Сейчас одна квота применяется ко всем копиям узла. В MVP ограничение 64 MiB и
1000 элементов относится прежде всего к очереди чужих контейнеров. Разделение
исходящих сообщений, входящих сообщений и Relay-очереди будет уточняться в
Lantern Core и не должно случайно ослабить общий предел памяти.

## Tombstone-хранилище

После удаления по TTL или квоте узел сохраняет только `message_id`, локальное
время записи и срок действия. Полный контейнер, payload и маршрутные поля в
tombstone не копируются.

Если сосед снова предлагает активный `message_id`, попытка учитывается как
сетевой трафик, но получает результат `tombstone_rejected`. Основная очередь и
бюджет Spray-and-Wait при этом не изменяются.

Экспериментальные значения по умолчанию:

- не больше 2000 tombstone-записей на узел;
- срок хранения 7 суток.

Оба значения ограничены и настраиваются. При заполнении удаляется самая старая
локальная tombstone-запись, при равном времени используется `message_id`. После
истечения или вытеснения записи старый контейнер снова может попасть в очередь.

Эти параметры не являются решением для Lantern Core. Их нужно проверить под
нагрузкой и сопоставить с размером основной очереди, TTL и доступной памятью.

## Сценарий на десятках узлов

`uniform_contacts` создаёт ограниченный синтетический след встреч. Сначала узлы
образуют кольцо, чтобы каждый из них участвовал хотя бы в одной встрече. После
этого пары выбираются равномерно с помощью отдельного генератора с фиксированным
seed. Встреча узла с самим собой не создаётся.

Сообщения получают случайные, но разные узлы отправителя и получателя. Время их
создания находится в первой четверти сценария. Поток случайных чисел для
сообщений отделён от потока встреч. Поэтому изменение количества сообщений не
меняет след контактов и не мешает честно сравнивать эксперименты.

Жёсткие границы генератора:

- от 4 до 50 узлов;
- от 1 до 100 сообщений;
- от количества узлов до 10 000 встреч;
- те же ограничения payload, TTL и hop limit, что в основном симуляторе.

Границы проверяются до создания больших списков. Они защищают учебный
эксперимент от случайного заполнения памяти, но не являются защитой настоящего
сетевого узла от враждебного трафика.

Параметры сценария записываются в поля `scenario` и `scenario_parameters`.
Один и тот же сценарий можно запустить с Direct Delivery, Epidemic и Binary
Spray-and-Wait. При одинаковом seed все политики получают одни сообщения и один
след встреч.

Проверочный запуск с seed `20260716`, 20 узлами, 10 сообщениями и 200 встречами
дал следующие результаты без потерь и дубликатов:

| Политика | Доставлено | Средняя задержка | Передачи | Трафик, байт | Пик копий |
| --- | ---: | ---: | ---: | ---: | ---: |
| Direct Delivery | 6 из 10 | 90,67 | 6 | 1 536 | 16 |
| Epidemic | 10 из 10 | 30,10 | 190 | 48 640 | 200 |
| Spray-and-Wait, бюджет 4 | 10 из 10 | 58,00 | 39 | 9 984 | 40 |

Это результат одного небольшого синтетического сценария, а не основание для
окончательного выбора алгоритма. Кольцо искусственно гарантирует участие всех
узлов, равномерный выбор пар не описывает человеческое движение, а единичный
seed не показывает разброс результатов.

## Запуск демонстрации

Epidemic передаёт сообщение через Relay:

```bash
python -m lantern_sim --policy epidemic
```

Direct Delivery не отдаёт сообщение Relay, потому что Relay не является
получателем:

```bash
python -m lantern_sim --policy direct
```

Binary Spray-and-Wait с двумя токенами доставляет сообщение через один Relay и
не хранит больше двух копий одновременно:

```bash
python -m lantern_sim --policy spray --copy-budget 2
```

С одним токеном политика переходит в фазу Wait сразу. В стандартном сценарии
Alice не встречает Bob напрямую, поэтому сообщение не будет доставлено:

```bash
python -m lantern_sim --policy spray --copy-budget 1
```

Сценарий на 20 узлах запускается отдельно:

```bash
python -m lantern_sim --scenario mesh --policy spray \
  --nodes 20 --messages 10 --encounters 200 \
  --seed 20260716 --copy-budget 4 --summary
```

`--summary` убирает из JSON подробные списки передач, удалений и отказов. Все
счётчики и итоговые метрики сохраняются. Сейчас симулятор всё равно хранит
подробные записи внутри процесса, поэтому этот флаг уменьшает только вывод, а
не расход памяти во время эксперимента.

Сценарий Alice -> Relay -> Bob остаётся вариантом по умолчанию. Параметры
`--nodes`, `--messages` и `--encounters` используются только вместе с
`--scenario mesh`.

Полная потеря попыток:

```bash
python -m lantern_sim --policy epidemic --loss-percent 100
```

Дублирование каждой успешно доставленной попытки:

```bash
python -m lantern_sim --policy spray --copy-budget 2 \
  --duplicate-percent 100
```

Потери, дубликаты и перестановку можно включить вместе:

```bash
python -m lantern_sim --policy epidemic --loss-percent 20 \
  --duplicate-percent 10 --reorder
```

Квота задаётся отдельно для каждого узла. В стандартном сценарии сообщение
размером 256 байт не помещается в квоту 128 байт и отклоняется без передачи:

```bash
python -m lantern_sim --policy epidemic \
  --max-stored-messages 2 --max-stored-bytes 128
```

FIFO-вытеснение нескольких сообщений проверяется отдельным сценарием теста:

```bash
python -m pytest tests/test_simulation.py \
  -k count_quota_evicts_oldest_copy -vv
```

Параметры tombstone-хранилища задаются отдельно:

```bash
python -m lantern_sim --policy epidemic \
  --max-tombstones 2000 \
  --tombstone-retention-seconds 604800
```

Немедленный возврат после вытеснения проверяется интеграционным сценарием:

```bash
python -m pytest tests/test_simulation.py \
  -k tombstone_blocks_immediate_return -vv
```

Результат выводится как JSON с отсортированными ключами. При одинаковом seed
результат должен быть одинаковым.

Можно указать собственный seed и размер непрозрачного payload:

```bash
python -m lantern_sim --policy epidemic --seed 20260716 --payload-size 256
```

Допустимый размер payload - от 1 байта до 64 KiB.

TTL и максимальное число переходов задаются отдельно:

```bash
python -m lantern_sim --policy epidemic --ttl-seconds 300 --max-hops 2
```

В сценарии Alice -> Relay -> Bob нужны два перехода. При `--max-hops 1`
передача на Relay будет запрещена, потому что она потратит единственный переход
и не доставит сообщение Bob.

## Тесты

```bash
python -m pytest
```

Тесты проверяют валидацию, повторяемость идентификаторов, прямую доставку,
трёхузловой сценарий, дедупликацию, порядок событий, уменьшение TTL, удаление
просроченных копий, обе границы hop limit, деление токенов, соблюдение бюджета
копий, потерю, повторную встречу, дублирование, перестановку, границы квоты,
атомарный отказ, FIFO-вытеснение, tombstone-повторы, истечение и переполнение
tombstone-хранилища. Для сценария на десятках узлов отдельно проверяются жёсткие
границы, кольцо встреч, раздельные потоки случайных чисел, одинаковый след для
трёх политик и повторяемость запуска с ошибками канала.

## Важное различие политик

Direct Delivery передаёт сообщение только при встрече с конечным получателем.
Это создаёт мало трафика, но не доставляет сообщение по цепочке.

Epidemic передаёт соседу все неизвестные ему сообщения. В маленьком сценарии
это повышает вероятность доставки, но число копий быстро растёт. TTL и hop limit
ограничивают распространение во времени и по длине пути, но не задают общий
бюджет копий.

Binary Spray-and-Wait ограничивает число копий, но при слишком маленьком бюджете
может не успеть передать сообщение в нужную часть сети. При слишком большом
бюджете его поведение приближается к Epidemic и создаёт больше трафика.

Сейчас очистка просроченных копий выполняется только в моменты событий. Если
эксперимент завершился раньше следующего события, копия может остаться в
финальном снимке, хотя её TTL истёк бы позже. Это свойство дискретной модели, а
не правило будущей реализации Lantern.

Текущая модель не имитирует частичные пакеты, повреждение байтов, задержку
подтверждения, коррелированные радиопомехи или разрыв посередине передачи.
Перестановка ограничена контейнерами внутри одной встречи. Эти упрощения будут
учитываться при выводах из экспериментов.

Сценарий `uniform_contacts` также не моделирует скорость, расстояние, стены,
распорядок дня, группы людей и повторяющиеся социальные связи. Последовательные
встречи разделены одной условной единицей времени. Проверочное кольцо повышает
связность искусственно, поэтому результаты нельзя переносить на реальную сеть
без дополнительных сценариев и серии seed.

Tombstone не доказывает подлинность сообщения и не заменяет криптографическую
защиту от повторов на конечном устройстве. Злонамеренный узел может использовать
много новых `message_id`, вытеснить старые tombstone-записи и затем повторить
ранее удалённый контейнер.
