# Симулятор Lantern

Статус: второй рабочий блок Этапа 1.

Симулятор проверяет маршрутизацию сообщений между узлами без настоящей сети и
криптографии. Payload представлен только размером. Текст сообщения, ключи и
другие секреты не создаются.

## Что уже работает

- детерминированные идентификаторы с фиксированным seed;
- узлы, сообщения и встречи;
- создание сообщения в заданный момент времени;
- Direct Delivery;
- Epidemic routing;
- сценарий Alice -> Relay -> Bob;
- дедупликация при повторной встрече;
- TTL для каждой локальной копии;
- ограничение числа переходов сообщения;
- явные причины удаления и запрета передачи;
- количество доставленных сообщений;
- задержка доставки;
- число и объём передач;
- пиковое число и объём сохранённых копий.

## Чего пока нет

- Spray-and-Wait;
- потеря и перестановка событий;
- вытеснение по квоте;
- случайное движение узлов;
- пакетные эксперименты;
- графики.

Эти функции будут добавляться отдельными проверяемыми блоками Этапа 1.

## Ограничения безопасности модели

Идентификаторы создаются через `random.Random`. Это правильно для
воспроизводимого эксперимента, но недопустимо для настоящего протокола.
Production-идентификаторы позже будут создаваться системным криптографически
стойким генератором случайных чисел.

Симулятор не доказывает безопасность шифрования, сети или хранилища. Он измеряет
только поведение алгоритмов маршрутизации.

## Время жизни и число переходов

`Message` содержит неизменяемые параметры сообщения. В том числе исходный TTL
и максимальное число переходов. Эти значения одинаковы для всех копий.

`StoredMessage` описывает конкретную копию на конкретном узле. В ней хранится
оставшийся TTL, момент получения и уже пройденное число переходов. При передаче
TTL уменьшается на время, прошедшее с момента получения копии, а счётчик
переходов увеличивается на один. Поэтому Relay не может случайно начать отсчёт
TTL заново.

Допустимый TTL - от 60 секунд до 7 суток. Допустимое число переходов - от 1 до
16. Значения по умолчанию - 300 секунд и 16 переходов.

Копия с истёкшим TTL удаляется перед обработкой следующего события. В результате
записывается причина `ttl_expired`. Если передача потратит последний разрешённый
переход, но не доставит сообщение получателю, она не выполняется. Причина
`hop_limit_before_destination` записывается отдельно, а копия остаётся у
отправителя. Он ещё может встретить конечного получателя.

## Установка на Arch Linux

Нужен Python 3.12 или более новая совместимая версия:

```bash
sudo pacman -S --needed python

cd "/home/qual/Desktop/prog/Lantern proj/simulator"
python -m venv .venv
source .venv/bin/activate
python -m pip install -r requirements-dev.txt
```

Runtime-зависимостей нет. Для тестов используется `pytest==9.1.1`. Его
транзитивные зависимости тоже зафиксированы, чтобы повторная установка не
подтянула другие версии без проверки.

| Пакет | Версия | Лицензия |
| --- | --- | --- |
| pytest | 9.1.1 | MIT |
| iniconfig | 2.3.0 | MIT |
| packaging | 26.2 | Apache-2.0 OR BSD-2-Clause |
| pluggy | 1.6.0 | MIT |
| Pygments | 2.20.0 | BSD-2-Clause |

Все пять пакетов используются только при разработке и тестировании. Их версии
и лицензии проверены по metadata установленных дистрибутивов.

## Запуск демонстрации

Epidemic передаёт сообщение через Relay:

```bash
python -m lantern_sim --policy epidemic
```

Direct Delivery не отдаёт сообщение Relay, потому что Relay не является
получателем:

```bash
python -m lantern_sim --policy direct
```

Результат выводится как JSON с отсортированными ключами. При одинаковом seed
результат должен быть одинаковым.

Можно указать собственный seed и размер непрозрачного payload:

```bash
python -m lantern_sim --policy epidemic --seed 20260716 --payload-size 256
```

Допустимый размер payload - от 1 байта до 64 KiB.

TTL и максимальное число переходов задаются отдельно:

```bash
python -m lantern_sim --policy epidemic --ttl-seconds 300 --max-hops 2
```

В сценарии Alice -> Relay -> Bob нужны два перехода. При `--max-hops 1`
передача на Relay будет запрещена, потому что она потратит единственный переход
и не доставит сообщение Bob.

## Тесты

```bash
python -m pytest
```

Тесты проверяют валидацию, повторяемость идентификаторов, прямую доставку,
трёхузловой сценарий, дедупликацию, порядок событий, уменьшение TTL, удаление
просроченных копий и обе границы hop limit.

## Важное различие политик

Direct Delivery передаёт сообщение только при встрече с конечным получателем.
Это создаёт мало трафика, но не доставляет сообщение по цепочке.

Epidemic передаёт соседу все неизвестные ему сообщения. В маленьком сценарии
это повышает вероятность доставки, но число копий быстро растёт. TTL и hop limit
ограничивают распространение во времени и по длине пути, но не задают общий
бюджет копий. Следующий блок добавит Spray-and-Wait и ограничение числа копий.

Сейчас очистка просроченных копий выполняется только в моменты событий. Если
эксперимент завершился раньше следующего события, копия может остаться в
финальном снимке, хотя её TTL истёк бы позже. Это свойство дискретной модели, а
не правило будущей реализации Lantern.
