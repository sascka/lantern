# Лицензии Lantern

Статус: решение Этапа 0.

Lantern будет использовать несколько лицензий для разных частей проекта. Это
нужно, потому что приложение, библиотека и документация используются по-разному.

Этот документ описывает выбранную схему, но не заменяет юридическую
консультацию. Перед публичным релизом нужно ещё раз проверить лицензии всех
зависимостей и добавить полные тексты используемых лицензий в репозиторий.

## Схема лицензирования

| Часть проекта | Лицензия | SPDX-идентификатор |
| --- | --- | --- |
| приложения и будущий интернет-реле | GNU AGPL 3.0 или более поздняя | `AGPL-3.0-or-later` |
| Rust-библиотеки и симулятор | Mozilla Public License 2.0 | `MPL-2.0` |
| спецификация протокола | Creative Commons Attribution 4.0 | `CC-BY-4.0` |
| остальные документы | Creative Commons Attribution-ShareAlike 4.0 | `CC-BY-SA-4.0` |

Пока в репозитории есть только документы. Для них лицензии уже указаны через
SPDX-комментарии в начале файлов. Лицензии будущего кода начнут действовать,
когда появятся соответствующие файлы с лицензионными уведомлениями.

## Приложения и сервер

Будущие пользовательские приложения и интернет-реле лицензируются под
`AGPL-3.0-or-later`.

AGPL требует сохранять доступ к исходному коду изменённой версии не только при
распространении программы, но и для пользователей, которые работают с ней по
сети. Это подходит будущему Relay: оператор не сможет развернуть изменённый
сервер для пользователей и скрыть его исходный код, если на него действует
AGPL.

Предполагаемые каталоги:

```text
apps/
server/
```

Точная граница приложения будет проверена после появления структуры workspace.

## Rust-библиотеки и симулятор

Переиспользуемое ядро и исследовательский симулятор лицензируются под `MPL-2.0`.

MPL применяет copyleft на уровне файлов. Изменённый MPL-файл при распространении
остаётся открытым, но его можно объединять с файлами под другими совместимыми
лицензиями. Это позволяет другим разработчикам использовать Lantern Core, не
перелицензируя весь свой проект.

Предполагаемые каталоги:

```text
crates/
simulator/
```

Мы не будем помечать MPL-файлы как `Incompatible With Secondary Licenses`. Это
сохраняет предусмотренную MPL возможность объединения с совместимым кодом из
семейства GPL при выполнении условий обеих лицензий.

## Спецификация протокола

Файл `PROTOCOL.md` лицензируется под `CC-BY-4.0`.

Эта лицензия позволяет копировать, переводить и изменять спецификацию, в том
числе для независимой совместимой реализации. Нужно указать авторство, дать
ссылку на лицензию и отметить внесённые изменения.

Спецификация описывает протокол, но не лицензирует программный код, патенты,
товарные знаки или сторонние материалы.

## Остальная документация

Следующие документы лицензируются под `CC-BY-SA-4.0`:

```text
README.md
VISION.md
MVP.md
THREAT_MODEL.md
LICENSES.md
```

Будущие общие документы по умолчанию получают ту же лицензию, если в самом
файле не указано другое.

CC BY-SA разрешает копирование и изменение документации при указании авторства.
Производные материалы нужно распространять под той же лицензией.

## Авторство

Для атрибуции используется имя `Lantern contributors`. После публикации
репозитория к атрибуции нужно добавлять ссылку на исходный проект и отмечать
изменения.

Лицензия применяется только к материалу, права на который принадлежат его
авторам. Сторонний текст, изображение, тестовый вектор или фрагмент кода нельзя
перелицензировать без подходящего разрешения.

## Новые файлы

Каждый новый текстовый файл получает SPDX-идентификатор в комментарии:

```text
SPDX-License-Identifier: AGPL-3.0-or-later
SPDX-License-Identifier: MPL-2.0
SPDX-License-Identifier: CC-BY-4.0
SPDX-License-Identifier: CC-BY-SA-4.0
```

Указывается только идентификатор, подходящий конкретному файлу. Несколько строк
из примера нельзя копировать в один файл.

Для сгенерированных файлов лицензионное уведомление хранится в исходном шаблоне
или рядом с генератором. Сторонние файлы сохраняют собственные уведомления.

## Зависимости

Добавление зависимости требует отдельной проверки:

1. найти официальное название и версию лицензии;
2. проверить совместимость с лицензией компонента Lantern;
3. записать источник и версию зависимости;
4. сохранить обязательные уведомления об авторстве;
5. проверить, не добавляет ли лицензия неизвестные ограничения;
6. не подключать зависимость до завершения проверки.

Лицензия зависимости не доказывает качество или безопасность её кода. Эти
проверки проводятся отдельно.

## Вклад других разработчиков

Если для вклада не заключено отдельное письменное соглашение, он принимается
под лицензией того файла, в который внесено изменение. Автор должен иметь право
передать свой вклад на этих условиях.

Отдельный CLA для v0.1 не требуется. Если позже появится организация или
двойное коммерческое лицензирование, это решение придётся обсудить до изменения
условий приёма вкладов.

## Официальные тексты

- AGPL 3.0: <https://www.gnu.org/licenses/agpl-3.0.html>
- MPL 2.0: <https://www.mozilla.org/MPL/2.0/>
- CC BY 4.0: <https://creativecommons.org/licenses/by/4.0/legalcode>
- CC BY-SA 4.0: <https://creativecommons.org/licenses/by-sa/4.0/legalcode>

Перед первым программным релизом полные неизменённые тексты нужных лицензий
должны находиться в каталоге `LICENSES/`.

## Ограничения решения

- Схема с несколькими лицензиями сложнее одной общей лицензии.
- Граница между AGPL-приложением и MPL-библиотекой должна оставаться явной.
- Совместимость будущих криптографических и сетевых зависимостей ещё не
  проверена.
- Документы не проходили проверку юристом.
- Название и логотип Lantern потребуют отдельной политики, если станут товарным
  знаком.
